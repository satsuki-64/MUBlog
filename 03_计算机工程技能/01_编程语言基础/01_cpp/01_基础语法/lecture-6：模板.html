Lecture 6：模板

泛型编程与模板

泛型编程

概念：泛型编程

  • 概念：
    • 泛型编程是指独立与任何类型的方式编写代码。泛型编程和面向对象编程，都依赖与某种形式的多态；
    • 面向对象编程的多态性在运行时应用于存在继承关系的类，一段代码可以可以忽略基类和派生类之间的差异；
    • 在泛型编程中，编写的代码可以用作多种类型的对象；
    • 面向对象编程所依赖的多态性称为运行时多态性，泛型编程所依赖的多态性称为编译时多态性或参数式多态性；
  • 关系：
    • 模板是泛型编程的基础。模板是创建类或者函数的公式；

什么是模板

概念：模板

  • 概念：模板的声明
    • 模板函数：template <typename T> int compare (T t1, T t2);
    • 模板类：template <typename T> class compare;
  • 概念：什么是模板
    • 模板定义并不是真正的定义了一个函数或者类，而是编译器根据程序员缩写的模板和形参来自己写出一个对应版本的定义，这个过程叫做模板实例化；
    • 编译器生成的版本通常被称为模板的实例，编译器为程序员生成对应版本的具体过程，类似宏替换；
    • 模板类在没有调用之前是不会生成代码的，由于编译器并不会直接编译模板本身，所以模板的定义通常放在头文件中；
    • 正是由于模板的出现，才使用 STL 这样的泛型库的实现成为可能；
  • 特点：
    • （1）模板就是占位符，不需要字符拼接，和普通的代码别无二致，仅仅多了一项模板参数声明；
    • （2）报错和调试都能准确的指向模板定义的位置，而不是模板实例化的位置；
    • （3）支持模板参数自动推导，不需要显式写出类型参数，同时也支持显式指定类型参数；
    • （4）支持隐式实例化 (implicit instantiation)，即由编译器自动实例化使用到的函数。也支持显式实例化 (explicit instantiation)，即手动实例化；
  • 示例：模板函数的定义
template <typename T>
int compare(T & t1, T & t2)
{
    if(t1 > t2) 
        return 1;
    if(t1 == t2)
        return 0;
    if(t1 < t2)
        return -1;
}
  • 示例：模板类的定义
template <typename T>
class compare
{
private:
    T _val;
public:
    explicit compare(T & val) : _val(val) { }
    explicit compare(T && val) : _val(val) { }
    bool operator==(T & t)
    {
        return _val == t;
    }
};

模板实例化

概念：模板实例化

  • 概念：隐式实例化
    • C++ 模板（函数模板 / 类模板）本身不是可执行代码，只是编译器生成具体函数 / 类的 “蓝图”；
    • 实例化就是编译器根据这份蓝图，结合具体的类型/值，生成可执行的具体代码的过程；
    • 隐式和显式实例化的核心区别在于：实例化是编译器自动触发，还是程序员手动强制触发；
  • 特点：隐式实例化
    • （1）编译器只生成程序实际用到的模板实例，未用到的模板分支 / 类型不会生成代码；
    • （2）实例化发生在编译阶段，而非运行阶段；
  • 示例：隐式实例化
#include <iostream>
using namespace std;

// 模板蓝图：未生成任何可执行代码
template <typename T>
T add(T a, T b) {
    return a + b;
}

int main() {
    // 调用add(1,2)：触发隐式实例化，编译器生成 int add(int, int)
    cout << add(1, 2) << endl; 
    // 调用add(1.5, 2.5)：触发隐式实例化，编译器生成 double add(double, double)
    cout << add(1.5, 2.5) << endl; 
    return 0;
}
  • 概念：显式实例化
    • 程序员通过特定语法，强制编译器为指定的类型/值生成模板实例，无论程序是否实际使用该实例；
    • 显式实例化是 “主动触发”，而非编译器 “被动响应”；
    • 显式实例化主要解决模板分离编译的问题（模板声明和定义分离时的编译错误），也可用于提前生成实例、减少重复编译开销；
    • 比如：模板的声明和定义分离是 C++ 的常见写法（声明在 .h，定义在 .cpp），但隐式实例化会报错，原因是编译器编译使用模板的文件时，找不到模板的定义（只有声明），无法完成隐式实例化；
  • 示例：错误的情况
// my_template.h
template <typename T> T add(T a, T b); // 仅声明，无定义

// my_template.cpp
#include "my_template.h"
template <typename T>
T add(T a, T b) { // 模板定义
    return a + b;
}

// main.cpp
#include "my_template.h"
int main() {
    add(1, 2); // 隐式实例化失败：编译器找不到add<int>的定义
    return 0;
}
  • 示例：显式实例化
// my_template.cpp
#include "my_template.h"
template <typename T>
T add(T a, T b) {
    return a + b;
}

// 显式实例化：强制生成int版本的add
template int add<int>(int, int);
// 若需要double版本，再添加：template double add<double>(double, double);

分析：模板特化

  • 示例：
    • 上述模板的特化版本；
  • 分析：
    • 下述特化的函数能正确比较字符串，由于特化版本和普通版本的函数是可以实现重载的，并且优先匹配普通函数；
  • 代码：
template<typename T>
int compare(T a, T b)
{
	cout << "T" << endl;
	return a == b ? 0 : (a > b ? 1 : -1);
}

template<>	//必须写，不然就是重载函数，而不是函数模板，特化版本了
int compare(const char* str1, const char* str2)
{
	cout << "const char *" << endl;
	return strcmp(str1, str2);
}
  • 问题：
    • 那么将这个特化放在何处？显然是要放在模板的头文件中。但这样会导致符号多重定义的错误。原因很明显，模板特化是一个函数，而非模板；
  • 解决：
    • 函数模板特化即函数，而非模板的概念，完全与普通函数一样；加上 inline 关键字或者分文件实现都是可以的；
    • 因为编译器直接扩展内联函数，不产生外部符号，在多个模块中 #include 它们没有什么问题。链接器不会出错，因为不存在多重定义的符号。对于像 compare 这样的小函数来说，inline 怎么说都是你想要的（它更快）；
  • 代码：
template<>
inline int compare(const char* str1, const char* str2)
{
	cout << "特化 const char *" << endl;
	return strcmp(str1, str2);
}

显式实例化 显式特化（Explicit Specialization）

Concept

概念：Concept

  • 引入：
    • 在没有引入 Concept 检查之前，需要在函数内使用假言实现类型检查：
    • template <typename T> T inc(T& a) { static_assert(std::is_integral<T>::value);return ++a;}
    • 虽然可以解决问题，但是 T 的检查是在函数内实现的，调用时出错了，用户才知道。不能从函数 inc 的声明直观看出 T 的要求。而使用 concept 可以直接报错；
  • 概念：
    • C++20 引入了 concept 以在编译期检查模板实参是否满足指定的约束；
  • 示例：
    • 对整型的类型限制：
      • template <typename T> concept integral = std::is_integral<T>:: value;
    • 使用：
      • template <integral T> T inc(T& a) { return ++a; }
