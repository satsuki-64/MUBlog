Lecture 3：帧同步基础

帧同步引入

概念：帧同步

  • 引入：
    • 其实“帧同步”属于一个翻译上的失误，宽泛一点来讲“帧同步”是指包含各种变形算法的 Lockstep，严格来讲就是指最基本的 Deterministic Lockstep。我猜测国内在引入这个概念的时候已经是 2000 年以后（具体时间没有考证），lockstep 算法已经有很多变形，时间帧的概念也早已诞生，所以相关译者可能就把“lockstep”翻译成了“帧同步”。当然也可能是引入的时候翻译成了“按帧锁定同步”，后来被大家以简化的方式（帧同步）传递开来；
    • 但不管怎么说，“帧”在实际应用中更多的是指画面渲染的频率，lockstep 里面的“step”概念要更宽泛一些才是；
  • 概念：
    • 了解游戏的朋友，都知道游戏是有帧率（FPS）的，每一帧都会运行相当复杂的运算（包括逻辑和渲染），如果运算规模达到一定程度就会拉长这一帧的时间，帧率也就会随之下降。所有影视作品的画面都是由一张张图构成的，每一个画面就是一帧，一秒能放多少个画面，就是有多少帧。在游戏里面，渲染器会不停的向渲染目标输出画面，并在帧与帧之间游戏处理各种逻辑，逻辑会改变游戏世界对象的行为，这些行为又会影响游戏画面的变化，这就是游戏的核心框架；
    • 早期的 lockstep 里面渲染和逻辑都是放在一个帧里面去处理的，这样一旦命令受到网络延迟的影响，玩家本地就会卡在一个画面直到消息的到来；
    • 为了解决这个问题，有一些游戏会将逻辑和渲染分开处理（分为逻辑帧和渲染帧），逻辑帧每隔固定的时间去处理所有逻辑事件；
    • 在不是严格锁帧的情况下，你本地即使没有收到网络数据，也可以在继续执行其他的逻辑并维持高频率的渲染（正在移动的对象不会由于短暂的延迟而静止不动）。这里面的逻辑帧就是 lockstep 里面的“Step”，也可以叫做“turn”，“bucket”或者“步”；
  • 总结：
    • 网络同步，其目标就是时刻保证多台机器的游戏表现完全一致；
    • 但是由于网络延迟的存在，时刻保持相同是不可能的，所我们要尽可能在一个回合（turn）内保持一致，如果有人由于某些原因（如网络延迟）没有同步推进，那么我们就需要等他，这就是 Lockstep（或者说一种停等协议）
    • LockStep 其实是很朴素的一种思想，很早就被用于计算机仿真模拟、计算机数据同步等领域。后来网络游戏发展起来后，也很自然的被开发者们拿到了系统设计中。早期 lockstep 被广泛用于局域网游戏内（延迟基本可以保持在 50ms 以内），所以这种策略是很有效的。lockstep 每个回合的触发，并不是由收到网络包驱动，也不是由渲染帧数驱动（当然渲染帧率稳定的话也可以以帧为单位，每 N 帧一个回合），而是采用客户端内在的时钟稳定按一定间隔（ 比如100ms） 的心跳前进；
    • 游戏的一开始，玩家在本地进行操作，操作指令会被缓存起来。在回合结束前（网络延迟在 50ms 以内），我们会收到所有其他客户端的指令数据，然后和前面缓存的指令一同执行并推进到下一个回合。如果玩家在一个回合开始到 50ms（网络延迟的一半）都没有任何操作，我们可以认为玩家发出了一个 Idle 指令，打包发给其他客户端；
    • 换个角度来看，假如一场游戏持续了 20 分钟，不考虑延迟的情况下整场游戏就是 12000 个回合（所有客户端都是如此）。现在我们反过去给每个回合添加指令，确保每个回合都收集到所有玩家的指令，那么就可以严格保证所有客户端每个回合的表现都是一样的。假如我们再把这些指令都存储起来，那么就推演出整场比赛，这也是为什么 lockstep 为什么做回放系统很容易；
    • 至于 lockstep 为什么要发送指令而不是状态，其实是与当时的网络带宽有关。很多游戏都有明确的人数限制（一般不超过 10 个人），玩家在每个回合能做的操作也有限甚至是不操作，这样的条件下所有玩家的指令一共也占用不了多少带宽。如果换成同步每个角色的状态数据，那么数据量可能会膨胀 10 倍不止；
    • 从原则上说，锁步数据既可以是游戏角色的状态信息也可以是玩家的操作指令，只不过由于各种原因没有采取状态数据罢了；
  • 图示：传统的 Lockstep
  • 问题：
    • 仔细分析一下 lockstep，其实就能发现不少缺点。首当其冲的就是网络条件较差的客户端很容易影响其他玩家的游戏体验。为了尽可能保证所有玩家的游戏体验，开发者们不断的对游戏进行更深一步的分析，先后提出了各种优化手段和算法，包括使用乐观帧锁定，把渲染与同步逻辑拆开，客户端预执行，将指令流水线化，操作回滚等；
    • 其次，lockstep 的另一个问题就是很难保证命令一致的情况下，所有客户端的计算结果完全一致，只要任何一个回出现了一点点的误差就可能像蝴蝶效应一样导致两个客户端后面的结果截然不同。这个问题听起来容易，实际上执行起来却有很多容易被忽略的细节，比如 RPC 的时序，浮点数计算的偏差，容器排序的不确定性，随机数值计算不统一等等。浮点数的计算在不同硬件（跨平台更是如此）上很难保持一致的，可以考虑转换为定点数，随机计算保持各个端的随机种子一定也可以解决，但是具体实现起来可能还有很多问题，需要踩坑之后才能真正解决；

帧同步基础

什么是帧同步

概念：什么是帧同步

  • 概念：
    • 帧同步（Frame Synchronization）是一项确保多人在线游戏中各玩家之间游戏状态一致性的技术；
    • 其基本原理是在每一帧内，将玩家的操作指令发送给其他玩家，以实现同步；
    • 具体而言，游戏中的服务器按照固定的时间频率（即帧率）接收并转发来自玩家的操作指令，这些指令会依照服务器设定的频率逐一传递给所有其他玩家，确保每个玩家的操作都在相同的时间点发生；
    • 在这一过程中，玩家接收到操作指令后，会按照服务器设定的帧率精确执行相应的操作，确保每个玩家的行为与其他玩家保持一致。这样，所有玩家在任何时刻看到的游戏状态，包括角色的动作、场景的变化以及其他游戏事件，都是一致的，没有延迟或差异；
    • 帧同步技术不仅消除了操作不同步的现象，还有效避免了因网络延迟引起的游戏卡顿或异常，使得游戏体验更加流畅、稳定；
    • 通过帧同步，游戏中的每一个小细节都能够在玩家之间保持一致性，包括角色的移动、技能的释放、物体的交互等，确保玩家之间的互动公平且精准。更重要的是，这种技术的使用能显著提升竞技类游戏中的公平性，避免了因不同玩家操作延迟产生的优势差距；
  • 分析：帧同步的适用场合
    • 虽然帧同步适用于某些类型的游戏，但它并不总是最适合所有游戏。像大多数大型多人在线游戏（MMO）或第一人称射击游戏（FPS）中，状态同步（Server-side synchronization）通常更合适，因为帧同步会带来更高的带宽和计算开销，而这些游戏往往对时延要求更高，反而需要通过服务器控制状态来确保稳定性；
      • 不过并不一定正确。因为帧同步只需要同步命令，而状态同步需要同步服务器计算得到的状态。而命令所占的字节数很少，因此即使每帧进行同步，也不一定带宽更高；
    • 帧同步确实能有效减少玩家之间的输入差异和时延问题，但它无法完全消除所有类型的延迟问题，尤其是在网络状况较差时。因为每一帧的数据都需要所有玩家在同一时刻保持同步，如果某个玩家的网络较差，可能会出现“卡顿"现象，导致游戏体验受损；
      • 游戏特别卡的时候，可能会导致卡帧，比如说游戏很卡的时候，可以看到人先是卡住，然后连续的闪烁瞬移，移动到面前 -> 此时是在将数据追到当前最新的帧，因此会进行快速渲染；
    • 虽然帧同步适合需要高度实时互动的竞技类游戏，但并非所有竞技类游戏都适合使用帧同步。并不是所有 MOBA 游戏都是帧同步，例如大型团队对抗（DOTA2）的 MOBA 游戏或大型竞技射击游戏，可能更适合使用其他同步方式（如状态同步或混合同步）来保持较高的稳定性和玩家体验；
      • 其实更类似于状态帧同步；
  • 优点：
      1. 一致性保证：
      • 帧同步能够保证所有玩家的游戏状态一致；
      • 这意味着每个玩家看到的游戏画面和操作结果是完全同步的，避免了因网络延迟或不稳定导致的画面错乱和不同步问题；
      • 这样玩家体验会更流畅，感觉更真实；
      1. 公平性：
      • 因为所有玩家的游戏状态都基于相同的时间和事件，这避免了某些玩家在延迟较小的情况下获得不公平的优势。游戏的竞技性和公平性大大提高；
      1. 容易调试：
      • 帧同步让开发者可以在相同的状态下进行调试，找出潜在的 bug 或者平衡问题。每个事件都对应一个时间点，这样更容易追踪和复现问题；
      1. 开发相对容易
      • 服务器只需要少量逻辑，大部分逻辑都在客户端；
      • 因为如果是使用状态同步，很多逻辑是在服务端；
  • 缺点：
    • 下面的缺点只建立在标准的帧同步下，其实游戏开发中大多都会在帧同步的设计方案上增加一些手段来避免下面的一些问题；
      1. 作弊工具与外挂：
      • 在帧同步的环境中，作弊工具可能会通过修改客户端的内存、模拟操作或其他手段，使得玩家能以不正常的方式控制自己的角色。而且因为服务端并没有跑客户端的逻辑，因此此时服务点不能知道客户端是否作弊；
      • 尤其是在一些高频次的同步游戏中，外挂工具可以通过修改帧数据或改变同步流程来绕过防护，获取游戏优势；
      1. 数据包篡改：
      • 由于每一帧都要传输大量数据，特别是游戏中的位置、操作指令等，这些数据可以通过网络被拦截和篡改；
      • 服务器只是每一帧的收集数据然后转发，并没有进行验证，因此可以通过一些外挂把每一帧数据变得非常庞大或者复杂；
      • 如果没有充分的加密或验证机制，恶意玩家可以通过篡改数据包来伪造游戏状态，从而获得不公平的优势；
      1. 服务器验证困难：
      • 因为服务器只是转发操作数据，所以无法验证客户端玩家发送数据；
总结：帧同步与状态同步优缺点
  • 帧同步：同步的内容为输入数据，例如：王者荣耀、CookingBattle、荒野乱斗等 MOBA 型游戏；
  • 状态同步：同步的内容为状态数据，例如：完美世界、魔兽世界等 MMOARPG

帧同步与网络协议

概念：帧同步与网络协议

  • 概念：帧同步更适合 UDP
    • UDP 不能保证数据送达、不保证顺序，没有重传机制，丢了就丢了。因为没有 TCP 的重传机制，可以在延迟低的情况下自定义重发机制，所以延迟相对 TCP 要低；
    • 帧同步更适合 UDP，因为帧同步对丢包的容忍度较高，帧同步通常是连续的数据流。帧丢了，下一帧会覆盖更新，可以通过帧同步的逻辑来避免消息顺序问题；
    • （1）对丢包的容忍度较高：
      • 比如当前服务器已经发了第二帧和地三帧，但客户端当前只接收到了第一帧，此时服务器会把第二帧和第三帧的数据一起发过来，因此可以把丢包的问题解决了；
    • （2）低延迟和低抖动：
      • 可以自定义或者干脆不做确认机制；
      • 带来了更低的延迟；
    • （3）灵活的应用层补偿机制：
      • 补帧、追帧、跳帧；
      • 包序号控制；
  • 原理：
    • 帧同步的基本思想，是让所有客户端在相同的逻辑帧执行相同的操作，从而保证游戏状态的一致性；
    • 与状态同步不同，帧同步只同步玩家的输入指令，而不是游戏对象的状态数据。所有客户端运行完全相同的游戏逻辑代码，接收到相同的输入序列后，每个操作都是在相同的时间间隔执行。理论上会产生完全相同的游戏状态；
    • 所有逻辑运算都在客户端进行，客户端保证彼此之间的一致性；

Bucket Synchronization

概念：Bucket Synchronization

  • 引入：
    • 传统的确定性锁步同步（Deterministic Lockstep）有一个简洁的规则：每一逻辑帧，服务器必须收集到所有玩家的输入指令后，才能将这些指令广播给所有客户端，各客户端才能据此推进游戏逻辑；
    • 这种模式带来了一个致命问题，即 “木桶效应”：整个游戏的推进速度取决于网络延迟最高的那个玩家。只要有一个玩家因为网络波动导致指令包迟到，其他所有玩家，包括网络通畅的玩家，都必须暂停游戏逻辑，一起等待；
    • Bucket Synchronization 正是为了打破这一困境而诞生的；
  • 概念：
    • Bucket Synchronization 核心改进在于，将游戏的流畅性从“最慢的玩家”手中解放出来，通过一种更为灵活和宽容的机制，在保证最终一致性的前提下，极大提升了游戏的流畅体验；
    • 算法将连续的时间轴划分为一系列固定长度的时间片段，每个片段就像一个“桶”（Bucket）。这个时间片段的长度是固定的，例如 40ms 或 50ms，这实际上定义了游戏逻辑更新的基本时间单位；
    • 玩家的操作指令不再要求在同一帧内必须被收集和执行。相反，指令在产生时会被打上一个时间戳，并放入其对应的未来某个时间桶中；
    • 服务器不会无限期等待未到达的指令。每个时间桶都有一个截止时间（Playout Delay）。当到达某个时间桶的截止点时，服务器就会将这个桶内已经收集到的所有指令广播给所有客户端，游戏逻辑随即推进。那些迟到的指令不会被丢弃，而是会被放入它们本应属于的后续时间桶中等待执行；
  • 机制：
    • （1）指令提交：
      • 玩家在客户端执行操作，客户端立即将带有精确时间戳的指令发送给服务器（或在 P 2 P 架构下发送给其他玩家）
    • （2）分桶归档：
      • 服务器根据指令的时间戳，将其归入对应的未来时间桶；
      • 例如，一个本应属于第 101 号桶的指令，即使迟到，也会被尝试放入当前即将执行的第 102 号桶；
    • （3）定时广播：
      • 服务器以固定的节奏（如每 50 ms）检查当前时间桶的截止时间是否到达。时间一到，便将该桶内已收集的所有指令打包，广播给所有客户端；
    • （4）客户端执行：
      • 所有客户端同时收到指令包，并在本地完全一致地执行这些指令，从而推进游戏世界；
  • 特点：
    • 实时性更好，一致性更差；
  • 图示：
    • 如图可知，当前 Player B 发送的 B1 和 B2 数据，由于 B2 在 Bucket 2 才送达，因此它会在 Bucket 2 的末尾才会被发送到 Player A，Player A 不会被阻塞；

帧同步调度机制

逻辑帧

概念：服务端实现

  • 概念：服务端的作用
    • 在帧同步架构中，服务端并不负责繁重的游戏逻辑计算（这部分工作主要在客户端完成），而是作为一个的“调度员”和“记录员”，其主要职责和运行机制如下：
      1. 接收所有客户端发送的操作指令（如移动、施法），并按帧号打包广播给所有客户端；
      1. 通过管理随机数种子、校验逻辑版本等方式，为所有客户端提供完全一致的运算基础；
      1. （可选）如果需要反作弊，则还需要服务端会运行一套与客户端相同的逻辑，将自身运算结果与客户端上报的结果进行比对，从而判断客户端是否作弊；
  • 分析：逻辑帧及其频率
    • 逻辑帧的频率是帧同步游戏的“心跳”，逻辑帧的频率通常在每秒 10 帧到 20 帧​ 之间，即每帧时长 100ms 到 50ms。逻辑帧处理所有确定性计算（如物理碰撞、技能效果、数值变化），由服务器统一驱动；
    • RTS（即时战略）或 MOBA（多人在线战术竞技）游戏需要较快的反应，频率可能偏高（如 15-20 fps）。而节奏稍慢的卡牌或回合制游戏可以稍低（如 10 fps） ；
    • 为什么大家都推荐 15 帧，20 帧可以吗？10 帧可以吗？其实这个帧选择多少，主要还是看人反应的速度，就比如说，你一秒钟你能快速按几个键。一般的人一秒最多按 3-5，有些高手可以 6-10。按照 10 帧来算，我一秒最多按到 10，那我做成 15 帧，另外 5 帧可能就没有数据；
    • 15 帧是某些特别成熟的游戏他们测试出来的，经过他们的大量测试，发现 15 帧能最少减少服务器的性能和带宽，还可以提升玩家的手感和流畅度；
    • SLG 大多用帧同步来设计，这样的游戏的帧就可以更低些，比如 10、8 帧；
    • 因此，逻辑帧的大小需要在确保操作流畅性的同时，找到性能和体验的最佳平衡点。帧率越高，客户端计算压力也越大；

帧锁定与乐观帧

概念：乐观帧与帧锁定

  • 引入：传统帧锁定同步
    • 服务器必须收集到所有玩家当前帧的操作指令后，才广播并推进到下一帧。因此此时严格同步，所有客户端帧进度绝对一致，可以适用于早期 RTS 游戏或对延迟极度不敏感、玩家数量少的场景；
    • 其基本流程为：
      • （1）在每个逻辑帧开始时，服务器等待接收所有客户端的操作指令；
      • （2）如果某个客户端的指令因网络延迟未到达，服务器会停止并等待，游戏帧暂停推进；
      • （3）一旦收齐所有指令，服务器将它们打包成一个“帧数据包”，广播给所有客户端；
      • （4）所有客户端收到同一个帧数据包后，才在本地的同一逻辑帧中执行这些指令，进行计算和表现；
    • 游戏进程会被网络最慢的玩家“锁住”。因此，在现代对实时性要求高的游戏中已较少采用纯帧锁定模式；
  • 概念：乐观帧与帧锁定
    • 服务端不会因为等待某个客户端的操作指令而卡住整个游戏进程；
    • 它会按照自己固定的逻辑帧的时常，广播当前收集到的所有操作。如果某个客户端因为网络延迟在某帧没有指令上传，服务端就广播一个空指令或让其重复上一帧的操作；
  • 特点：
    • 优点：
      • 避免了因个别玩家网络卡顿导致“一人卡顿，全场等待”的糟糕体验；
    • 缺点：
      • 网络延迟高的玩家会体验到操作延迟，因为他的指令需要时间传到服务器再广播出去；
  • 实现：
    • 服务端：
      • 服务器不再等待所有玩家，每到一次间隔，就将当前已经收到的所有玩家指令广播出去，无论是否收齐；
      • 网络差的玩家指令可能在这帧被遗漏，那么他的角色就会维持上一帧的操作或执行空操作；
    • 客户端：
        1. 指令缓存与快进播放：每个客户端都会本地缓存自己的操作指令。如果网络短暂卡顿后，一次性收到服务器发来的多个帧数据，客户端会识别出缺失的帧，并像快进播放录像一样，连续执行这些帧的逻辑，快速追上当前服务器的帧进度；
        1. 将逻辑帧与渲染帧分离：这是实现流畅表现的关键。逻辑帧以较低的固定频率（如 15fps）运行，由服务器驱动。渲染帧则以更高的频率（如 60fps）运行，在逻辑帧之间进行插值平滑计算，使画面流畅，避免因逻辑帧率低而产生的“幻灯片”效果；

Frame Buffer

概念：Frame Buffer

  • 引入：
    • 想象一下玩一款多人对战游戏，你的每一个操作（比如移动、攻击）都会被封装成一个数据包，通过网络发送给其他玩家。但由于网络状况复杂多变，这些数据包可能会延迟到达、顺序错乱甚至丢失；
    • 如果游戏客户端每收到一个数据包就立刻处理，游戏画面必然会因为等待而卡顿，或者因为执行顺序错误而导致不同玩家看到的状态不一致；
    • Frame Buffer 就是为了解决这些问题而生的。它的核心思想是：不是收到数据就立刻处理，而是先“存起来”，在最合适的时机，按照正确的顺序，批量交付给游戏逻辑执行；
  • 概念：
    • （1）数据缓存与排序：
      • Frame Buffer 会暂存陆续到达的网络数据包；
      • 它依据每个数据包携带的帧编号（Frame ID）对其进行排序，确保游戏逻辑总是按照 1, 2, 3... 的正确顺序来处理指令，即使数据包是 2, 1, 3 的顺序到达的；
    • （2）对抗网络抖动：
      • 网络延迟时高时低，这就是抖动；
      • Frame Buffer 通过引入一个短暂的缓冲延迟，比如 80-150 毫秒，来“熨平”这种波动。它会让游戏逻辑基于一个固定的、稍晚的时间点来运行，为大多数数据包的到达留出时间，从而有效减少因个别数据包迟到而导致的卡顿；
    • （3）依赖关系管理：
      • 在复杂的游戏中，某些帧的执行可能依赖于之前帧的结果；
      • Frame Buffer 能够识别这种依赖关系，确保只有在所有前置依赖帧都就绪后，才将当前帧标记为“可执行”，从而维护逻辑的正确性；
    • （4）渲染平滑处理：
      • 网络逻辑帧率（如 15Hz 或 20Hz）通常低于屏幕渲染帧率（60Hz）；
      • Frame Buffer 层可以提供稳定的历史状态数据，供渲染层在两个逻辑帧之间进行插值计算，使得物体运动看起来非常平滑，而不是生硬地“跳跃”；

逻辑与渲染分离

概念：逻辑与渲染分离

  • 概念：逻辑和渲染
  • 概念：
    • 必须创建一个与渲染更新解耦的、独立的逻辑帧更新循环。这个循环以固定的时间间隔（如每 66ms）触发，驱动整个游戏世界的逻辑前进；
    • 逻辑层与渲染层如何安全、高效地通信是关键。主要有两种架构思路：
      1. 事件驱动模式：逻辑层在状态发生变化时（如角色移动、创建新物体）触发事件。渲染层订阅这些事件，并据此更新视觉表现。这种模式解耦彻底，逻辑层完全不知道渲染层的存在；
      1. 数据轮询模式：每个渲染对象内部持有一个对应逻辑对象的引用。在渲染层的每一帧（如 Update 中），从逻辑对象中读取当前状态（位置、旋转等），然后更新自己的表现；

追帧与回滚

概念：追帧与回滚

  • 概念：追帧
    • 概念：
      • 也称为“赶帧”或“加速同步”；
      • 当某个客户端由于网络延迟或短暂断线而落后于服务器或其他玩家当前帧进度时，它需要一种机制来快速追赶上当前的游戏进度。这个过程不是简单的“瞬移”，而是让客户端在短时间内以数倍于正常的速度（例如 5 倍速、10 倍速）执行落后的逻辑帧，直到其状态与权威状态同步。这就像录像带快进，旨在解决“落后”的问题；
    • 实现：
      • 追帧通常在两种情况下触发：
          1. 断线重连：玩家网络中断后恢复，需要同步到最新状态；
          1. 网络波动：持续的高延迟导致积压了大量未处理的逻辑帧；
      • 高效的追帧依赖于游戏状态快照。服务器会定期（如每 30 或 50 帧）生成一个全量快照，记录此时所有游戏对象的完整状态；
      • 帧与帧之间则生成增量快照，只记录状态变化部分。当客户端需要追帧时，服务器会发送一个最近的全量快照以及从此到当前帧的所有增量快照。客户端先加载全量快照获得一个正确的起点，然后快速执行增量帧，这比从第 0 帧开始重放所有输入要高效得多；
      • 在追帧模式下，客户端逻辑更新循环会脱离固定的渲染帧率限制。例如，正常情况每秒执行 30 逻辑帧，追帧时可能每秒执行 100 帧甚至更多；
      • 关键在于，追帧过程只计算逻辑，不播放或简化表现。在追帧期间，可能会跳过非关键的视觉特效、声音或复杂动画，或者仅播放简化版本，以确保追帧速度，待追上进度后，再恢复正常表现；
  • 概念：回滚
    • 概念：
      • 这是一种更为积极的“乐观”策略。为了极致的操作响应，客户端不会被动等待服务器确认，而是会立即根据本地输入进行预测执行和表现。如果之后收到的服务器权威帧数据与本地预测一致，则继续执行；如果不一致，客户端就需要“回滚”到之前某个正确的确定状态，并依据正确的输入重新模拟至当前帧。这旨在解决“预测错误”的问题；
    • 实现：
      • 预测执行与状态保存：
        • 客户端收到玩家操作输入后，不会等待网络往返，而是立即基于此输入推进本地游戏逻辑并给予视觉反馈，这就是预测执行；
        • 为了应对可能的预测错误，系统需要在执行预测前，保存当前帧的完整游戏状态快照。这些快照保存在一个容量有限的环形缓冲区中；
      • 一致性校验与回滚触发：
        • 客户端会持续接收服务器广播的包含所有玩家权威输入的帧数据；
        • 收到后，会将本地之前预测执行的输入与服务器权威输入进行比对。一旦发现不一致（例如，本机预测你向右移动，但服务器数据显示你当时被击中僵直，实际无法移动），回滚即被触发；
      • 状态回滚与重模拟：
        • 系统会从保存的快照缓冲区中，找到发生预测错误的前一帧的正确状态，并将其加载为当前状态；
        • 然后，从这一帧开始，使用刚从服务器收到的、经过校验的权威输入序列，重新模拟（Re-simulation）游戏逻辑，直至当前帧；
  • 注意：
    • 确定性是帧同步的核心前提。意味着相同的初始状态 + 相同的输入序列 + 相同的计算顺序 = 完全相同的结果；
    • 这要求对浮点数运算（常使用定点数替代）、随机数生成器（使用相同种子）、容器遍历顺序等有严格的规范；

帧同步与确定性物理

概念：帧同步与确定性物理

  • 引入：帧同步为什么需要确定性
    • 帧同步的目标是确保所有客户端在相同游戏帧下，基于相同的输入，经过相同的计算过程，得到相同的游戏状态，因此其核心在于确定性（Determinism）；
    • 确定性物理是实现帧同步的基石。它要求物理计算在不同硬件、平台、设备上，只要初始状态和输入序列相同，每一帧的物理状态演变结果都完全一致；
    • 而 Unity 内置的 PhysX 物理引擎基于浮点数计算。浮点数在不同平台或硬件上可能存在细微精度差异。在单机游戏中这无关紧要，但在帧同步网络游戏中，即使是最微小的差异（例如 0.0000001），随着帧数累积（“蝴蝶效应”），也会在数百上千帧后导致各客户端出现肉眼可见的、灾难性的不同步；
  • 概念：确定性物理的核心机制 - 定点数
    • 为了解决浮点数精度不确定性，常用方法是使用定点数。定点数通过固定小数点位置，用整数运算来模拟小数运算，从而保证确定性；
    • 例如，使用 Q12.20 格式（32 位整数中，12 位表示整数部分，20 位表示小数部分）

代码：简单的定点数实现示例

using System.Collections.Generic;
using UnityEngine;

// 简单的定点数结构 (Q12.20格式)
public struct FixedInt
{
    private long rawValue;
    private const int FRACTION_BITS = 20;
    private const long MAX_VALUE = long.MaxValue >> FRACTION_BITS;
    private const long MIN_VALUE = long.MinValue >> FRACTION_BITS;

    public FixedInt(long value)
    {
        rawValue = value << FRACTION_BITS;
    }

    public FixedInt(int value)
    {
        rawValue = (long)value << FRACTION_BITS;
    }

    public static FixedInt FromFloat(float value)
    {
        FixedInt result;
        result.rawValue = (long)(value * (1 << FRACTION_BITS));
        return result;
    }

    public float ToFloat()
    {
        return rawValue / (float)(1 << FRACTION_BITS);
    }

    public static FixedInt operator +(FixedInt a, FixedInt b)
    {
        a.rawValue += b.rawValue;
        return a;
    }

    public static FixedInt operator -(FixedInt a, FixedInt b)
    {
        a.rawValue -= b.rawValue;
        return a;
    }

    public static FixedInt operator *(FixedInt a, FixedInt b)
    {
        long result = (a.rawValue * b.rawValue) >> FRACTION_BITS;
        return new FixedInt { rawValue = result };
    }

    public static FixedInt operator /(FixedInt a, FixedInt b)
    {
        long result = (a.rawValue << FRACTION_BITS) / b.rawValue;
        return new FixedInt { rawValue = result };
    }

    public override string ToString()
    {
        return ToFloat().ToString("F6");
    }
}

// 定点数版本的Vector3
public struct FixedVector3
{
    public FixedInt x;
    public FixedInt y;
    public FixedInt z;

    public FixedVector3(FixedInt x, FixedInt y, FixedInt z)
    {
        this.x = x;
        this.y = y;
        this.z = z;
    }

    public static FixedVector3 FromVector3(Vector3 v)
    {
        return new FixedVector3(
            FixedInt.FromFloat(v.x),
            FixedInt.FromFloat(v.y),
            FixedInt.FromFloat(v.z)
        );
    }

    public Vector3 ToVector3()
    {
        return new Vector3(x.ToFloat(), y.ToFloat(), z.ToFloat());
    }

    public static FixedVector3 operator +(FixedVector3 a, FixedVector3 b)
    {
        a.x += b.x;
        a.y += b.y;
        a.z += b.z;
        return a;
    }

    public static FixedVector3 operator *(FixedVector3 v, FixedInt scalar)
    {
        v.x = v.x * scalar;
        v.y = v.y * scalar;
        v.z = v.z * scalar;
        return v;
    }

    public FixedInt Magnitude()
    {
        // 简化处理，实际应用可能需要更精确的开方实现
        return FixedInt.FromFloat(Mathf.Sqrt(
            (x * x + y * y + z * z).ToFloat()
        ));
    }
}

客户端实现

概念：客户端实现

  • 概念：
    • 客户端都是先采集指令，采集指令后发送给服务器；但是采集的时候呢，并不是马上执行这个逻辑。比如鼠标点了某个点，然后客户端采集、将其发送给服务器，然后服务器根据当前帧率，打包发送给所有客户端；
    • 然后服务器发送完成之后，客户端拿到帧数据，开始执行指令，然后渲染层根据这个指令进行渲染；
    • 因此客户端执行的永远是上一帧的内容，这个上一帧的内容是由服务器发送而来的，称之为权威帧；
