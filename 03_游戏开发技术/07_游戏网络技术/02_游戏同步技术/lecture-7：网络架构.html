各个子系统服务

概念：各个子系统服务

  • 概念：Login Server
    • 客户端想要连接服务器时，需要先通过 Login Server 与服务器进行连接；
  • 概念：Gateway
    • Gateway 将服务器的内外网隔离开，作为防火墙来保护 Business Layer 当中的各种内容，并且对客户端的加密的数据进行解密、解压缩等等；
    • 因此 Gateway 通常会随着玩家数量变多，数量变得更多；
    • 客户端只能通过 Gateway 提供的外网接口，访问到内网当中的服务；
  • 概念：Lobby
    • Lobby 是一个缓存池，先把玩家信息加载进来、等待；
  • 概念：Character Server
    • 这个服务的负担特别重，所有玩家数据均集成于单一系统内进行管理，例如账户信息、角色信息、背包数据、邮件系统等；
  • 概念：Trading System
    • 在市场上购买和出售物品，通过游戏内的邮件系统向其他玩家发送物品或游戏币，游戏设计师需要密切关注市场价格，以防止经济失衡；
    • 为了让持久化的游戏世界维持稳定的经济体系，必须在货币来源与消耗途径之间取得平衡，玩家可以使用现实货币购买特定的游戏内物品；
  • 概念：Matchmaking
    • 概念：
      • 匹配机制需要综合考量玩家技能、等级、网络延迟、等待时间等多种属性。比如匹配游戏时进来优先匹配延迟类似的玩家；
      • 总体而言，打造优质的匹配服务是游戏设计的核心环节，而面向全球玩家群体部署此类服务，则会面临一系列截然不同的复杂挑战；
    • 图示：
  • 概念：Data Storage
    • 游戏数据体系具有高度复杂性与多样性，具体包含：玩家数据（公会、副本进度、仓库库存等）、系统监控数据、数据挖掘指标；
    • 所有数据均需实现安全持久化存储，并通过高效的组织结构以支持快速检索与分析等操作；
    • 最初使用关系型数据库，但是现在使用非关系型数据库越来越多，存储一些临时的 Game State 非关系数据库速度更好；
    • 内存数据库在现在游戏当中用的也很多，因为在游戏进行时会产生大量的中间数据，这些数据需要快速的读取写入，但是又不需要频繁将其写入磁盘，因此可以将其存储到 redis 等内存数据库当中；

网络游戏的分布式系统

概念：网络游戏的分布式系统

  • 概念：分布式的系统
    • 分析：分布式的系统的挑战
      • 数据访问互斥；
      • 分布式缺陷呈指数级扩散：一个很低频的问题出现之后，在各个服务器系统之间来回传输错误的数据，进而导致错误的数据被计算、然后发给其他的服务器。因此这个错误的信息可以在服务器当中震荡、反复的被计算，最后导致服务器性能下降，错误被指数级的扩散；
      • 幂等性、一致性与共识机制、故障与局部失效、分布式事务、不可靠网络环境；
    • 图示：
  • 概念：Load Balancing - 负载均衡
    • 指的是将一系列任务分配到一组资源（计算单元）上的过程，其目的是提高整体处理效率；
    • 具体目标包括：优化响应时间、避免计算节点负载不均（防止部分节点过载而其他节点处于闲置状态）；
    • 将所有玩家均衡分配至多个服务器（例如 Server1、Server2...）上实现负载均衡；
  • 概念：一致性哈希
    • 目标：
      • 其设计初衷是为了解决在集群中动态增删服务器时，需要重新分配所有玩家的问题；
      • 如果每一次要对玩家的数据进行更改时、都轮询所有的服务器、问服务器有没有当前玩家的数据，则需要花费大量的时间在轮询上；
      • 因此可以通过一致性哈希、将玩家 ID 和固定的 Server 映射上，使得当前对玩家的数据的服务器上的增删，可以通过哈希函数直接查询得到；
    • 分析：
      • 服务器有一个分布，所有的角色也有一个分布，假设有一个哈希算法，可以把数据分布在各个服务器上；
      • 需要服务器和客户端可以均匀的分布；
      • 假设当前 P3、P4 退出游戏，则将其释放；
      • 只要给定了哈希函数，就可以通过同样的哈希算法、将 Player 分配到不同的对于的服务端；

带宽优化

为什么带宽优化很重要

概念：为什么带宽优化很重要

  • 带宽过大的问题：
    1. 按使用量计费：例如移动服务、云服务；
    1. 带宽不足导致延迟加剧：数据包分片/丢包；
    1. 消息过载引发连接中断；

数据压缩 - 浮点数压缩

概念：数据压缩 - 浮点数压缩

  • 概念：
    • 游戏重最常用的数据压缩的方法，就是把浮点数压缩为定点数；
    • 游戏同步数据中存在大量浮点数，例如坐标位置、旋转角度、移动速度等，选择合适的浮点数精度可以显著节省带宽占用，以人类奔跑速度为例，仅需半精度浮点数即可满足需求；
  • 图示：
  • 概念：定点数与游戏分区
    • 在表示玩家位置时，由于移动速度的限制，玩家的活动范围会被控制在特定区域内，我们可以将地图划分为多个小块，通过相对坐标来表示玩家位置，从而降低同步位置数据所需的浮点数精度；

数据压缩 - AOI

概念：数据压缩 - 基于物体相关性

  • 概念：
    • 对象关联机制：玩家仅会接收到与其相关游戏对象的状态更新。通常，这些对象是玩家可见且可交互的；
    • 最简单的实现方式是所有客户端同步全部对象状态（适用于少量玩家场景），其复杂度为 ²，这种同步机制会成为限制最大同时在线玩家数量的关键因素；

    • 玩家/NPC 仅会与特定区域内的对象产生关联，只能看到并交互范围内的对象，通过此机制可有效过滤不必要的网络数据传输；

    • 计算当前需要同步的对象；
  • 概念：AOI - 空间划分网格

反作弊

作弊的方式

概念：反作弊的方式

  • 游戏代码篡改；
  • 修改或读取内存数据；
  • 破解客户端程序；
  • 系统软件非法调用；
  • Direct3D 渲染钩子；
  • 模拟鼠标键盘操作；
  • 网络数据包拦截；
  • 发送伪造数据包；
  • 篡改数据包内容；

反作弊的方式

概念：反作弊的方式

  • 概念：内存加壳
    • 打包器会对源代码进行混淆处理并附加解压代码，解压代码将优先执行，并在内存中对源程序进行解密；
  • 概念：网络包的截取和修改
    • 当游戏数据未加密或遭破解时，玩家甚至无需启动游戏即可基于数据包构建游戏逻辑。此类作弊程序往往演变为牟利工具，严重损害游戏的总体收益；
    • 两类加密算法：
    • 对称密钥算法
      • 使用相同密钥进行数据混淆与还原
      • 加解密速度快、效率高
    • 非对称加密
      • 加密与解密使用不同密钥
      • 处理速度较慢，仅用于加密关键数据
    • 图示：
      • 同时使用非对称和对称钥匙；
  • 概念：引擎注入
    • 该系统能检测游戏运行过程中因文件冲突引发的恶意行为，彻底阻止存在风险的玩家进入游戏，有效防范任何可能利用游戏漏洞的非法修改与配置变更；
  • 概念：AI 作弊
    • 支持全平台运行，无需修改代码，独立于游戏程序运行，通过游戏画面进行目标检测，自动移动准星并执行开火操作；
    • 收集用户的游戏数据（如胜率、暴击率等指标），通过对比自身历史数据与阈值规则，或参考其他玩家举报记录，对可疑玩家进行标记，最终通过人工审核确认是否存在作弊行为；
