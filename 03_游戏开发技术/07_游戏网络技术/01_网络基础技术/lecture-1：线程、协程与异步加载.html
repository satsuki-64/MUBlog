Unity 中的线程

Unity 多线程机制

概念：Unity 多线程机制

  • 概念：
    • Unity 引擎的核心操作（如渲染、物理计算、GameObject 的生命周期管理）必须在主线程执行；
    • 非主线程直接调用 Unity API（如 GameObject.Instantiate）会导致崩溃或未定义行为，因为 Unity 通过单线程设计保证内部状态一致性，避免多线程竞争问题；
  • 概念：主线程
    • 执行所有脚本逻辑（MonoBehaviour 生命周期方法如 Update、FixedUpdate、Start）；
    • 处理物理模拟（PhysX 引擎的同步调用）、动画状态机更新、场景对象管理（GameObject 创建/销毁/父子关系变更）；
    • 调度资源加载（Resources.Load）与场景切换（SceneManager.LoadScene）；
  • 概念：渲染线程
    • 渲染线程基于主线程提交的数据（如变换矩阵、材质属性）执行 GPU 指令，通常落后主线程 1 帧以实现双缓冲同步；
    • 主线程通过 CommandBuffer 将渲染指令封装为缓冲区，渲染线程仅读取该缓冲区，禁止反向修改主线程数据；
  • 概念：禁止跨线程操作
    • 绝大多数 UnityEngine API 仅在主线程有效。子线程调用会触发异常：
      • UnityException: get_transform can only be called from the main thread.
    • Unity 对象（如 GameObject）是 C++ 底层对象的托管封装，其生命周期和状态依赖主线程的单线程维护机制，直接跨线程访问可能破坏 C# 与 C++ 层间的对象桥接（如 IntPtr 指针失效）；

Task 与线程池

概念：C# 中的 Task

  • 分析：Task 与 Thread 区别
    • Task 是建立在线程池（ThreadPool）​​ 之上的高级抽象，其执行最终由线程池中的线程或操作系统线程承载，比如：Task.Run(() => { ... }) 会将任务提交到线程池队列，由池中的空闲线程执行；
    • 而 Thread 直接对应操作系统线程，需手动管理生命周期，资源开销大；
    • 因此，使用 Task 封装异步操作逻辑，自动复用线程池线程，减少资源消耗，支持返回值、取消、异常聚合等高级功能；
  • 概念：什么是 Task
    • Task 是任务并行库（Task Parallel Library, TPL）​的核心类型，其位于 System.Threading.Tasks 命名空间，代表一个可异步执行的操作单元；
    • 本质上，Task 是异步操作的逻辑抽象，​不绑定固定线程，而是按需调度到可用线程执行；
    • Task 具备以下特性：
        1. 跟踪任务状态：包括 Running、Completed、Faulted 等；
        1. 返回结果：Task<TResult> 泛型类型支持返回值（如 int result = await task;）
  • 代码：Task 的使用
// 1. 创建与启动
// 1.1 创建并立即启动任务
Task.Run(() => 
{
    Console.WriteLine("Task running on thread pool thread.");
});

// 1.2 延迟启动的任务
Task task = new Task(() => { /* 逻辑 */ });
task.Start();

// 2. 异步等待，并获取结果
// 异步等待（非阻塞）
public async Task ProcessDataAsync()
{
	// 将Calculate方法在calculationTask中开始执行
    Task<int> calculationTask = Task.Run(() => Calculate(42));
    
    // 等待calculationTask执行完毕、获得返回值，并且不阻塞主线程
    int result = await calculationTask; 
    Console.WriteLine($"Result: {result}");
}

// 同步等待（阻塞主线程）
Task.WaitAll(task1, task2); // 等待多个任务完成
int value = task.Result;    // 阻塞直到任务完成（慎用）

// 3. 异常处理

{
    await Task.Run(() => { throw new Exception("Error!"); });
}
catch (Exception ex)
{
    Console.WriteLine(ex.Message); // 捕获AggregateException内部异常
}

Unity 中的协程

基本概念

概念：协程

  • 概念：
    • 协同程序，简称协程；
    • 协程是“假”的多线程，它并不是多线程。它的主要作用将代码分时执行，不卡主线程；
    • 简单理解，是把可能会让主线程卡顿的耗时的逻辑，分时分步的执行；
    • 主要使用场景：异步加载文件、异步下载文件、场景异步加载、批量创建时防止卡顿；
  • 区别：
    • 新开一个线程是独立的一个管道，和主线程并行执行；
    • 新开一个协程是在原线程之上开启，进行逻辑的分时分步执行；
    • 并且在协程中可以创建死循环，并且不会阻塞主循环；
  • 举例：使用了几个 yield return，就表示将当前函数分成了几个部分，每个部分分时执行；
IEnumerator MyCoroutine(int i,string str)
{
	print(i)
	// 注意：协程函数当中，必须使用yield return 进行返回
	yield return new WaitrForSeconds(5f);
	print(str);
	yield return new WaitrForSeconds(1f);
	print("2");
	yield return new WaitrForSeconds(1f);
	print("3");
}

yield return 内容

概念：yield return 内容

  • 总结：
      1. 下一帧执行：
      • yield return 数字;
      • yield return null;
      • 在 Update 和 LateUpdate 之间执行；
      1. 等待指定秒之后执行：
      • yield return new WaitForSeconds(秒);
      1. 等待下一个固定物理帧更新时执行：
      • yield return new WaitForFixedUpdate();
      1. 等待摄像机和 GUI 渲染完成后执行：
      • yield return new WaitForEndOfFrame();
      1. 一些特殊类型的对象，比如异步加载相关函数返回的对象
      • 这些一般在 Update 和 LateUpdate 之间执行；
  • 概念：
    • 协程开启后组件和物体销毁，协程不执行；
    • 物体失活，协程不执行，组件失活协程执行

协程的原理

基础概念

概念：什么是协程

  • 概念：
    • 协程可以分成两部分：
        1. 协程函数本体；
        1. 协程调度器；
    • 协程本体就是一个能够中间暂停返回的函数；
    • 协程调度器是 Unity 内部实现的，会在对应的时机帮助我们继续执行协程函数；
    • Unity 只实现了协程调度部分协程的本体，协程本质上就是一个 C# 的迭代器方法；
  • 分析：
      1. 协程函数本体
      • 如果我们不通过开启协程方法来执行协程，Unity 的协程调度器是不会帮助我们管理协程函数的；
      • 但是我们可以自己执行迭代器函数内容；
      1. 协程调度器
      • 继承 MonoBehavior 后开启协程；
      • 相当于是把一个协程函数 (迭代器)放入 Unity 的协程调度器中，帮助我们管理进行执行；
      • 具体的 yield return 后面的规则也是 Unity 定义的一些规则；
  • 总结
    • 你可以简化理解，迭代器函数 C# 看到迭代器函数和 yield return 语法糖，就会把原本是一个的函数变成"几部分"，我们可以通过迭代器从上到下遍历这“几部分"进行执行，就达到了将一个函数中的逻辑分时执行的目的；

协程原理分析

概念：协程原理分析

  • 概念：协程的原理 - 状态机 + 迭代器
    • 当编译器检测到包含 yield return 的方法时，会将其自动转换为一个状态机类，该状态机内部通过 switch 语句管理协程的执行进度（每个 yield 对应一个状态分支）
    • 协程通过迭代器的 MoveNext() 方法逐步推进状态。每次调用 MoveNext() 时，状态机根据当前 _state 执行对应代码块，直到遇到下一个 yield 或者结束；
  • 概念：协程函数本体
    • 协程函数本体是一个返回类型为 IEnumerator 的方法，内部使用 yield return 定义暂停点；
    • 协程运行在主线程，通过分帧避免阻塞，但非真正的多线程，每次暂停时，局部变量和指令位置被保存在状态机中，恢复时还原上下文；
    • 常用暂停指令：
  • 概念：协程调度器
    • 调度器负责追踪所有运行的协程，根据 yield return 返回的指令决定何时唤醒协程（如帧结束、物理更新后），通过 StartCoroutine() 启动协程，StopCoroutine() 强制终止，或自动销毁关联 MonoBehaviour 时清理协程；
    • 工作流程：
        1. 注册协程：调用 StartCoroutine() 将协程迭代器加入调度队列；
        1. 帧循环检测：每帧遍历所有协程，检查其暂停条件是否满足
        • 若条件满足（如等待时间结束），调用 MoveNext() 推进状态机；
        • 若条件未满足，保留至下一帧检测；
        1. 终止处理：
        • 协程自然结束（MoveNext() 返回 false）时移出队列，手动调用 StopCoroutine() 或销毁 GameObject 时立即终止；

异步加载

Resource 异步加载

概念：Resource 异步加载

  • 引入：
    • 如果我们加载过大的资源可能会造成程序卡顿，卡顿的原因就是从硬盘上把数据读取到内存中是需要进行计算，越大的资源耗时越长，就会造成掉帧卡顿；
    • 异步加载 (Resources.LoadAsync) 在后台线程执行 IO 和数据处理，主线程仅负责调度，避免卡顿；
  • 特点：
      1. 延迟性：异步加载至少需 1 帧才能获取资源，不可立即使用；
      1. 线程安全：加载完成后，资源操作（如实例化）仍需回到主线程执行；
      1. 资源缓存：重复加载同一资源时，Unity 直接从内存缓存读取，避免重复 IO；
  • 概念：异步加载方法
    • 方法：
      1. 通过异步加载中的完成事件监听使用加载的资源；
      1. 通过协程使用加载的资源；

基于事件监听进行异步加载

概念：基于事件监听进行异步加载

  • 核心：
    • ResourceRequest rq = Resources.LoadAsync<类型>("路径");
    • rq.completed += Fun;
  • 概念：ResourceRequest 类
    • LoadAsyns 方法的返回值 ResourceRequest，ResourceRequest 是 Unity 引擎中用于异步加载资源的核心类，属于 UnityEngine 命名空间。它通过后台线程加载资源，避免主线程阻塞，提升程序流畅性；
    • ResourceRequest 通过 Resources.LoadAsync<T>() 方法创建，返回一个异步操作对象，允许在后台线程加载资源，分为“加载中”和“加载完成”两个状态，通过 isDone 属性和 progress（0.0~1.0）监控进度（这两个属性定义在 ResourceRequest 的父类 Asyncoperation 类当中）；
    • 其中 asset 表示的是返回的资源；
  • 概念：Asyncoperation 类
    • Asyncoperation 是 ResourceRequest 的父类；
    • 其中的 completed 是异步加载完成之后、会触发的事件；
    • 其中的 isDone 表示是否加载完成；
    • 其中 priority 表示加载的进度；
  • 代码：
void Start()
{
	// Unity 在内部 会去开一个线程进行资源下载
	ResourceRequest rq = Resources.LoadAsync<Texture>("Tex/TestJPG");
	
	// 马上进行一个资源下载结束 的一个事件函数监听
	rq.completed += Loadover;
}

// 被加入到completed事件中的方法，比如传入 Asyncoperation 类型的参数
void Loadover(Asyncoperation rg)
{
	print("1");

	// 传入的参数是 Asyncoperation 的子类 ResourceRequest
	// ResourceRequest 的成员变量 asset 表示的是异步加载得到的资源
	tex = (rg as ResourceRequest).asset as Texture;
}

概念：基于协程进行异步加载

IEnumerator Load()

IO 完成端口

概念：IO 完成端口

  • 概念：
    • I/O 完成端口提供了一个高效的线程处理模型，用于处理多处理器系统上的多个异步 I/O 请求；
    • 当进程创建 I/O 完成端口时，系统会为线程创建关联的队列对象，其唯一目的是为这些请求提供服务；
    • 处理许多并发异步 I/O 请求的进程可以将 I/O 完成端口与预先分配的线程池结合使用，而不是在收到 I/O 请求时创建线程，从而更快高效地执行此操作；
  • 特点：
      1. 异步队列管理：操作系统将完成的 I/O 请求（如网络数据接收、文件读写）放入一个先进先出（FIFO）队列，由线程池按后进先出（LIFO）顺序处理；
      1. 线程池优化：通过限制并发线程数（通常等于 CPU 核心数），减少线程上下文切换开销。线程在等待 I/O 时被挂起，仅在有任务时唤醒；
      1. 事件驱动：硬件（如网卡、磁盘控制器）通过中断通知内核 I/O 完成，内核再将事件推送至 IOCP 队列，完全绕过用户态线程的主动轮询；

.NET 与 IO 完成端口

分析：.NET 与 IO 完成端口

  • 分析：
    • .NET 通过以下方式将 IOCP 机制抽象为开发者友好的 API：
      1. async/await 语法糖：编译器将异步方法转换为状态机（IAsyncStateMachine），在 await 时释放当前线程，待 IOCP 通知完成后再恢复执行；
      1. Task 与 TaskCompletionSource：底层 I/O 操作（如 Dns.GetHostEntryAsync）返回 Task 对象，实际由 IOCP 回调触发任务完成状态更新；
  • 分析：.NET 与 IO 完成端口协作流程
      1. 发起异步请求：
      • 调用 HttpClient.GetAsync 时，.NET 通过 WinHttp 或 WinSock 发起异步 HTTP 请求，并将套接字句柄关联到 IOCP；
      1. 内核处理：
      • 网卡接收数据后，内核生成 I/O 完成包并推入 IOCP 队列，期间不占用任何用户态线程；
      1. 线程池响应：
      • .NET 线程池中的工作线程通过 GetQueuedCompletionStatus 获取完成事件，触发 Task 状态更新，恢复 await 后的代码执行；
      1. 状态机恢复：
      • 编译器生成的 MoveNext 方法根据状态机阶段继续执行，处理返回数据；
