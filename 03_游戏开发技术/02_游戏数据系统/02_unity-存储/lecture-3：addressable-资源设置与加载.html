Lecture 3：Addressable 资源设置与加载

可寻址资源设置

资源设置为可寻址资源

概念：资源设置为可寻址资源

  • 概念：方式一
    • 选中资源，勾选 Inspector 窗口中的 Addressable；
    • 把资源设置为可寻址资源之后，其默认放入 Default Local Group (Default) 组当中；
  • 概念：方式二
    • 选中资源，拖入 Addressables Groups 窗口中；
  • 注意：
      1. Resources 文件夹下资源如果变为寻址资源，会移入 Resources_moved 文件夹；
      • 原因是因为 Resources 文件夹下资源会最终打包出去，如果变为可寻址资源，意味着想通过 Addressables 进行管理；
      • 那么它就没有必要通过 Resources 方式去加载和打包，所以会自动迁移，避免重复打包，浪费空间；
      1. 代码不可以作为可寻址资源；

概念：右键 Addressables Groups 中的资源

  • 概念：
    • Move Addressables toGroup：将该资源放入到现有的另一个组中；
    • MoveAddressablestoNewGourp：使用与当前组相同设置创建一个新租，并将该资源放入该新组中；
    • RmoveAddressables：移除资源，该资源会变为不可寻址资源；
    • simplify AddressableNames：简化可寻址资源名，会删除名称中的路径和拓展，简化缩短名称；

    • CopyAddresstoClipboard：将地址复制到剪贴板；
    • Change Address：改名；
    • Create New Group：创建新组；

资源组窗口

概念：资源组窗口

  • 概念：资源信息面板
      1. GroupName\AddressableName：分组名\可寻址名（可重名，描述资源）
      • 第一列是资源的分组名称；
      1. Path：路径（不可重复，资源定位）
      • 这个路径是不可重复的，点击之后可以看到 Project 下的文件；
      1. Labels：标签（可重复、可用于区分资源种类，例如青铜装备、黄金装备）
      • 可以利用标签来快捷方便的加载资源；
  • 概念：创建分组相关
    • Create->Group
        1. Packed Assets：打包资源分组
        1. Blank（no shema）：空白（无架构）
    • 区别：Packed Assets 默认自带默认打包加载相关设置信息，Blank 没有相关信息，需要自己关联
    • 组对于我们来说意义重大，之后在资源打包时，一个组可以作为一个或多个 AB 包，关于组设置相关信息，之后详细讲解；
  • 概念：选中某一组后右键
    • 概念：
      • RemoveGroup：移除组，组中所有资源恢复为不可寻址资源；
      • SimplifyAddressableNames：简化可寻址名称，会删除名称中的路径和拓展，简化缩短名称；
      • SetasDefault：设置为默认组，当直接勾选资源中的 Addressable 时，会自动加入该组；
        • 默认组不能被删除，并且设置资源为 Addressable 时、默认放入这个组；
      • InspectGroupSetting：快速选中关联的组相关配置文件；
      • Rename：重命名；

        • 在实际开发时，可以不使用这种模式，这种模式没有测试的意义；
        1. Simulate Groups (advanced) :
        • 模拟组（后期），一般在测试阶段使用，分析布局和依赖项的内容，而不创建资产绑定；

        1. Use Existing Build （requires built groups)：
        • 正儿八经的远端加载资源，使用现有构建（需要构建组），一般在最终发布测试阶段使用，从早期内容版本创建的捆绑包加载资产；
        • 在使用此选项之前，必须使用生成脚本（如默认生成脚本）打包资源。远程内容必须托管在用于生成内容的配置文件的 RemoteLoadPath 上；
      • 即：前两种都不需要真正的打包 AB 包，第二种是模拟 AB 包。第三种是真正的使用 AB 包；
  • 概念：构建打包相关
  • 概念：资源分组
    • 一个分组代表了一个或者多个 AB 包，可以按照音效、武器、装备、动画等资源进行分组，按照实际的需求来定；
注意：资源名注意事项
    1. 资源路径一定不允许相同（后缀不同，名字相同可以）
    • 同一种类型的资源，只能有一个。不过如果后缀名不同也可以；
    1. 资源名我们可以随意修改；
    • 可以在资源名种随意的修改名称；
    1. 之后在加载资源时我们可以使用 名字+标签 作为双标识加载指定资源；
    • 使用名字和 Label 进行组合、加载资源；

加载资源

概念：加载资源

  • 概念：方法一 - 通过时间函数传入参数判断加载是否成功并创建
private void Start()
{
	// assetReference 可以异步加载任意的类型  
	// 由于是异步加载，因此需要使用一个 Handle 来获得当前的异步方法
	AsyncOperationHandle<GameObject> handle = assetReference.LoadAssetAsync<GameObject>();

	handle.Completed += Handle_Completed;
}

private void Handle_Completed(AsyncOperationHandle<GameObject> handle)
{
	// 如果加载成功，使用加载得到的资源
	if (handle.Status == AsyncOperationStatus.Succeeded)
	{
		GameObject.Instantiate(handle.Result);
	}
}
  • 概念：方式二 - 通过标识类本身来创建
// 不建议这种方法，因为这里面并没有资源的加载状态，只有一个 IsDone 属性
GameObject temp = assetReference.Asset as GameObject;

释放资源

概念：释放资源

  • 代码：
private void Start()
{
	// assetReference 可以异步加载任意的类型  
	// 由于是异步加载，因此需要使用一个 Handle 来获得当前的异步方法
	AsyncOperationHandle<GameObject> handle = assetReference.LoadAssetAsync<GameObject>();

	handle.Completed += Handle_Completed;
}

// 加载游戏对象与游戏材质，然后更改其所有子物体的材质为 assetReferenceMaterialRed 材质
private void Handle_Completed(AsyncOperationHandle<GameObject> handle)
{
	// 如果加载成功，使用加载得到的资源
	if (handle.Status == AsyncOperationStatus.Succeeded)
	{
		GameObject temp = GameObject.Instantiate(handle.Result);
		
		// 注意：一定要资源异步加载完成之后，再对 assetReference 资源标识进行释放
		// 释放资源不会影响被实例化的对象，但是会影响使用的引用资源
		assetReference.ReleaseAsset();
		
		MeshRenderer[] renderers = temp.GetComponentsInChildren<MeshRenderer>();
		assetReferenceMaterialRed.LoadAssetAsync().Completed += (materialHandle) =>
		{
			foreach (MeshRenderer meshRenderer in renderers)
			{
				meshRenderer.material = materialHandle.Result;
			}    
			
			// 由于 assetReferenceMaterialRed 和 GameObject 不一样，它不是实例化出来的资源，而是引用资源
			// 因此如果加载完成之后、立马将其释放，会导致资源的丢失
			// assetReferenceMaterialRed.ReleaseAsset();
		};
	}
}

总结

总结

    1. 我们可以根据自己的需求，选择合适的标识类进行资源加载；
    1. 资源加载和场景加载都是通过异步进行加载；
    1. 需要注意异步加载资源使用时，必须保证资源已经被加载成功，否则会报错；

Label 标签

引入：我们会经常使用指定资源加载的方式加载资源吗？

  • 并不会经常使用，要根据实际情况；
  • 之前学了加载指定资源，但是这种模式我们必须在脚本中声明各种标识类来指定加载的资源，不够灵活，做一些小项目没问题；
    • 比如：public AssetReference assetReference;
    • 然后根据这个 assetReference 来加载；
  • 但是在实际商业项目开发中，很多时候加载什么资源都是根据配置文件决定的，往往都是动态加载；
  • 所以我们需要学习根据名字或标签去加载对应的资源，这样我们就可以读表进行加载；

标签的作用

概念：标签的作用

  • 举例：
    • 游戏装备中有一顶帽子：Hat，但是它可以有不同的品质，比如：红、绿、白、蓝；
    • 那么我们可以为这个帽子添加多个材质球资源（或者贴图资源），这些图都可以叫做：Hat_Mat（或者 Hat_Tex）
    • 他们可以同名，我们可以通过标签 Label 来区分他们，他们的 Label 可以是：Red、Green、White、Blue
  • 举例：
    • 游戏中我们经常根据设备好坏来选择不同质量的图片或者模型，比如：高清、标清、超清；
    • 不同标准下的资源会有所不同，比如模型面数更低、贴图质量更低；
    • 但是在不同标准下，这些模型的命名应该是同样的，比如角色 1，在高清、标清、超清下它的资源名都是角色 1；
    • 它的 Label 可以是：HD、SD、FHD；
  • 举例：
    • 游戏中我们经常在逢年过节时更换游戏中模型和 UI 的显示，比如：中秋节、春节、圣诞节；
    • 不同节日时角色或者 UI 等等资源看起来是不同的，但是在不同节日下，这些资源的命名应该都是同样的规范；
    • 比如登录面板，在中秋节、春节、圣诞节时它的资源名都是登录面板，它的 Label 可以是：MidAutumn、Spring、Christmas；
  • 总结：
    • 相同作用的不同资源（模型、贴图、材质 UI 等等），我们可以让他们的资源名相同。通过标签 Label 区分他们来加载使用；
    • 利用名字和标签可以单独动态加载某个资源，也可以利用它们共同决定加载哪个资源；
