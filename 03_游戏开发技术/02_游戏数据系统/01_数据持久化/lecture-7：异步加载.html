Lecture 7：异步加载

两套主流异步方案

概念：AsyncOperation + 协程 方案

  • 概念：
    • 基于 IEnumerator 迭代器，通过 yield return 暂停执行并等待特定条件恢复。Unity 的协程调度器在主线程中管理协程的生命周期，每次 yield 后由引擎在下一帧或指定时间点恢复执行；
    • AsyncOperation 是 Unity 内置的异步操作类，通过协程的 yield return 等待操作完成。其底层依赖 Unity 引擎的异步任务队列，与主线程强绑定；
  • 线程模型：
    • 完全运行在主线程，协程和 AsyncOperation 的等待逻辑均由主线程处理，不会触发多线程操作。资源加载、场景切换等耗时操作由 Unity 引擎在主线程分帧执行，避免阻塞渲染；
  • 特点：
    • 优点：
      • 与 Unity 生命周期深度集成，协程中可以调用主线程的资源，自动随 MonoBehaviour 启用/禁用，对象销毁时协程自动停止，避免内存泄漏；
      • 通过 yield return 实现等待，代码直观；
    • 缺点：
      • 需通过回调或全局变量传递结果，代码结构易臃肿；
      • 大量协程同时运行会增加主线程调度压力，可能引发卡顿；

概念：async/await + Task 方案

  • 概念：
    • Task 基于 C# 的 System.Threading.Tasks 库，从线程池中取出一个线程来执行。async 方法通过 await 关键字挂起当前任务，允许控制权返回到调用线程（通常是主线程），待异步操作完成后恢复执行；
    • Unity 通过此上下文将 await 后的代码调度回主线程，确保 Unity API 的线程安全。非 Unity API（如文件 I/O、网络请求）可在后台线程执行；
  • 线程模型：
    • Task.Run 可将 CPU 密集型任务（如数据处理）分配到线程池，主线程仅处理回调。但涉及 Unity API 的操作（如 Instantiate）需通过 SwitchToMainThread 切换回主线程；
  • 特点：
    • 优点：
      • Task<T> 可直接返回结果，Task.WhenAll 实现并行任务，适合复杂逻辑（如多资源加载）
      • 通过 Task.Run 释放主线程压力，适用于计算密集型任务（如 AI、物理模拟）
    • 缺点：
      • 误用后台线程操作 Unity API（如 GetComponent）会引发崩溃，需严格切换回主线程；
      • 频繁创建 Task 对象可能增加内存分配；
总结：

异步方案：AsyncOperation + 协程

协程

分析：协程

  • 概念：
    • 协程不会阻塞主线程，允许其他游戏逻辑在这一帧内继续执行。并且协程的执行与 Unity 的主循环帧同步，当遇到 yield 语句后，协程在特定点挂起，Unity 保存其上下文环境，然后下一帧当中、Unity 在适当时机恢复执行；
  • 代码：

{
	StartCoroutine(DownLoadHttp());
}

	if (www.error == null)
	{
		image.texture = www.texture;
	}
}

public event Action<AsyncOperation> completed
{
	add
	{
		// 若异步操作已完成（isDone==true），立即执行回调
		if (this.isDone)
			value(this);
		else
			// 否则将回调加入m_completeCallback委托链
			this.m_completeCallback += value;
	}
	
	remove => this.m_completeCallback -= value;
}

概念：AsyncOperation 与协程

  • 概念：
    • AsyncOperation 是 Unity 引擎提供的基类，表示一个异步操作（如资源加载、场景切换等）。继承自 YieldInstruction，允许协程通过 yield return 等待其完成；
  • 关系：
    • AsyncOperation 需要协程的 yield return 来驱动执行流程，协程挂起时，AsyncOperation 在后台继续执行；完成后自动恢复协程，然后 AsyncOperation.Result 可将异步结果传递给协程后续逻辑；
    • Unity 的主线程负责执行协程迭代器（IEnumerator 的 MoveNext 方法）。当协程遇到 yield return AsyncOperation 时，引擎将控制权交还给主线程，同时注册回调监听异步操作完成事件；
    • AsyncOperation 对象由引擎内部管理，协程结束时自动释放相关资源；
  • 区别：和传统回调方案的区别
  • 代码：AsyncOperation 与协程协作模式

AsyncOperation 类与场景加载

概念：AsyncOperation 类与场景加载

  • 概念：
    • 使用 SceneManager.LoadSceneAsync 加载场景，将会返回一个 AsyncOperation 对象，然后可以通过 AsyncOperation 对象的 isDone 变量来判断当前场景是否加载完成；
  • 代码：
public void LoadSceneAsync(string name, UnityAction function)
{
	MonoMgr.GetInsance().StartCoroutine(ReallyLoadSceneAsync(name, function));
}

private IEnumerator ReallyLoadSceneAsync(string name, UnityAction function)
{
	// asyncOperation.progress 表示当前加载的进度
	AsyncOperation asyncOperation = SceneManager.LoadSceneAsync(name);
	while (asyncOperation.isDone == false)
	{
		// 往事件中心向外分发事件，外面可以更新
		EventCenterMgr.GetInstance().EventTrigger("进度条更新",asyncOperation.progress);
		
		// 在这里来更新进度条
		yield return asyncOperation.progress;
	}

	yield return asyncOperation;

	// 加载完成之后，再去执行回调的方法
	function.Invoke();
}

概念：ResourceRequest

public class ResourceRequest : AsyncOperation
{
	internal string m_Path;
	internal System.Type m_Type;
	
	protected virtual Object GetResult() => Resources.Load(this.m_Path, this.m_Type);

	// 核心：在外部可以通过 asset 变量来获得 Resources.Load 加载的结果
	// 底层通过 InternalSetManagedObject 确保资源加载完成后再访问
	public Object asset => this.GetResult();
	
	public ResourceRequest() {}
	
	protected ResourceRequest(IntPtr ptr) : base(ptr) {}
	
	internal new static class BindingsMarshaller
	{
		public static ResourceRequest ConvertToManaged(IntPtr ptr) => new ResourceRequest(ptr);
	}
}

public void LoadAsync<T>(string name,UnityAction<T> callback) where T : Object
{
	MonoMgr.GetInsance().StartCoroutine(ReallyLoadSceneAsync(name,callback));

private IEnumerator ReallyLoadSceneAsync<T>(string name,UnityAction<T> callback) where T : Object
{
	// 使用 Resources.LoadAsync 接口加载资源，返回一个 ResourceRequest 对象
	ResourceRequest request = Resources.LoadAsync<T>(name);
	
	// 协程执行到 yield return request 时暂停，AsyncOperation 在后台继续加载资源
	// 由于 Resources.LoadAsync 是异步操作，因此需要使用 yield 来等待 request 返回
	yield return request;

	// 当资源返回后，可以往下执行，并调用外部传入的回调
	if (request.asset is GameObject)
	{
		callback(GameObject.Instantiate(request.asset) as T);
	}
	else
	{
		callback(request.asset as T);
	}
}

async/await 结合 Task.Run 实现异步

概念：async/await 结合 Task.Run 实现异步

  • 代码：标准异步实现
// 1. 定义异步方法（返回 Task/T）
public async Task<TResult> ProcessDataAsync(byte[] rawData)
{
    // 2. 启动后台线程执行耗时操作
    var result = await Task.Run(() => 
    {
        // CPU密集型计算或同步API调用
        return ComputeHash(rawData);
    });
    
    // 3. 自动回到主线程处理结果
    return FormatResult(result);
}

// 4. 调用方法
public async Task UseAsyncMethodAsync()
{
    try
    {
        var result = await ProcessDataAsync(largeData);
        UpdateUI(result); // 主线程安全操作
    }
    catch (Exception ex)
    {
        Debug.LogError($"处理失败: {ex.Message}");
    }
}
  • 解释：
    • Task.Run 将任务排队到线程池，await 后自动返回原始上下文；
    • 主线程在等待期间可处理其他任务，避免界面卡顿，后台线程抛出的异常会封装到 AggregateException 中，并通过 try-catch 捕获；
  • 比较：与传统异步流程的区别
// 通过`async/await`将异步代码写成同步风格，避免回调地狱

// 传统回调方式
LoadDataAsync(path, (result) => { /* 处理结果 */ });

// async/await方式
var result = await LoadDataAsync(path);

从远端下载 AB 包资源

分析：从远端下载 AB 包资源

  • 分析：
    • Task.Run 用于将 CPU 密集型或同步操作转移到线程池执行，而 async/await 用于管理异步任务的非阻塞等待和上下文切换；
    • await 暂停当前方法执行，释放调用线程，待任务完成后自动恢复，并且 await 允许通过 try-catch 捕获后台线程的异常；
    • 若移除 async/await，代码将无法正确控制异步流程，导致以下问题：
      • Task.Run 在后台线程执行，但 .Result 会阻塞当前线程（如 Unity 主线程），形成死锁；
    • 而以下代码通过 async/await 与 Task.Run 正确协作，实现了以下目标：
  • 代码：
public async void DownLoadABFile(UnityAction<bool> overCallBacl,UnityAction<string> updateProcess)
{
	string localPath = Application.persistentDataPath + "/";
	int downLoadMaxNum = downLoadList.Count;
	int downLoadOverNum = 0;
	bool isOver = false;
	List<string> succeddLoadedList = new List<string>();

	while (downLoadList.Count > 0 && redownLoadMaxNum > 0) 
	{
		for (int i = 0; i < downLoadList.Count; i++) 
		{
			isOver = false;
			await Task.Run(() =>
			{
				// 执行耗时的下载操作，从远端服务器（比如FTP服务器）下载资源，可能被阻塞
				// 因此使用 Task.Run 从线程池取出线程来执行此耗时的操作
				isOver = DownLoadFile(downLoadList[i], localPath + downLoadList[i]);
			});
		
			// 如果当前资源下载成功，将其加入到succeddLoadedList列表当中
			if (isOver)
			{
				// 2. 要知道现在下载了多少，直到结束
				updateProcess(++downLoadOverNum + "/" + downLoadMaxNum);
				
				// Debug.Log("下载进度：" + (++downLoadOverNum) + "/" + downLoadList.Count);
				// Debug.Log(downLoadList[i] + "文件下载结束");
				succeddLoadedList.Add(downLoadList[i]);
			}
		}
		
		// 把下载成功的文件名，从待下载列表中移除
		for (int i  = 0; i < succeddLoadedList.Count; i++) 
		{
			downLoadList.Remove(succeddLoadedList[i]);
		}

		--redownLoadMaxNum;
	}

	overCallBacl(downLoadList.Count == 0);
}

Socket 的异步机制

概念：Socket 的异步机制

  • 概念：基于事件的异步模式（Event-Based Async）
    • Socket 的 ConnectAsync 方法采取了基于事件的异步模式，通过订阅 Completed 事件处理异步操作结果，使用 XXXAsync 方法启动异步操作（如 ConnectAsync、ReceiveAsync），通过事件回调（如 Completed）处理完成后的逻辑；
    • 而异步操作由底层 Socket 库在后台线程执行，事件回调在 线程池线程触发，而非主线程；
    • 因为 ConnectAsync 和 ReceiveAsync 已经是 Socket 封装好后的异步方法，调用后立即返回，不阻塞当前线程，因此不需要配合 await 方法；
    • Socket 的异步操作（如连接、接收）由系统自动分配到线程池执行，无需手动切换线程；
  • 区别：
  • 代码：
public void Connect(string ip, int port)
{
	if (socket != null && socket.Connected)
	{
		return;
	}

	IPEndPoint ipep = new IPEndPoint(IPAddress.Parse(ip), port);
	socket = new Socket(AddressFamily.InterNetwork, SocketType.Stream, ProtocolType.Tcp);
	
	// 使用 Socket 的 Async 结尾的方法，需要先声明 SocketAsyncEventArgs
	SocketAsyncEventArgs agrs = new SocketAsyncEventArgs();
	agrs.RemoteEndPoint = ipep;
	agrs.Completed += (socket, args) =>
	{
		if (args.SocketError == SocketError.Success)
		{
			Debug.Log("客户端连接成功");
			
			// 连接成功之后，客户端开始接收消息
			SocketAsyncEventArgs receiveArgs = new SocketAsyncEventArgs();
			receiveArgs.SetBuffer(receiveCache, 0, receiveCache.Length);
			receiveArgs.Completed += ReceiveCallBack;
			this.socket.ReceiveAsync(receiveArgs);
		}
		else
		{
			Debug.Log("客户端连接失败" + args.SocketError);
		}
	};
	
	// 开启异步操作
	socket.ConnectAsync(agrs);
}

Addressable 的异步机制

概念：Addressable 的异步机制

  • 概念：
    • Addressables 的所有资源加载、依赖解析、资源包下载与实例化等操作都是异步的，返回 AsyncOperationHandle 以用于管理操作的生命周期；
    • 调用方通过 await handle.Task、yield return handle 或 handle.Completed 事件在主线程获得结果。底层会先解析依赖、必要时下载远程 AssetBundle、将所需 AssetBundle 加载到内存，最终把请求的对象填充到 Result；
  • 分析：Addressables.LoadAssetAsync<GameObject>
    • 你的代码在主线程调用例如 Addressables.LoadAssetAsync<GameObject>，该方法立即返回一个未完成的 AsyncOperationHandle，此时不会阻塞主线程；
    • Addressables 的资源系统会把耗时工作尽量放到后台线程执行，典型包括：解析地址/定位器、枚举依赖、检查并下载远程 AssetBundle、读取解压 AssetBundle 元数据、从磁盘/网络流中读取资源数据等。这些阶段可能跨多个帧分布执行，以减少主线程卡顿；
