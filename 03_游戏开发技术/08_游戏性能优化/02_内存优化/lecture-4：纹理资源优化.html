纹理大小

概念：纹理大小分析与设置

  • 选择合适纹理大小应尽量遵循以下经验：
  • （1）纹理大小通常是 2 的幂次方，比如 1024*1024、512*256，如果不符合这个比例，则 Unity 会自动为其适应纹理大小，但这样可能会造成资源的浪费；
  • （2）不同平台、不同硬件配置，应当选择不同的纹理大小，Unity 下可以采用 bundle 变体设置多套资源、通过 Mipmap 限制不同平台加载不同 level 层级的贴图，这两个都是 Unity 提供的现成的解决方案；
  • （3）根据纹理用途的不同选择不同的纹理加载方式，如流式纹理加载 Texture Streaming、稀疏纹理 Sparse Texture、虚拟纹理 VirtualTexture 等方式；
  • （4）不能让美术人员通过增加纹理大小的方式，来增加纹理细节。可以选择细节贴图 DetailMap 或增加高反差保留的方式；
  • （5）在不降低视觉效果的情况下，尽量减小贴图大小，最好的方式是：纹理映射的每一个纹素的大小正好符合屏幕上显示像素的大小，如果纹理小了会造成欠采样，纹理显示模糊，如果纹理大了会造成过采样纹理显示噪点；
    • 这一点做到完美很难保障，可以充分利用 SceneView->DrawMode->Mipmap 来查看在游戏摄像机视角下哪些纹理过采样，哪些纹理欠采样来调整纹理大小；

纹理颜色空间

概念：纹理颜色空间

  • 默认大多数图像处理工具都会使用 SRGB 颜色空间处理和导出纹理；
  • 但如果你的纹理不是用作颜色信息的话，那就不要使用 SRGB 空间，如金属度贴图、粗糙度贴图或者法线贴图等；
  • 一旦这些纹理使用 SRGB 空间，会造成视觉表现错误；

纹理压缩

概念：纹理压缩

  • 概念：为什么要对纹理压缩
    • 纹理压缩是指图像压缩算法，保持贴图视觉质量的同时，尽量减小纹理数据的大小；
    • 默认情况下我们的纹理原始格式采用 PNG 或 TGA 这类通用文件格式，但与专用图像格式相比，他们访问和采样速度都比较慢，无法通用 GPU 硬件加速，同时纹理数据量大，占用内存较高；
    • 所以在渲染中我们会采用一些硬件支持的纹理压缩格式，如 ASTC、ETC、ETC2、DXT 等；
  • 图示：
  • 概念：不同纹理压缩格式
    • ETC：Ericsson Texture Compression，ETC1 ​ 是初始版本，一个主要限制是不支持透明通道（Alpha 通道）​。对于不透明的纹理（如漫反射贴图、法线贴图），它是非常安全的选择。主要用于需要最大限度兼容老旧 Android 设备的不透明纹理，对于带透明度的纹理，需要将颜色和 Alpha 通道拆分成两张 ETC1 纹理，这会增加工作量和不便；
    • ETC2：ETC 1 的升级版，是 ​ OpenGL ES 3.0 ​ 的标准功能，支持透明通道（Alpha 通道）​，并且压缩质量比 ETC1 更好，​ Android 开发的首选格式，可以统一设置为 ETC2，并为不支持 ETC2 的老设备在 ​​“ETC2 fallback”​​ 选项中设置一个后备格式（如 ETC1），这样就能在保证新设备高质量的同时兼容老设备；
    • ASTC：提供从高压缩率（低质量）到低压缩率（高质量）的多种块大小选项（如 4x4, 6x6, 8x8 等），开发者可以根据纹理内容在质量和大小之间进行精细权衡。支持透明通道和各种颜色格式，质量通常优于 ETC2，需要设备硬件支持（并非所有 Android 设备都支持，但近年来主流中高端芯片均已支持）；
    • DXT：S3 Texture Compression，也称为 S3TC，是 PC 和游戏主机（如 Xbox, PlayStation）上的传统标准格式，​主要用于 PC（Windows/Mac/Linux）和主机平台的游戏开发。在移动平台（Android/iOS）上应避免使用；
    • PVRTC：PowerVR Texture Compression，是苹果设备（iOS, tvOS）芯片（历代 A 系列芯片）的专属原生格式，在 iPhone 和 iPad 上具有最佳的兼容性和性能；

纹理图集

概念：纹理图集

  • 概念：
    • 纹理图集是一系列小纹理图像的集合；
  • 特点：
    • 优点：
      • 一是采用共同纹理图集的多个静态网格资源可以进行静态合批处理，减少 DrawCall 调用次数；
      • 二是纹理图集可以减少碎纹理过多因为他们打包在一个图集里，通过压缩可以更有效的利用压缩，降低纹理的内存成本和冗余数据；
    • 缺点：
      • 美术需要合理规划模型，并且要求模型有相司的材质着色器，或需要制作通道图去区分不同材质。制作和修改成本较高；

纹理 Mipmap

概念：纹理 Mipmap

  • 概念：
    • 逐级减低分辨率来保存纹理副本。相当于生成了纹理 LOD，渲染纹理时，将根据像素在屏幕中占据的纹理空间大小选择合适的 Mipmap 级别进行采样；
  • 特点：
    • 优点：
      • GPU 不需要在远距离上对对象进行全分辨率纹理采样，因此可以提高纹理采样性能；
      • 同时也解决了远距离下的过采样导致的噪点问题，提高的纹理渲染质量；
    • 缺点：
      • 由于 Mipmap 纹理要生成低分辨率副本，会造成额外的内存开销；

纹理导入设置与优化

纹理导入基础设置

概念：纹理导入基础设置

  • 概念：Alpha Source
    • 默认选择 Input Texture Alpha 就好，如果确定不使用原图中的 Alpha 通道，可以选择 None；
    • 另外 From Gray Scale 我们一般不会选用；
  • 概念：Alpha ls Transparency
    • 指定 Alpha 通道是否开启半透明；
    • 如果位图像素不关心是否要半透明，可以不开启此选项，这样 Alpha 信息只需要占 1 bit，节省内存；
  • 概念：Ignore Png file gamma
    • 是否忽略 png 文件中的 gamma 属性，这个选项是否忽略取决于 png 文件中设置不同 gamma 属性导致的显示不正常，一般原图制作流程没有特殊设置，这个选项一般默认就好；

纹理导入的 Mipmaps 相关设置

概念：纹理导入的 Mipmaps 相关设置

  • 概念：Generate Mipmaps
    • 是否生成 Mipmaps 的选项。通常情况都开启，除非时是 2D 场景或者是固定视角的场景（摄像机无法缩放远近）
  • 概念：Border Mip Maps
    • 默认不开启，只有当纹理的是 Light Cookies 类型时，开启此选项来避免 Color bleeding 现象导致颜色渗透到较低级别的 Mip Level 纹理边缘上；
  • 概念：Mip Map Filtering
    • Mip Map 的过滤选项，其中 Box 最简单，随尺寸减小，Mipmap 纹理变得平滑模糊；
    • Kaiser，避免平滑模糊的锐化过滤算法；
  • 概念：Mipmaps Preserve Coverage
    • 只有需要纹理在开启 mipmap 后也需要做 AlphaCoverage 时开启，默认不开启；

纹理过滤设置

概念：纹理过滤设置

  • 概念：
    • （1）用双线性过滤平衡性能和视觉质量；
    • （2）有选择地使用三线性过滤，因为与双线性过滤相比，它需要更多的内存带宽；
    • （3）使用双线性和 2x 各向异性过滤，而不是三线性和 1x 各向异性过滤，因为这样做不仅视觉效果更好，而且性能也更高；
    • （4）保持较低的各向异性级别。仅对关键游戏资源使用高于 2 的级别；

其他纹理优化技术

概念：其他纹理优化技术

  • （1）尽可能将类似生命周期的小纹理打包到同一图集；
  • （2）不合理的半透明纹理，将为造成 Overdraw 与内存开销；
    • 如下图所示的半透明纹理会占据屏幕的大量区域，造成 Overdraw 与内存开销；
    • 这种情况应当从 UI 设计层面来避免；
  • （3）过多的序列帧 2D 动画，而且不将其打成图集，将造成大量浪费；
  • （4）大量颜色渐变贴图不压缩，而且也不采用单像素宽度；
    • 比如：
    • 渐变贴图的特点是颜色变化平滑连续，相邻像素的颜色值非常接近。传统的块压缩格式（如 ASTC、ETC2）是为处理颜色突变的复杂图像设计的，对平滑渐变可能产生可见的色带或块状瑕疵。但不压缩意味着放弃了大幅减少内存占用和带宽压力的机会，是一种巨大的浪费；
    • 这种情况下，贴图可以替换为曲线数据，或采取单像素梯度纹理，以减少纹理开销；
    • 单像素纹理指的是：对于一条从左到右的线性渐变，理论上只需要一横排像素来记录颜色信息。因此，可以将纹理的高度设置为 1，创建一张 N x 1的纹理（如 256x1）。这样，纹理体积就缩小为原来的 1/N（对比 N x N的正方形纹理）

补充知识

AssetBundle 变体

概念：AssetBundle 变体

  • 概念：
    • AssetBundle 变体（Variant）是 Unity 提供的功能，它允许你为同一组逻辑资源创建多个不同的版本；
    • 这些版本共享相同的 AssetBundle 主名称，但通过一个称为“变体”的后缀来区分。在运行时，你可以根据当前平台、硬件配置、语言设置等条件，动态加载最合适的那个变体，而无需修改加载资源的代码逻辑；
  • 图示：加载过程
  • 分析：
    • AssetBundle 变体的核心在于，​不同变体包中的资源必须保持一一对应关系。这意味着，如果你有一个名为 Hero 的预制体在 hd（高清）变体中，那么在 sd（标清）变体中也必须有一个同名、同结构的 Hero 预制体；
    • Unity 依靠这种一致性，确保你在切换变体时，只需要关心加载哪个“后缀”的包，而不需要修改资源加载路径或逻辑；
    • 因此游戏代码只需要知道资源的逻辑名称（如 "UI/Icon"），通过一个配置变量控制当前使用的变体后缀（如 "hd" 或 "sd"），就能加载到正确的资源。当需要增减画质档次或适配新平台时，主要工作是准备新资源并设置变体，核心代码无需大改；

纹理加载方式

概念：纹理加载方式

  • 概念：流式纹理加载
    • 动态加载/卸载整个 Mipmap 层级。根据物体与相机距离，仅将当前所需的 Mip 层级（如 Mip 5）及其相邻层级加载到显存；
    • 其避免一次性将纹理的所有 Mipmap 层级（占原始纹理 4/3 倍内存）全部加载，优化显存；
    • 这种方式需要纹理本身开启 Mipmaps。不适合 UI 等需要始终保持最高精度的纹理；
  • 概念：稀疏纹理加载
    • 将巨型纹理分割成小块（Tile）​，仅将当前可见区域的图块加载到显存，处理单张超大尺寸纹理​（如 8192 x 8192 或更大），使其无需全部驻留显存；
  • 概念：虚拟纹理加载
    • 将多个高分辨率纹理打包到一个图集（Atlas）中，并分割成图块（Tile），仅流式传输可见的图块；
    • 处理大量高分辨率纹理的密集场景，实现极致的显存利用，允许场景使用的纹理总量远超物理显存；
    • 其运行时开销较大。涉及运行时图块打包、映射查询，着色器采样需要额外的间接查找步骤；
    • 仅支持高清渲染管线（HDRP），设置复杂，对性能预算有要求；

纹理的异性过滤

概念：纹理的异性过滤

  • 引入：Mipmaps 的问题
    • 当三维模型上的一个表面，比如游戏中的地面，其与相机屏幕平面不平行时，根据透视原理，这个表面在屏幕上的投影会发生变形。此时一个屏幕像素在纹理空间中所对应的区域不再是一个正方形，而是一个被拉长的近似椭圆形​（这个区域被称为 Footprint）；
    • 而常规的 Mipmap 具有局限​：传统的双线性和三线性过滤技术，在处理这种情况时，会简单地认为这个 Footprint 是一个正方形。它们会选择一个大致能覆盖这个“正方形”的 Mipmap 层级；
    • 更关键的是对于被拉长的椭圆区域，如果不使用异性过滤技术，会导致这些拉长了区域使用了过低的 Mipmap 层级（即更高层、更模糊的纹理），从而丢失大量细节，使得倾斜视角的纹理看起来非常模糊；
  • 概念：什么是纹理的异性过滤
    • 各向异性过滤认识到了 Footprint 的非正方形（各向异性）​​ 特性，它会沿着椭圆的长轴方向进行多次采样​（例如 4 次、8 次或 16 次），然后将这些采样的结果进行混合，最终得到当前像素的颜色。这样就能在保持抗锯齿效果的同时，最大限度地保留纹理的细节和清晰度；
  • 分析：不同过滤模式以及使用异性过滤后的效果

分析：Unity 中 ​Aniso Level​ 选项

  • Unity 中的 ​ Aniso Level​ 选项（取值范围通常为 1 到 9）直接控制着这个采样次数的上限：
    1. Aniso Level 1 ​：这相当于关闭各向异性过滤；
    1. Aniso Level 2/4/8/16 ​：这些级别（如 2x, 4x, 8x, 16x）代表各向异性过滤的最大采样倍数；
    • 例如，设置为 16x 时，意味着 GPU 最多会沿着 Footprint 的长轴方向进行 16 次采样，来计算一个像素的最终颜色；
  • 各向异性过滤会显著增加纹理采样的次数，因此对 GPU 的内存带宽有更高的要求，可能带来性能开销；

半透明物体与 Overdraw

分析：半透明物体与 Overdraw

  • 概念：透明与不透明物体的渲染顺序
    • 一般的不透明物体的渲染​依靠深度测试（Z-Test）​，管线会先对不透明物体进行排序（通常是从前向后），并为每个像素记录一个深度值。当渲染后续物体时，如果某个片元的深度值比当前记录的深度值更远（即被遮挡），它就会被直接丢弃，避免了不必要的着色计算。这是一种非常高效的像素级剔除机制；
    • 但是半透明物体是从后向前渲染。因为半透明物体需要与背景颜色进行混合（Blending），​不能使用深度写入（Z-Write）​，如果写入深度，就会阻挡后面本该透过它显示出来的物体。因此，半透明物体的渲染必须严格按从后向前的顺序进行绘制；
    • 而这个顺序通常以物体为单位进行排序，但一个物体的不同部分可能存在交叉，因此无法像不透明物体那样进行精确的每像素深度剔除；
  • 分析：大尺寸半透明物体
    • 一张巨大的半透明纹理，比如全屏的烟雾、水面或者 UI 面板，意味着它本身就会覆盖成千上万的屏幕像素；
    • 由于半透明渲染需要从后向前，当多个这样的半透明层（如粒子特效、多层 UI）叠加在一起时，​这些像素就会被反复计算和混合；
    • 每一次混合操作都涉及读取当前帧缓冲区的颜色和写入新的结果，这会消耗大量的内存带宽；
