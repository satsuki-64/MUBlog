性格优化概论

性能优化的步骤

概念：性能优化的步骤

  • 概念：性能优化的步骤
    • 第一步：确定经常会出现哪些问题 – Profile
    • 第二步：确定在哪些方向进行性能优化 – Analyze
    • 第三步：再尽可能将问题逐个解决 – Solve
  • 分析：
    • 因此，在性能优化当中，一定是先学会工具 Profile，再进行优化；
    • 常见的工具有一些是引擎和 IDE 自带的，比如 Unity 自带的 Profiler，就包含了 CPU，GPU，Memory 等等各式各样的性能分析工具，其他的比如 GPA，Xcode Instrument 和 Visual Studio，Intel 自带的内存管理工具在必要的时候也使需要去学习和使用；
    • 另外一些工具，就需要根据游戏的需求去编写了，比如一键关闭所有特效，一键更改分辨率等等，一键设置场上 NPC 数量，简单的游戏如啪啪三国是做成快捷键开启 Profile 功能的，更为复杂的游戏如神秘海域则是通过游戏内控制台来进行更为细致的 Profie；

性能优化的组成

概念：性能优化的组成

  • 总结：
    • 性能优化主要由以下几个部分组成：
      1. CPU；
      1. GPU；
      1. 内存；
      • 以上三个部分是任何类型的游戏都会涉及到的优化。部分涉及大量网络 IO 和文件流操作的，还有以下优化：
      1. IO 和网络；
  • 概念：CPU 优化
    • 问题：
      • 由于短时间内的计算量太大，导致画面流畅性降低，俗称跳帧；
      • 发热严重，耗电量高；
    • 方法：
      • （1）将计算分到多个逻辑帧中进行计算，避免短时间内的性能超过负荷，俗称“分帧” time-slice；
      • （2）将可以缓存的数据尽可能的缓存起来，避免重复计算和重复分配内存，常见的示例为“内存池”；
      • （3）使用合理的算法和数据结构，比如：冒泡排序和直接插入排序在整体数组比较有序的情况下效率大大好于快速排序。把快排替换成是优化程序排序效率的一个常见的思路；
  • 概念：GPU 优化
    • 问题：
      • 发热严重，耗电量高；
      • FPS 降低；
    • 方法：
      • （1）优化美术资源：比如合理规划图集，约定好模型的最大三角形面数，制定合理的粒子效果规范。这个可以说是游戏优化中最重要的一个，因此，技术美术在游戏开发中作用巨大；
      • （2）简化或者优化着色器（shader），如在游戏开始前就对 Shader 进行编译和加载；
      • （3）进行合批，尽量减少 DrawCall，例如静态合批、动态合批、GPU 实例化；
      • （4）使用平台推荐的压缩格式，比如安卓平台的 ETC1 和 IOS 平台的 PVRTC；
  • 概念：内存
    • 问题：
      • 闪退和卡死，比如安卓的 Low Memory Killer 会在低内存情况下杀掉内存占用过大的程序；
    • 方法：
      • （1）动态加载和卸载资源，比如在游戏内的时候，我们可以把游戏外的一些 UI 图集卸载掉；
      • （2）降低资源质量或屏幕分辨率，降低纹理、模型、音频等资源的内存占用，这是内存开销的大头。这是有损优化，一般作为最后的手段；
      • （3）托管堆内存优化：管理 C# 脚本分配的内存，避免托管堆无限膨胀或频繁垃圾回收（GC）导致卡顿；
  • 概念：IO 与网络优化
    • 问题：
      • 网络延迟甚至掉线；
      • 加载资源导致的跳帧；
      • 加载时间过长；
    • 方法：
      • （1）使用独立的线程进行加载，有些引擎如 Unity 中还能利用协程；
      • （2）网络包资源优化；
      • （3）合并小包，减少请求数据的次数；
      • （4）分帧对回包进行处理、限制一定时间内的发包频率；

项目总览

概念：项目总览

  • 打开新项目后，做以下几个步骤，明确一些优化方向；
    1. 在 Project 按照不同资源类型进行筛选，观察不同类型的资源的分别数量，比如音频、材质、脚本、Shader、预制体等等；
    1. 搜索常见灯光的数量，这影响光照复杂度和阴影复杂度；
    1. 搜索摄像机数量，这直接影响渲染流程；
    1. 观察当前场景的图形相关设置，比如渲染路径；
    1. 观察当前场景的运行情况，比如当前的平均三角形面数、三角形面数峰值、渲染批次、SetCall 等；
    1. 进行烧录真机测试，关闭移动平台的垂直同步选项；
    • 观察有无明显渲染错误、帧率等；
    1. 使用 Profiler 查看下真机运行时的内存情况与性能耗时分布；
    • 内存使用情况；
    • 显存使用情况；
    • 纹理、音效、模型的分布；
    • 各个线程的 CPU 占时；

Unity 完整事件时序

概念：Unity 完整事件时序

  • 图示：
  • 概念：
    • 主线程 (Main Thread)：主要负责游戏逻辑，如 Update (), FixedUpdate ()，物理计算，动画状态机等；
    • 渲染线程 (Render Thread)：专门负责处理渲染相关的指令，将主线程准备好的渲染数据（顶点、材质、纹理等）转换成 GPU 能理解的命令列表（Command Buffer），并提交给 GPU 执行；
    • 其中虚线的部分表示非当前帧的逻辑；
  • 概念：主线程详解
    • 主线程的时间轴被几个关键的 [wait] 阶段分隔开，这揭示了线程间的同步点；
    • （1） Logic & scripting - 逻辑与脚本
      • 这是新一帧的开始。主线程执行所有游戏逻辑代码，例如 Update() 和 LateUpdate() 函数、物理引擎计算碰撞和刚体运动（物理也可以在独立的线程中运算，但结果依然需要同步），在此阶段，游戏世界的最新状态（物体的位置、旋转、缩放等）被确定下来；
    • （2）[wait] WaitForPendingPresent
      • 这是第一个关键的同步点。主线程的逻辑计算完成后，需要将渲染指令交给渲染线程。但渲染线程可能还在处理上一帧的命令，因此主线程会等待，直到收到渲染线程“我已准备好接收新命令”的信号。这个等待是为了确保线程安全，避免数据竞争；
      • 如果这个 [wait] 时间很长，通常意味着渲染线程是瓶颈。渲染线程过于繁忙，导致主线程不得不空等；
    • （3）FinishFrameRendering - 完整帧渲染
      • 一旦通过 WaitForPendingPresent 同步，主线程就开始全力构建本帧的渲染数据。这个阶段包含两个核心子任务：
        1. ScriptableRenderContext.ExtractAndExecuteRenderPipeline：提取并执行渲染管线；
        • 这是 Unity 可编程渲染管线（URP/HDRP）的核心；
        • 引擎根据当前摄像机和渲染设置，收集所有需要渲染的物体，并按照渲染管线的各个 Pass（如深度预处理、阴影绘制、不透明物体渲染、透明物体渲染等）来组织渲染队列；
        1. Write graphics commands (GfxDeviceClient)：写入图形命令
        • 将上一步组织好的渲染队列，转换成一系列具体的、底层的图形 API 指令（如 DirectX 或 Vulkan 的“DrawCall”命令）
        • 这些命令不会直接发送给 GPU，而是先写入到一个名为命令缓冲区 (Command Buffer)​ 的内存区域中；
    • （4）[wait] WaitForLastPresentationAndGetTimestamp, WaitForTargetFrameRate
      • 这是第二个关键的同步点。主线程已经完成了本帧所有渲染指令的提交，此时，主线程会等待两个事件：
          1. 等待 GPU 完成上一帧的最终呈现 (Present)：确保 GPU 已经将上一帧画面显示到屏幕上，避免“屏幕撕裂”；
          1. 等待达到目标帧率：如果设置了垂直同步 (VSync) 或帧率限制，主线程会在此等待，直到满足下一帧开始的时间点，以控制帧率、节省功耗；
      • 如果这个 [wait] 时间很长，通常意味着 GPU 是瓶颈，GPU 绘制速度太慢，导致 CPU 必须等待；
  • 概念：渲染线程详解
    • （1）Gfx.PresentFrame：
      • 这是渲染线程一帧的结束动作。它负责将上一帧已经由 GPU 处理完成的最终图像，提交到屏幕进行显示；
    • （2）[GPU Work]：
      • 这表示 GPU 正在执行渲染线程之前提交的命令；
      • 注意，这个工作是在 GPU 上并行进行的，而 CPU 线程（包括渲染线程）在此期间可以继续做其他事情。图中特意标出，说明渲染线程在提交命令后，可以继续准备下一批命令，而不必等待 GPU 完成；
    • （3） Read & execute graphics commands (GfxDeviceWorker) - 读取并执行图形命令
      • 这是渲染线程的核心工作。一个名为 GfxDeviceWorker 的工作线程，从主线程写入的命令缓冲区中读取图形命令，然后，它将这些命令进行优化、组合，并最终通过图形 API（如 DirectX, OpenGL, Vulkan）提交给 GPU 的指令队列；
      • GPU 随后会异步地执行这些命令；
      • 性能提示：这个阶段耗时过长，通常意味着 DrawCall 数量过多或渲染状态切换过于频繁，导致渲染线程成为瓶颈；
