Lecture 3：性能优化工具

性能优化工具是什么

概念：性能优化工具是什么

  • 概念：
    • 性能分析工具主要指 Unity 提供给我们的自带的很多分析性能问题的工具，主要集中在 Unity 工具栏的 Window（窗口）页签的 Analysis（分析）选项中；
  • 概念：分析工具
    • Profiler：分析器；实时查看 CPU、GPU、内存、UI、音频等模块的性能；
    • Profiler（StandaloneProcess）：独立运行的分析器；独立运行的 Profiler 进程，避免自身性能干扰；
    • MemoryProfiler：内存分析器；抓取内存快照、分析内存使用、泄漏、GC 等；
    • Frame Debugger：帧调试器；分析一帧的 Draw Call、材质切换、渲染指令；
    • Physics Debugger：物理调试器；检查物理组件状态、触发器、重叠问题；
  • 概念：Unity 中的其他内容
    • ImportActivity：导入活动；查看资源导入过程中的详细日志；
    • PerformanceMarkers：性能标记；查看 Unity 内部系统的 Profiler 标记点；
    • InputDebugger：输入调试器；查看 UnityInput System 的输入事件、设备状态；
    • RenderingDebugger：渲染调试器；HDRP/URP 开发时调试渲染效果与性能；
    • RenderGraphViewer：渲染图像查看器；分析 URP/HDRP 渲染顺序与资源依赖；
    • IMGUIDebugger：GUI 调试器；编辑器工具开发、IMGUIDebugUI 修复；
  • 总结：最重要的三个工具
      1. Profiler：分析器；实时查看 CPU、GPU、内存、UI、音频等模块的性能；
      1. Frame Debugger：帧调试器；分析一帧的 Draw Call、材质切换、渲染指令；
      1. MemoryProfiler：内存分析器；抓取内存快照、分析内存使用、泄漏、GC 等；

Unity Profiler

Unity Profiler 两种分析器窗口的区别

概念：两种分析器窗口的区别

  • 概念：
    1. Profiler：
    • 内嵌在 Unity 编辑器中的性能分析工具，默认使用，方便调试；
    • 它同属 UnityEditor 进程，会对游戏性能造成一定干扰（特别是深度采样或低端设备）
    • 更适合日常开发调试、小项目、局部功能分析；
    1. Profiler (Standalone Process)
    • 独立运行的分析器，可最大限度减少对被分析游戏的性能干扰，更适合真实测试；
    • 更适合复杂场景、大型项目、真机性能分析；

CPU Usage

CPU Usage 基础概念

概念：CPU Usage 基础概念

  • 概念：
    • CPU Usage（使用率），是 UnityProfiler 中最重要的性能分析模块之一；
    • 用于展示每一帧 CPU 所花费的时间，按类别详细划分不同系统的消耗情况，帮助开发者诊断性能瓶颈；
  • 概念：CPU 使用率窗口
    • 作用：
      • 在这里通过不同模块不同颜色表达，我们可以直观的看到每一个模块对 CPU 的使用情况；
    • 图示：
    • 概念：
      • Rendering：花费多少时间准备渲染命令，如 Camera.Render、DrawCalls 生成阶段；
      • Scripts：花费多少时间执行 C# 脚本（如 Update、Start、协程)；
      • Physics：花费多少时间进行物理计算，包括碰撞检测、刚体模拟（2D/3D）
      • Animation：花费多少时间处理动画相关，比如带有蒙皮网格渲染器、Animation、Animator 等等和动画相关的组件；
      • GarbageCollector：花费多少时间处理 GC（垃圾回收），即 GC 垃圾回收造成的性能抖动；
      • VSync：花费多少时间等待垂直同步完成，可能会造成卡帧；
      • GlobalILLumination：花费多少时间来处理实时全局光照；
      • UI：花费多少时间来处理 UI 相关，比如 Canvas 更新、重建、批处理等 UI 绘制任务；
      • Others：花费多少时间来处理不属于上述模块的其他任务，比如 Unity 编辑器本身的消耗或未分类的任务或引擎底层消耗；
  • 概念：CPU 详细信息模块 - TimeLine
    • 概念：TimeLine
      • TimeLine 在视图中可以显示所有线程的性能分析数据，而 Hierarchy 只能在视图中显示一个线程的性能分析数据，需要我们手动切换线程；
      • TimeLine（时间线视图）显示特定帧的时间细分信息，以及该帧长度的时间轴；
    • 图示：
      • 在 Timeline 当中，可以看到各个线程的消耗；
    • 作用：
      • 看每个函数调用在时间上所占的位置和长度，清楚显示函数调用之间是否并发/串行，非常适合排查主线程阻塞、线程切换耗时等问题；
  • 概念：CPU 详细信息模块 - Hierarchy
    • 概念：
      • 按时间数据的内部层级结构对这些数据分组，显示方式以调用栈的方式展示函数调用结构，从根函数一路展开；
      • 结构是每一行是一个函数调用，子函数缩进在其下方；
    • 作用：
      • 快速发现“树顶函数”调用最多的分支，定位哪一个函数调用下挂着最多性能开销；

Hierarchy 下关键参数

概念：Hierarchy 下关键参数

  • 概念：Live
    • 开启后可在模块详细信息面板中显示有关当前帧或选定帧的信息，关闭后，只有选择一帧时才会显示；
  • 概念：Main Thread 或 Render Thread
    • 一般只需要看主线程或者渲染线程；
  • 概念：Total
    • 占该帧总时间的百分比，表示当前函数（含其所有子函数）总耗时占该帧总耗时的百分比，通常用来判断这一支调用链是否值得深挖；
  • 概念：Self
    • Self 表示此函数占该帧总时间的百分比，并且其相比于 Total 而言，其不包含 Unity 调用子函数所花费的时间；
    • 比如：某个函数花费大量时间去执行，但是是因为它调用了大量的系统的绘制和剔除函数。但是，当排除它调用的函数的样本时，只有 0.2% 的时间花在该函数本身上；
    • 其可用于判断函数本身的逻辑是否耗时大；
  • 概念：Calls
    • 在此帧中该函数的调用次数，高调用频率可能是性能瓶颈的来源；
  • 概念：GCAlloc
    • 此函数在该帧中触发的 GC 内存分配量（单位：Byte），过高可能导致频繁 GC；
  • 概念：Time ms
    • 当前函数（含子函数）在该帧的总耗时，单位是毫秒；
  • 概念：Self ms
    • 在该帧中自身花费的总时间，不包含 Unity 调用子函数所花费的时间；
  • 概念：EditorLoop
    • Unity 编辑器运行时的主循环（非 Play 模式下），编辑器中的各面板更新，Ul 重绘，资源导入，编译，插件逻辑等等都会在该循环中执行；
  • 概念：PlayerLoop
    • 游戏运行时的主循环，几乎所有的游戏逻辑、渲染、物理、输入、协程、动画等等都；
    • 在这个循环中执行，展开后，就是在该次循环中执行的函数相关信息；
  • 图示：
  • 概念：详细信息
    • 概念：
      • NoDetails：无详细信息
        • 不显示任何额外信息，仅显示左侧的函数调用树及其时间、GC、调用次数等；
      • Related Data：
        • 相关数据；显示与当前选中函数相关的上下文信息
        • 内联事件（Markers）、线程关联、调用位置的上下文等等；
        • 这个对分析线程间协作和事件调度有帮助，但使用较少
      • Calls：调用
        • “调用”视图显示了所选示例的调用来源以及它调用的其他函数；
        • 用于分析该函数是在哪个逻辑路径下被调用的，可以帮助你定位谁在调用它，尤其在频繁调用的函数追溯调用源时很实用；
    • 总结：使用建议：
        1. 只关心函数耗时，使用 NoDetails 即可；
        1. 想找出某函数是被谁调用，使用 Calls；
        1. 想看函数在不同线程或上下文的分布，使用 RelatedData；

PlayerLoop

概念：PlayerLoop

  • 解释：PlayerLoop
    • 解释：
      • Unity 游戏运行时的核心循环驱动器。它本身不包含具体逻辑，而是一个容器或调度框架，负责按预定义的顺序调用所有其他子系统（如物理、渲染、输入等）
    • Application.FlushRenderStats：
      • 刷新和收集渲染相关的统计信息（如三角形数量、绘制调用次数等），以便在 Game 视图的状态栏或 Profiler 中显示。在发布版本中这项开销通常极小；
    • RenderPlayModeViewCameras：
      • 这是编辑器模式下的核心渲染入口。当您在 Unity 编辑器中播放游戏时，它负责协调和执行所有 Game 视图摄像机的渲染流程。它会调用底层的渲染管线；
      • Camera.Render：
        • 单个摄像机渲染的传统入口点。在旧的 Built-in 渲染管线中，这是主要的渲染调用。在现代可编程渲染管线（URP/HDRP）中，它通常会将实际渲染工作委托给 RenderPipelineManager；
      • DestroyCullResults：
        • 在摄像机完成渲染后，清理和销毁上一帧的视锥体剔除结果；
        • 剔除会计算哪些物体在摄像机的视野内，这些数据在渲染完成后需要被释放，为下一帧的准备；
      • FrameEvents.NewInputBeforeRenderUpdate：
        • Unity 新一代输入系统（Input System）的事件回调；
        • 这标志着开始处理新一帧的输入信息，并确保这些输入数据在渲染更新之前就准备就绪，供脚本查询；
      • FrameEvents.OnBeforeRenderCallback：
        • 一个通用的“渲染前”回调点；
        • Unity 内部的一些系统（如 UI、动画）或某些插件可以在此刻执行需要在摄像机渲染前完成的逻辑；
      • FrameEvents.XRBeginFrame：
        • 如果项目启用了 VR/AR（统称 XR）功能，这个回调用于处理 XR 设备帧渲染开始前的特定准备工作，例如更新头盔位姿等；
      • Gfx.EndAsyncJobFrame：
        • 与图形 API（如 DirectX, Vulkan, Metal）的异步操作相关；
        • 它负责结束或同步当前帧中提交给 GPU 的异步计算任务（如异步上传纹理、缓冲区等），确保 CPU 和 GPU 之间的协调；
      • PlayerEndOfFrame：
        • 标志着一帧中所有游戏逻辑和渲染的正式结束；
        • 这是帧生命周期的最后阶段之一，常用于执行一些需要在帧末处理的逻辑；
      • UGUI.Rendering.EmitWorldScreenspaceCameraGeometry：
        • Unity 内置 UGUI 系统的渲染部分；
        • 对于配置为 "World Space"或"Screen Space - Camera"的 Canvas，此方法负责将 UI 元素的几何数据发射到渲染管线中，以便被指定的摄像机渲染；
      • UnityEngine.Rendering.RenderPipelineManager.DoRenderLoop：
        • 现代渲染管线（URP/HDRP）的核心渲染方法；
        • 它接管了 Camera.Render 的具体工作，执行整个可编程渲染管线的流程，包括渲染阴影、不透明物体、透明物体、后处理等所有 Pass，比如光照计算等等；
      • 另外三个用于激活渲染管线实例、处理渲染请求、特性检测的方法；
    • UpdateScene：
      • 包含一系列的 EarlyUpdate 的方法，用于：
        1. 更新与分析核心（Analysis Core）相关的统计信息；
        1. 清理中间渲染器：用于清除在上一帧中临时使用但本帧不再需要的渲染资源；
        1. 清除由 Debug.DrawLine 等调试方法绘制的调试线；
        1. 分发事件队列中的事件：这是 Unity 新一代输入系统的核心部分；
        1. 执行已安排到主线程的作业（Jobs）
        • 虽然 Unity 使用 C# Job System 进行多线程计算，但有些任务必须在主线程执行（如与渲染 API 交互），这个方法就是执行这些排队的主线程任务；
        1. 等等；

CPU Usage 的作用

概念：CPU Usage 的作用

  • 概念：
      1. 找 CPU 性能瓶颈
      • 查看每帧中哪个模块耗时异常（颜色块明显突起），快速定位卡顿来源，哪帧卡了、哪个模块卡了；
      1. 找性能抖动：
      • 检查 GC 是否频繁（黄色峰值），还是某个帧的脚本逻辑执行消耗过大（蓝色占比高）
      1. 找优化依据
      • 明确是脚本、渲染、物理还是 VSync（垂直同步）等模块引起的卡顿；
      1. 帧率诊断
      • 对比 16 ms（60 FPS）、33 ms（30 FPS)标尺，判断是否达标；
  • 补充：VSync
    • 显示器通常以固定的频率刷新画面，而显卡渲染图像的速度通常不固定，有时快于显示器刷新，有时慢于；
    • 垂直同步（VSync）会强制显卡
        1. 等显示器准备好再提交新帧（即等一次“垂直回扫”信号），避免屏幕撕裂
        1. 如果显卡渲染太快，它要等一等
        1. 如果渲染太慢，它只能跳过一帧，延后一轮提交
    • 影响帧率主要有两种情况
        1. 帧率略高于刷新率→强制等待
        • 比如显卡渲染能力是 90 FPS，而显示器是 60 Hz；
        • VSync 开启后，GPU 只能每 16.67 ms 渲一帧→被“卡"在 60 FPS
        1. 帧率低于刷新率→被“降档”处理
        • 这会造成严重卡顿或掉帧感；

GPU Usage

GPU Usage 基础概念

概念：GPU Usage 基础概念

  • 概念：
    • GPU Usage（使用率）是 UnityProfiler 中最重要的性能分析模块之一，用于分析 GPU（图形处理器）在一帧中在哪些渲染阶段上花了多少时间；
    • 游戏开发中会造成 GPU 开销的主要有：
        1. 几何处理：模型顶点相关处理
        1. 光照计算：阴影、反射等
        1. 渲染输出：Shader 处理、对象渲染、游戏画面显示
        1. 屏幕后处理 Shader 部分：屏幕画面特殊效果（模糊、黑白、景深等等）
        1. 纹理处理：采样、解压缩、贴图操作等
        1. 特效处理：粒子、流体、烟雾等等
      • 等等
    • 项目出现卡顿、掉帧问题、图像问题、发热、掉电快等情况时，可以着重排查 GPU 和 CPU 使用情况，它们的异常表现几乎一致，根本原因通常是计算压力过大；
  • 分析：GPU 和 CPU 计算的区别
    • GPU 和 CPU 计算的区别在于：
    • CPU 专注于游戏逻辑处理相关（物理、脚本、AI、动画控制等等）
    • GPU 专注于图形渲染处理相关（Shader、模型绘制、光照、特效、后处理等等）
注意：GPU Usage 注意事项
    1. GPU Usage 的数据收集会影响性能表现；
    1. 会禁用 GPU Jobs（图形工作线程优化）；
    1. 并不支持显示详细的 URP 和 HDRP 相关数据；

GPU Usage 各参数功能的含义

概念：GPU Usage 概念

  • 概念：
    • 采集 GPUProfiler 数据会禁用图形作业、增加性能开销，并降低 CPU 模块的准确性，如果你不需要这些数据，请关闭此模块；
    • HDRP 和 URP 渲染器当前不受支持，如果启用了这些渲染器，Profiler 将不会显示大多数 GPU 标记；

      1. GraphicsJobs 是 Unity 用于加速渲染流程的多线程技术，开启 GPUProfiler 会关闭它，影响性能测试的真实性；
      1. 在使用 URP（通用渲染管线）或 HDRP（高清渲染管线）时，GPU 模块不会显示详细的 GPU 时间采样
      • 比如各个 Pass、Shader 的耗时，所以数据不完整；
  • 建议：
      1. 如果你只关心 CPU 性能或者使用 SRP（如 URP/HDRP），可以暂时关闭 GPUProfiler；
      • 除非你在测试传统内置渲染管线的 GPU 性能；
      1. 对于 SRP（URP/HDRP）我们可以使用 RenderingDebugger 进行性能检测；

Rendering

Rendering 是什么

概念：Rendering 是什么

  • 概念：
    • Rendering（渲染）模块用于显示一帧渲染时的数据量（批次数、顶点数等）
    • 在游戏中如果出现以下问题时，可以着重观察这里的内容
        1. 怀疑是因为 DrawCa 1 l 过高导致 cPU 性能问题（卡顿、掉帧）
        1. 在中低端手机上发热严重、掉电快
        1. 开启大量实时光照、粒子、动态对象后出现掉帧
        1. 画面复杂卡顿、简单不卡顿时
      • 等等
  • 概念：Rendering 的作用
    • Rendering 可以帮助我们
      1. 定位 CPU 渲染压力的关键依据
      • 批处理次数多，意味着 DrawCall 高，意味着 CPU 消耗高；
      • 材质切换次数，意味着 CPU 频繁设置材质，意味着可能带来批处理失败；
      1. 判断 GPU 处理压力是否过大
      • 三角形面数、顶点数都会直接应该 GPU 渲染耗时；
      • 数据量过大往往是因为模型太复杂，粒子特效过多，动态对象过多；
      1. 判断画面复杂度对性能的影响
      • 当进入不同场景、摄像机角度变化、特效显示隐藏；
      • 都可能带来各种数据的变化，这些数据可以让性能下降有据可查；
      1. 分析批处理的策略是否生效
      • 看看使用了静态、动态批处理后，批处理参数是否还是很高等等

Rendering 各个参数的含义

概念：Rendering 各个参数

  • 概念：BatchesCount（批处理数）
    • 该帧内批处理数量（和 DrawCall 次数很相关）
    • 值越高，CPU 压力越大，需要合批优化；
  • 概念：SetPassCallsCount（SetPass 调用次数）
    • 该帧内用于染游戏对象的着色器染通道（Pass）调用次数；
    • 即每帧需要切换 Shader /材质状态的次数，值越高，GPU 状态切换开销大。尽量减少材质和渲染通道 Pass 的数量
  • 概念：TrianglesCount（三角形数量）：
    • 该帧内渲染了多少个三角形（所有网格顶点组成的面），影响 GPU 处理负荷，特别是在移动端需控制数量；
  • 概念：VerticesCount（顶点数量）
    • 该帧内染了多少个顶点（所有网格顶点数量），影响 GPU 顶点处理性能；

Memory

Memory 模块基础概念

概念：Memory 模块基础概念

  • 概念：
    • Memory（内存）模块是 Unity 中用于监视内存使用情况的关键工具；
    • 它能帮助你识别哪些资源在占用大量内存，是否存在内存泄漏或垃圾回收频繁等问题；
    • 游戏开发中会造成内存开销的主要有：
      1. 纹理贴图
      1. 网格
      1. 音频/视频
      1. 动画
      1. 材质球和 Shader
      1. C# 托管堆内存
      1. 场景中的各 Gameobject 以及各组件；
    • 等等
  • 概念：
    • 在游戏中如果出现以下问题时，可以着重观察这里的内容
      1. 游戏卡顿、掉帧
      • 表现：游戏突然变慢、不定期小卡顿、操作延迟等；
      • 可能原因：内存不足导致系统频繁进行垃圾回收（GC）降低游戏帧率；
      1. 闪退、崩溃
      • 表现：玩着玩着突然黑屏退出、特定场景（如加载大地图）崩溃；
      • 可能原因：设备分配不出更多内存，系统强制杀掉进程；
      1. 加载时间变长
      • 表现：加载界面等待时间变长，场景切换明显卡顿；
      • 可能原因：加载资源时需要更多内存，系统调度与 GC 频繁运行；
      1. 发热严重、掉电快
      • 表现：手机发烫严重、电池消耗快；
      • 可能原因：设备内存使用达到极限，系统/游戏频繁调度内存资源，CPU、GPU 都高负载运转；
      1. 低端设备无法运行；
      • 表现：打开即闪退、进主界面就崩溃
      • 可能原因：老旧设备的可用内存更少，超出后甚至无法加载首屏；

Memory 各参数功能的含义

概念：Memory 各参数功能的含义

  • 概念：
      1. TotalUsedMemory：
      • 已使用的内存总量（包括纹理、网格、对象、脚本等所有内存），是最直观的内存压力指标；
      1. Texture Memory：
      • 贴图纹理使用的内存总量（包括 Mipmap），通常是内存占用的最大部分；
      1. Mesh Memory：
      • 网格使用的内存总量（顶点、索引缓冲等），和场景中物体数量密切相关；
      1. Material Count：
      • 材质数量，材质越多越难合批、内存占用也高；
      1. Object Count：
      • 内存中的 UnityEngine.Object 实例总数，包括贴图、网格、脚本、GameObject 等，过高可能意味着资源未卸载干净；
      1. GCUsed Memory：
      • GC（托管堆）使用的内存，堆已分配但仍被使用；
      1. GCAllocated InFrame：
      • 该帧分配的托管堆内存
  • 概念：使用建议：
      1. 排查 GC 问题时，可以重点观察 GCAllocatedInFrame 是否频繁跳动；
      1. TextureMemory 如果过高，要检查是否贴图分辨率太大、未压缩、重复加载；
      1. MeshMemory 可反映模型复杂度，是否存在不合理的高面数网格；
      1. ObjectCount 超高时，注意是否存在对象未正确卸载，是否存在未销毁的临时 GameObject；
