Lecture 4：动画优化

动画导入设置与优化

Rig 下动画导入设置

概念：Rig 下动画导入设置

  • 概念：Animation Type
    • None 无动画；
    • Legacy 旧版动画，不要用；
    • Generic 通用骨骼框架；
    • Hunmannoid 人形骨骼框架；
  • 方法：选择原则
      1. 无动画选择 None；
      1. 非人形动画选择 Generic；
      1. 人形动画：
      • （1）人形动画需要 Kinematices 或 Animation Retargeting 功能，或者有自定义骨骼对象时选择 Humanoid Rig；
      • （2）其他都选择 Generic Rig，在骨骼数差不多的情况下，Generic Rig 会比 Humanoid Rig 节省 30% 甚至更多的 CPU 的时间；

Rig 下动画关键设置

概念：动画关键设置

  • 概念：Skin Weights
    • 默认 4 根骨头，但对于一些不重要的动画对象可以减少到 1 根，节省计算量；
  • 概念：Optimize Bones
    • 建议开启，在导入时自动剔除没有蒙皮顶点的骨骼；
  • 概念：Optimize Game Objects
    • 在 Avatar 和 Animatior 组件中，删除导入游戏角色对象的变换层级结构，而使用 Unity 动画内部结构骨骼，消减骨骼 transform 带来的性能开销；
    • 开启这个选项可以提高动画角色性能，但有些情况下会造成角色动画错误，这个选项可以尝试开启，但要看表现效果而定；
    • 注意：如果你的角色是可以换装的，在导入时不要开启此选项，但在换装后在运行时代码中通过调用 AnimatorUtility.OptimizeTransformHierarchy 接口仍然可以达到此选项效果；

补充：关于 Optimize Game Objects 选项

  • 一个带骨骼的模型（如人形角色）导入 Unity 后，其每一根骨骼都会在场景中创建一个对应的 GameObject，这些 GameObject 组成了复杂的层级树；
  • 动画系统每帧都需要遍历并更新这成百上千个 GameObject 的 Transform 组件​（位置、旋转、缩放），这个过程主要在 CPU 主线程上进行，涉及大量的矩阵计算和层级遍历，​ CPU 开销巨大。而勾选 Optimize Game Objects 后，Unity 在导入时会做两件事：
    • （1）删除层级​：
      • 将场景中那些只包含 Transform 组件的骨骼 GameObject 从层级视图中移除；
    • （2）使用内部骨架​：
      • 角色的 SkinnedMeshRenderer 不再依赖于场景中的 GameObject 层级，而是直接使用 Unity 动画系统内部维护的一个高度优化的骨骼矩阵数组；
      • 动画系统直接在内部内存中计算和更新所有骨骼的最终变换，这个过程效率更高，并且更容易实现多线程优化；

Animation 设置

概念：Animation 设置

  • 概念：Resample Curves
    • 将动画曲线重新采样为四元数值，并为动画每帧生成一个新的四元数关键帧，仅有当导入动画文件包含欧拉曲线时才会显示此选项；
  • 概念：Anim.Compression
    • 当开启任意一种压缩选项之后，都会出现三个 Error 选项；
    • 其表示角度、缩放、位置的关键帧之间的区别小于此数值时、表示为吧被压缩的关键帧；

动画文件优化后的检查原则

概念：动画文件优化后的检查原则

  • 一看效果差异 (与原始制作动画差异是否明显)；
  • 二看曲线数量 (总曲线数量与各种曲线数显，常量曲线比重大更好)；
  • 三看动画文件大小 (动画文件在小几百 k 或更少合理，查过 1M 以上的动画文件考虑是否合理)；

编辑器内动画资源优化

Animation 注意事项

概念：Animation 注意事项

  • （1）播放单个 AnimationClip 速度，Legacy Animation 系统更快，因为老系统是直接采样曲线，并直接写入对象 Transform；
    • 而新版本的系统会使用用于混合的临时缓冲区，并会对采样曲线和其他数据进行额外的复制操作；
  • （2）针对动画的缩放曲线比位移、旋转矩阵开销，因此尽量少对对象制作缩放的动画；
  • （3）更大常数曲线不会每帧写入场景，更高效

Animator 及其使用优化

概念：Animator 及其使用优化

  • 概念：Animator 内部动画更新流程
  • 流程：
    • （1）State machine update ​：
      • 这是流程的起点，Unity 会更新所有动画状态机（Animator Controller）；
      • 它检查所有配置的转换条件​（Parameters），并根据条件决定是否要从当前状态切换到下一个状态；
    • （2）OnStateMachineEnter/Exit ​：
      • 如果上一步发生了状态切换，Unity 就会在这一步触发相应的事件；
      • 例如，从“待机”状态切换到“奔跑”状态，会先调用“待机”状态的 OnStateExit，再调用“奔跑”状态的 OnStateEnter。这是你执行状态相关初始化或清理代码（如播放脚步声）的关键时机；
    • （3）ProcessGraph​：
      • 这个过程会计算并混合所有动画状态和子状态机的权重，最终确定一个“当前帧应该显示的最终动画姿势”；
    • （4）​ Fire animation events ​：
      • 如果当前播放的动画片段（Animation Clip）上绑定了动画事件，Unity 会在此刻检查并触发这些事件；
    • （5）StateMachineBehaviour callbacks ​：
      • 如果状态机或某个状态上挂载了自定义的 StateMachineBehaviour 脚本，Unity 会调用其中的回调，如 OnStateUpdate，让你可以编写更复杂的状态相关逻辑；
    • （6）OnAnimatorMove ​：这是处理根运动的关键环节
      • 如果脚本中没有覆写此方法，Unity 会使用动画本身的位移来移动游戏对象；
      • 如果你在角色的脚本中覆写了这个方法，Unity 就会将动画产生的位移和旋转数据交给你处理。你可以完全控制角色如何移动，例如，将动画位移施加到自己的角色控制器上，而不是直接改变 Transform；
    • （7）ProcessAnimation ​：
      • 这是动画系统的核心计算环节，Unity 根据上一阶段确定的最终动画姿势，进行骨骼变换计算，计算出角色模型中每一根骨骼在当前帧应该处于的准确位置和旋转；
    • （8）OnAnimatorIK​：
      • ​逆向运动学回调。如果开启了 IK，并且 IK 权重 > 0，就会调用此方法；
      • 它允许你修改角色的手、脚等部位的位置和旋转，使其与环境互动，例如让脚踩在台阶上、让手抓取物体。​此回调的执行结果会覆盖掉 ProcessAnimation 中计算出的相应部位的姿势；

概念：Animator 使用注意事项

  • （1）不要使用字符串来查询 Animator；
  • （2）使用曲线标记来处理动画事件；
  • （3）使用 Target Marching 函数来协助处理动画；
  • （4）将 Animator 的 CullingMode 设置成 Based On Renderers 来优化动画，并禁用 SkinMesh Renderer 的 Update When Offscreen 属性来让角色不可见时动画不更新；

Animation 与 Animator 对比

分析：Animation 与 Animator 对比

  • （1）Animation 可以将任何对象属性制作成 Animation Clip，Animator 是将 AnimaitonClip 组织到状态机流程图中使用；
  • （2）Animation 与 Animator 播放动画时的效率是有个临界点的，这个临界点是根据动画曲线条数来的，当动画曲线条数小于这个临界点时 Animation 快，当动画曲线条数大于这个临界点时 Animator 快；
  • （3）当 CPU 核数较少时，Animation 播放动画有优势，当 CPU 核数较多时，Animator 表现会更好；
  • （4）Animator Controller Graph 中的所有动画节点的 Animation Clip 都会载入到内存中当有海量动画状态机节点时，内存开销较大；

Playable API

概念：什么是 Playable API

  • 概念：
    • Playables API 是以一套树形结构来组织数据源，并允许用户通过脚本来创建和播放自定义的行为，支持与动画系统、音频系统等其他系统交互，是一套通用的接口；
  • 图示：
  • 特点：
    • Playable API 优点：
      • 支持动态动画混合，可为场景中的对象提供自己的动画，并可以动态添加到 PlayableGraph 当中使用；
      • 允许创建播放单个动画，而并不会产生创建和管理 AnimatorControler 资源所涉及的开销，可更加灵活的控制 PlayableGraph 的数据流，可以插入自定义的 AimationJob；
      • 可以控制动画文件加载策略，按需加载、异步加载等；
      • 允许用户动态创建混合图，并直接逐帧控制混合权重 (甚至可以混合 AniationClip 与 AnimatorController 动画)；
      • 可以运行时动态创建，根据条件添加可播放节点。而不需要提前提供一套 PlayableGraph 运行时启动和禁用节点，可以做到自由度更高的 override 机制；
      • 可加载自定义配置数据，更加方便的和其他游戏系统整合；
    • Playable APl 缺点：
      • 没有直接使用 Animator 直观；
      • 混合模式没有现成的，需要自己实现；
      • 需要开发更多的配套工具；
      • 有一定学习成本；

总结：解决方案选择

总结：解决方案选择

  • （1）一些简单、少量曲线动画可以使用 Animation 或动画区间库如 Dotween\iTween 等完成，如 Ul 动画，Transform 动画等；
  • （2）角色骨骼蒙皮动画如果骨骼较少，Animation Clip 资源不多，对动画混合表现要求不高的项目，可以采用 Legacy Animation。注意控制总体曲线数量；
  • （3）一些角色动画要求与逻辑有较高的交互、并且动画资源不多的项目，可以直接用 Animator Graph 完成；
  • （4）一些动作游戏，对动画混合要求较高、有一些高级动画效果要求、动画资源量庞大的项目，建议采用 Animator + Playable API 扩展 Timeline 的方式完成；
