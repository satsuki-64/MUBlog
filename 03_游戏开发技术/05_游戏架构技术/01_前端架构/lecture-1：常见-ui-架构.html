Lecture 1：常见 UI 架构

Retained Mode

什么是 Retained Mode

概念：什么是 Retained Mode

  • 概念：直接模式
    • 像在一张画布上作画。画家（应用程序）需要每时每刻记住每个图形的位置、颜色，并且当需要改变（比如移动一个图形）时，必须擦掉整个画布，然后重新绘制所有图形，因此这非常低效；
  • 概念：保留模式 - Retained Mode
    • 像使用一个智能的图形编辑器，创建一些图形对象（如按钮、文本框）后，编辑器会保留所有这些对象及其状态（位置、颜色、文字）。当需要改变其中一个时，只需修改那个对象，编辑器会自动计算出需要重绘的部分，并高效地更新屏幕；
    • 它的核心是 ​​“面向对象”​​ 和 ​​“状态保留”​​：
      1. 面向对象：UI 中的每个元素（按钮、文本框等）都是一个对象。如：Button button = new Button();，我们创建了一个按钮对象；
      1. 状态保留​：这个按钮对象自己持有与它相关的所有状态和数据，例如它的文字 setText("Hello!")、宽高 setWidth(12)、以及点击后的回调函数 SetCallback(&callback_func)；

工作流程

概念：工作流程

  • 概念：
    • 因为 UI 框架（如 QT, WPF）保留着所有小部件的完整信息，所以它不需要应用程序持续地发号施令。其工作流程是：
    • （1）初始化​：应用程序启动时，创建所有 UI 对象并设置好它们的初始状态；
    • （2）事件循环​：UI 框架进入一个事件循环，监听用户输入（点击、键盘等）；
    • （3）按需更新：
        1. 当有事件发生时（比如用户点击了按钮），框架会触发对应的回调函数​（callback_func）；
        1. 在回调函数中，应用程序可能只修改某一个按钮的文字或颜色。UI 框架会智能地判断出哪个部分需要更新，然后只重绘屏幕上发生变化的那一小块区域，而不是整个界面；
  • 特点：
    • 优点：易于添加新的 UI 组件，因为它们都是独立的对象。框架负责优化绘制，避免了不必要的全局重绘，效率更高；
    • 缺点：开发者需要理解消息队列/事件循环的概念，而不是直接控制绘制过程；
  • 举例：
    • Unreal UMG​：虚幻引擎的 UI 系统，所有控件都是可复用的蓝图或 C++ 对象；
    • WPF GUI ​：微软 .NET 平台的现代 UI 框架，完全基于组件和数据绑定；
    • QT GUI​：一个跨平台的 C++ 框架，其信号与槽（Signal & Slot）机制是处理回调的典范；

MVC 架构

概念：MVC 架构

  • 概念：M - Model
    • 模型代表应用程序的核心数据和业务逻辑。它不关心用户界面如何显示，只负责管理数据的状态和行为。其主要职责是封装应用程序的数据、定义数据处理逻辑（如数据验证、计算、持久化存储）、并在数据改变时通知观察者；
    • 独立于 UI ​：模型不持有或引用任何视图或控制器对象，这保证了其可复用性；
    • 可观察​：通常通过事件（如 C# 的 Action 或 event）或观察者模式，在数据变更时通知所有依赖方；
  • 概念：V - View
    • 视图是用户直接交互的界面，负责将模型中的数据以特定形式呈现给用户，并接收用户的输入；
    • 渲染用户界面、展示模型的数据、捕获用户的操作（如点击、输入）。视图应当尽可能“被动”，即它知道如何显示数据，但通常不包含决定显示什么数据的复杂逻辑；
    • 视图会订阅模型的变化事件，并据此自动更新显示，保持与模型数据的一致性；
    • 视图负责检测用户输入，但不应处理输入背后的业务逻辑。它通常将用户输入事件（如按钮点击）转发给控制器处理；
  • 概念：C - Controller
    • 控制器是模型和视图之间的协调者，充当桥梁角色。它接收来自视图的用户输入，调用模型进行业务处理，然后指示视图更新显示；
    • 其解释用户输入、调用模型的相应方法改变数据状态、并选择或更新视图来响应用户操作，一个核心原则是追求“瘦控制器”，即控制器本身不应包含复杂的业务逻辑，这些逻辑应归属于模型；

MVP 架构

基础概念

概念：MVP 基础概念

  • 概念：
    • MVP 相比于 MVC，其核心变化在于用 Presenter（呈现器）取代了 Controller（控制器），这种演变的目的是为了实现更彻底的关注点分离，尤其是让 View（视图）变得完全被动，从而彻底解耦 View 和 Model（模型），使得业务逻辑更易于测试和维护；
  • 概念：Model
    • Model 代表的是数据和业务逻辑核心，它定义了需要在用户界面中被显示或被操作的数据接口以及相关的业务规则；
    • 例如，它可能负责从数据库、网络等数据源获取数据，并进行计算、验证等处理。​Model 是完全独立于 View 和 Presenter 的，它不关心数据如何被显示，也不关心用户交互逻辑，只在数据变化时通知观察者（通常是 Presenter）；
  • 概念：View
    • View 是用户界面的呈现层，但它是一个非常被动的接口；
    • 它的职责有两个：
      • 第一，​显示数据，即根据 Presenter 提供的格式化后的数据来渲染 UI；
      • 第二，​路由用户命令，即捕获用户的输入操作（如点击按钮、触摸屏幕），但不进行任何逻辑处理，而是立即将这些操作事件“转发”给 Presenter 去决定该如何处理；
    • 在 MVP 中，View 不再直接与 Model 通信，这大大降低了它的复杂性；
  • 概念：Presenter
    • Presenter 是 MVP 架构的“大脑”和协调中心，作为 Model 和 View 之间的中介。它同时持有 View 和 Model 的引用；
    • 它的主要职责是：响应 View 传来的用户事件，调用 Model 的相应方法进行数据处理（例如，更新用户信息），然后从 Model 获取最新的数据，并进行格式化，最后驱动 View 更新显示；
    • Presenter 承担了绝大部分的展示逻辑，确保 View 只做最简单的显示工作；

MVC 和 MVP 的区别

分析：MVC 和 MVP 的区别

  • 分析：
    • 由于 View 是被动的且依赖于一个接口（通常在实践中，View 会实现一个 IView 接口），Presenter 只通过这个接口与 View 交互；
    • 因此，在单元测试时，我们可以轻松地创建一个模拟 View 来测试 Presenter 的逻辑，而无需启动任何 UI 组件，使得测试变得非常快速和可靠；
    • 并且 View 和 Model 被完全隔离开。修改 UI 界面（例如将文本框改为滑块）不会影响 Model 层的业务逻辑，反之亦然。代码结构清晰，易于维护和扩展；

MVP 在 Unity 开发中的使用

概念：MVP 在 Unity 开发中的使用

  • 概念：
    • Model​：通常是纯 C# 类（不继承自 MonoBehaviour），用于管理游戏状态，如玩家属性（生命值、分数）、库存数据、游戏规则等；
    • View ​：通常是挂载在 GameObject 上的 MonoBehaviour 脚本，负责控制 UI 元素或游戏实体的可视化表现（如动画、特效）的更新；
    • Presenter ​：同样是 MonoBehaviour 脚本，挂载在场景中的管理器对象上。它监听用户输入（如 Update 方法中的键盘事件）和 View 发出的事件，并驱动 Model 和 View 进行更新；
  • 示例：Model
using System;
using UnityEngine;

[System.Serializable]
public class CounterModel
{
    // 事件：当数值改变时通知所有订阅者（如View）
    public event Action<int> OnValueChanged;

    // 私有字段，通过属性进行访问
    [SerializeField] 
    private int _value = 0;

    // 公开属性，设置值时触发事件
    public int Value
    {
        get => _value;
        set
        {
            // 只有当值真正改变时才触发
            if (_value != value)
            {
                _value = value;
                // 发出通知，告知值已改变
                OnValueChanged?.Invoke(_value);
            }
        }
    }
}
  • 示例：View
using UnityEngine;
using UnityEngine.UI;

public class CounterView : MonoBehaviour
{
    [Header("UI元素")]
    [SerializeField] private Button _incrementButton;
    [SerializeField] private Button _decrementButton;
    [SerializeField] private Text _countText;

    // 事件：当UI上的按钮被点击时，通知Controller
    public event Action OnIncrementClicked;
    public event Action OnDecrementClicked;

    // 初始化UI，通常在Controller中调用
    public void Initialize()
    {
        // 为按钮添加监听器，当点击时触发对应事件
        _incrementButton.onClick.AddListener(() => OnIncrementClicked?.Invoke());
        _decrementButton.onClick.AddListener(() => OnDecrementClicked?.Invoke());
    }

    // 更新显示文本的方法，由Controller订阅Model事件后调用
    public void UpdateCountText(int newCount)
    {
        _countText.text = $"Count: {newCount}";
    }
}
  • 示例：Presenter
using UnityEngine;

public class CounterPresenter : MonoBehaviour
{
    [Header("MVC引用")]
    [SerializeField] 
    private CounterModel _model;
    [SerializeField] 
    private CounterView _view;

    private void Start()
    {
        // 初始化View
        _view.Initialize();

        // 订阅View的事件（用户输入）
        _view.OnIncrementClicked += IncrementCount;
        _view.OnDecrementClicked += DecrementCount;

        // 订阅Model的事件（数据变化）
        _model.OnValueChanged += _view.UpdateCountText;

        // 初始化View的显示，以反映Model的当前状态
        _view.UpdateCountText(_model.Value);
    }

    private void OnDestroy()
    {
        // 非常重要！在销毁时取消订阅，防止内存泄漏
        _view.OnIncrementClicked -= IncrementCount;
        _view.OnDecrementClicked -= DecrementCount;
        _model.OnValueChanged -= _view.UpdateCountText;
    }

    private void IncrementCount()
    {
        // 用户点击了增加按钮，直接更新Model
        _model.Value++;
        Debug.Log($"Count increased to: {_model.Value}");
    }

    private void DecrementCount()
    {
        // 用户点击了减少按钮，直接更新Model
        _model.Value--;
        Debug.Log($"Count decreased to: {_model.Value}");
    }
}
注意：取消订阅的注意事项
  • 由于在控制器中要负责添加订阅，因此在控制器销毁时，需要销毁时取消订阅，防止内存泄漏；

MVVM 架构

基础概念

概念：什么是 MVVM 架构

  • 概念：
    • MVVM 通过引入一个称为 ViewModel 的中间层，并利用“绑定”机制来实现用户界面（View）的开发与业务逻辑和数据（Model）分离；
    • “绑定”是 MVVM 架构的灵魂，它建立了 View 和 ViewModel 之间的自动连接通道。你可以把它想象成一根数据线：当 ViewModel 中的数据发生变化时，View 上的显示会自动更新；反之，当用户在 View 上执行操作（如在输入框中打字），对应的数据也会自动更新到 ViewModel 中。这就是所谓的双向数据绑定（DataBinding）；
  • 优点：
    • 开发者从此可以从繁琐、易错的 DOM 操作中解放出来，只需关心业务数据和逻辑（维护 ViewModel），UI 的更新工作由框架自动完成；

MVVM 和 MVP 的区别

分析：MVVM 和 MVP 的区别

  • 在 MVP 中的数据流是单向的，即：View → Presenter → Model → Presenter → View，其 View 和 Presenter ​强耦合，相互持有引用；
  • 而 MVVM 中的 ViewModel 可独立测试，View 和 ViewModel ​松耦合，通过绑定框架连接；
  • 具体而言，ViewModel 更像是 View 的“状态模型”，它只关心数据是什么，而不关心 View 具体如何显示。通过数据绑定，状态的改变会自动映射到 UI 上。这极大地解放了开发者，让开发者更专注于数据和业务逻辑。它非常适合数据驱动的现代应用，尤其是在使用支持强大数据绑定的框架​（如 Vue.js、Angular、WPF）时；
