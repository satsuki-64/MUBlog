转向系统

转向系统基础概念

概念：什么是转向

  • 概念：
    • 转向行为的本质是：通过计算角色当前状态与目标状态的差异，生成一个 “转向力”（Steering Force），并将其施加到角色的运动学模型上，从而驱动角色做出自然、智能的移动决策；
    • 它不是一次性计算出完整路径，而是在每一帧中，根据环境和目标，动态调整角色的加速度和方向，因此非常适合处理实时、动态的游戏场景；
  • 概念：位置与目标导向 - Seek
    • Seek（追逐）：
      • 计算角色位置到目标位置的向量，归一化后乘以最大速度，得到期望速度。转向力 = 期望速度 - 当前速度；
      • 效果：角色会直接、持续地向目标点移动；
    • Flee（逃离）
      • 原理：Seek 的反向操作。计算远离目标的向量，生成逃离的转向力；
      • 效果：角色会尽可能远离威胁或目标点；
    • Pursue（追击）
      • 原理：不仅看目标当前位置，还会预测目标未来的位置（基于其当前速度），然后向预测点 Seek；
      • 效果：让角色 “预判” 目标移动，实现更智能的追击，而不是像 Seek 那样总是跟在屁股后面；
    • Evade（规避）
      • 原理：Pursue 的反向操作，预测威胁的未来位置，然后向远离该点的方向 Flee；
      • 效果：用于躲避导弹、敌人等高速移动的威胁；
    • Wander（漫游）
      • 原理：在角色前方定义一个圆，在圆上随机选择一个点作为临时目标，然后 Seek 这个点；
      • 效果：产生自然、无规律的随机移动，常用于 NPC 巡逻或动物行为；
    • FlowFieldFollowing（流场跟随）
      • 原理：预计算一个网格流场，每个格子存储一个方向向量。角色根据自己所在格子的方向向量移动；
      • 效果：高效实现群体向特定区域（如出口）移动，是 RTS 游戏中群体寻路的常用技术；
    • PathFollowing（路径跟随）
      • 原理：将一条路径分解为一系列路点（Waypoints），角色依次 Seek 每个路点，到达后切换到下一个；
      • 效果：让角色沿着预设路线移动，常用于剧情动画或巡逻路线；
  • 概念：速度匹配与到达行为（VelocityMatch / Arrive）
    • VelocityMatch（速度匹配）
      • 原理：让角色的速度逐渐趋近于另一个目标角色的速度；
      • 效果：实现 “跟队” 效果，是群体行为（如羊群、鱼群）中 “Velocity Matching” 规则的基础；
    • Arrive（到达）
      • 原理：在 Seek 的基础上增加了减速区。当角色接近目标时，根据距离线性减小期望速度，直到速度为 0；
      • 效果：让角色平滑地减速并停在目标点，而不是 “撞” 上去。Arrive 行为内部会调用 VelocityMatch 来实现最终的速度归零；
  • 概念：朝向与视觉行为（Align）
    • Align（对齐）
      • 原理：计算角色当前朝向与目标朝向的角度差，生成一个扭矩（Torque）来平滑地转向目标方向；
      • 效果：让角色的身体朝向与目标方向一致，是群体行为中 “Alignment” 规则的基础；
    • Face（面向）
      • 原理：计算角色到目标的方向向量，将其作为目标朝向，然后调用 Align 行为；
      • 效果：让角色 “转头” 看向目标，常用于 NPC 与玩家对话或敌人锁定玩家；
    • LookWhereYoureGoing（看向行进方向）
      • 原理：将角色的速度向量作为目标朝向，然后调用 Align 行为；
      • 效果：让角色的身体始终朝向其移动的方向，使运动看起来更自然、更符合物理直觉；

概念：变量匹配

  • 引入：
    • 最简单的一类转向行为是通过变量匹配来运作的：它们会尝试让角色运动学状态中的一个或多个元素，去匹配单个目标的运动学状态。例如，我们可能会尝试匹配目标的位置，而不关心其他元素。这就需要朝着目标位置加速，并在接近目标时减速。或者，我们也可以尝试匹配目标的朝向，通过旋转使自己与目标对齐。我们甚至可以尝试匹配目标的速度，沿着平行路径跟随它，复制其运动，但保持固定的距离。变量匹配行为会将两个运动学状态作为输入：角色自身的运动学状态和目标的运动学状态。不同命名的转向行为会尝试匹配不同的元素组合，并添加额外的属性来控制匹配的执行方式。创建一个通用的变量匹配转向行为，并简单地告诉它要匹配哪些元素的组合，这是可行的，但并不是特别有用。我们已经在几种场合见过这种实现方式。当同时匹配运动学状态中的多个元素时，问题就出现了。它们很容易产生冲突。我们可以独立地匹配目标的位置和朝向。但位置和速度呢？如果我们在匹配它的速度，那么我们就无法再试图靠近它；
  • 概念：
    • 更好的方法是为每个元素设计独立的匹配算法，然后在之后以正确的组合将它们合并。这使我们能够使用本章中介绍的任何转向行为组合技术，而不是将其硬编码为单一行为。用于组合转向行为的算法旨在解决冲突，因此非常适合这项任务。对于每一种匹配转向行为，都存在一种相反的行为，它会尽可能地远离匹配状态。试图追上目标的行为，其相反行为就是试图躲避目标，依此类推；
    • 正如我们在运动学追逐（kinematic seek）行为中看到的，其相反形式通常只需对基础行为进行简单的调整即可。我们会将几种转向行为及其相反行为成对地进行介绍，而不是将它们分在不同的章节里；

Seek And Flee

概念：Seek And Flee

  • 概念：
    • 追逐行为会尝试让角色的位置与目标的位置相匹配，它会找到指向目标的方向，并以最快速度朝目标前进。因为现在转向行为的输出是加速度，所以角色会尽可能地加速；
    • 显然，如果一直加速，角色的速度会越来越大。大多数角色都有一个最大移动速度，它们不能无限制地加速。这个最大值可以是显式定义的，存储在变量或常量中；
    • 角色当前的速度（速度向量的长度）会被定期检查，如果超过了最大速度，就会被限制回去。这通常作为更新函数的后处理步骤来完成，不会在某个单独的转向行为内部执行；
  • 代码：
# 定义运动学结构体：存储角色/物体的核心运动状态信息
# （注：这是伪代码风格，常见于游戏AI/物理模拟的理论实现）
struct Kinematic:
    # 位置：2维或3维向量（如(x,y)或(x,y,z)），表示角色在世界坐标系中的坐标
    position # a 2 or 3D vector
    
    # 朝向：单个浮点数值，通常表示角色的旋转角度（如弧度/角度），
    # 2D场景中常用（3D场景会用四元数/欧拉角，此处为简化）
    orientation # a single floating point value
    
    # 速度：2维或3维向量，描述角色移动的方向和快慢（单位时间内的位置变化量）
    velocity # another 2 or 3D vector
    
    # 旋转角速度：单个浮点数值，描述角色朝向变化的快慢（单位时间内的角度变化量）
    rotation # a single floating point value
    
# 转向输出
struct SteeringOutput: 
	linear # a 2 or 3D vector 3 
	angular # a single floating point value
  • 代码：
// 运动学结构体
struct Kinematic:
    // … 之前定义过的成员数据 …

    // 更新函数：转向输出、最大速度、时间步
    def update(steering, maxSpeed, time):
        // 更新位置和朝向
        position += velocity * time
        orientation += rotation * time

        // 更新速度和旋转角速度
        velocity += steering.linear * time
        orientation += steering.angular * time

        // 检查是否超速，并进行截断
        if velocity.length() > maxSpeed:
            velocity.normalize()       // 归一化方向
            velocity *= maxSpeed       // 缩放到最大速度
  • 分析：
    • 另外，最大速度也可以通过阻力来实现：每一帧都给角色施加一点阻力使其减速。依赖物理引擎的游戏通常都会包含阻力。这类游戏不需要检查并截断当前速度；阻力（在更新函数中应用）会自动限制最高速度；
    • 阻力还能解决这个算法的另一个问题，因为加速度始终指向目标，如果目标在移动，追逐行为最终会让角色绕着目标转圈，而不是直接朝它移动。如果系统中有阻力，这个圆周运动就会变成向内螺旋。如果阻力足够大，玩家不会察觉到螺旋轨迹，只会看到角色直接移动到目标；
  • 代码：动态追逐的实现
class Seek:
    // 持有角色自身的运动学数据
    character
    // 持有目标的运动学数据
    target

    // 角色的最大加速度
    maxAcceleration

    // 计算并返回期望的转向输出
    def getSteering():
        // 创建结构体，用来存储输出结果
        steering = new SteeringOutput()

        // 计算指向目标的方向
        steering.linear = target.position - character.position
        // 逃离 Flee：steering.linear = character.position - target.position

        // 沿这个方向施加最大加速度
        steering.linear.normalize()
        steering.linear *= maxAcceleration

        // 该行为不使用角加速度
        steering.angular = 0
        return steering
  • 注意：
    • 去掉了运动学版本中对朝向的直接修改。可以像之前那样直接设置朝向，但更灵活的方式是使用变量匹配，让角色朝向正确的方向；

Arrive

概念：Arrive

  • 概念：
    • Seek（追逐）会始终以尽可能大的加速度朝目标移动。如果目标在不断移动，而角色需要全速追逐，这种行为是没问题的；
    • 但如果角色到达目标位置，它就会冲过头、反向移动、在目标点来回震荡，更常见的情况是绕着目标转圈，无法真正靠近；
    • 如果希望角色准确抵达目标，就需要让它减速，以便恰好停在正确位置，就像在运动学 Arrive 算法中看到的那样；
  • 图示：展示了针对固定目标时两种行为的表现。轨迹线分别是 Seek 和 Arrive 的路径
  • 分析：
    • Arrive 会笔直走到目标点，而 Seek 会绕一小段并最终震荡；
    • 动力学版本的 Seek 震荡没有运动学版本那么严重：因为角色不能立刻改变方向，所以看起来只是轻微晃动，而不是在目标点剧烈抖动；
    • 动力学 Arrive 行为比运动学版本稍微复杂一点。它使用两个半径：
      • 到达半径（target Radius）：和之前一样，让角色在足够接近目标时就视为已到达，避免因微小误差一直运动；
      • 减速半径（slow Radius）：比到达半径大得多，角色进入这个半径后就开始减速；
    • 算法会为角色计算一个理想速度：
      • 在减速半径上，理想速度等于最大速度；
      • 在目标点上，理想速度为 0（我们希望到达时速度为 0）；
      • 在两者之间，期望速度根据与目标的距离做插值；
    • 指向目标的方向和之前一样计算。然后将方向与期望速度结合，得到目标速度（target velocity）。算法查看角色当前速度，计算出将当前速度变为目标速度所需要的加速度；
    • 但我们不能瞬间改变速度，因此加速度是基于在固定时间内达到目标速度来计算的。这和运动学 Arrive 完全一样，之前我们尝试让角色在 1/4 秒内到达目标；
    • 动力学 Arrive 的固定时间可以更小一些，通常用 0.1 秒作为起点就很好。当角色速度太快无法准时到达时，它的目标速度会小于实际速度，因此加速度方向与运动方向相反，也就是在减速；
  • 代码：
class Arrive:
    # 角色自身的运动学数据（位置、速度、朝向等）
    character
    # 目标的运动学数据
    target

    # 角色的最大加速度
    maxAcceleration
    # 角色的最大速度
    maxSpeed

    # 到达半径：小于这个距离就认为已经到达目标
    targetRadius

    # 减速半径：进入这个范围就开始减速
    slowRadius

    # 达到目标速度所需的时间（默认0.1秒）
    timeToTarget = 0.1

    # 计算转向输出
    def getSteering(target):
        # 创建转向输出结构体
        steering = new SteeringOutput()

        # 1. 计算指向目标的方向与距离
        direction = target.position - character.position
        distance = direction.length()

        # 2. 如果已经在到达半径内，不需要任何动力
        if distance < targetRadius:
            return None

        # 3. 根据距离计算目标速度
        # 如果在减速半径外，直接用最大速度
        if distance > slowRadius:
            targetSpeed = maxSpeed
        # 否则按距离比例减速
        else:
            targetSpeed = maxSpeed * distance / slowRadius

        # 4. 组合方向与速度，得到目标速度
        targetVelocity = direction
        targetVelocity.normalize()       # 归一化方向
        targetVelocity *= targetSpeed     # 乘以目标速度

        # 5. 计算需要的加速度：(目标速度 - 当前速度) / 时间
        steering.linear = targetVelocity - character.velocity
        steering.linear /= timeToTarget

        # 6. 如果加速度超过最大值，进行限制
        if steering.linear.length() > maxAcceleration:
            steering.linear.normalize()
            steering.linear *= maxAcceleration

        # 7. 该行为不产生角加速度
        steering.angular = 0
        return steering

Align

概念：Align - 对齐

  • 概念：
    • Align 行为试图让角色的朝向与目标的朝向相匹配。它完全不关心角色或目标的位置、速度。在通用运动学模型中，朝向并不直接等同于移动方向，这个转向行为不会产生任何线性加速度，它只通过旋转来做出响应；
    • Align 的行为逻辑与 Arrive 非常相似：它试图到达目标朝向，并且在到达时让旋转速度为 0；
    • 因此可以复用大部分 Arrive 的代码，但朝向有一个额外的复杂问题需要处理。因为朝向是每 2π 弧度循环一圈的，我们不能简单地用 “目标朝向 − 角色朝向” 来直接决定需要旋转多少、；
  • 图示：
    • 图示展示了两种非常相似的对齐情况：角色离目标的角度差看上去一样，但直接相减的话，第一种会正确地顺时针小幅度旋转，第二种却会绕一大圈才能到达同一位置
  • 分析：
    • 为了找到真实的旋转方向，需要用目标朝向减去角色朝向，然后把结果映射到 范围内。我们通过加 / 减若干个 2π 来把结果放进这个区间，可以用取模函数配合一些简单处理来实现这个转换；
    • 之后我们就可以用转换后的值来控制旋转，整个算法就和 Arrive 非常像了。和 Arrive 一样，我们使用两个半径：一个用于开始减速，一个用于判定朝向已经足够接近目标；
    • 因为我们处理的是单个标量，而不是 2D/3D 向量，这里的 “半径” 相当于一个角度区间，在减去旋转值时不会有这个问题。与朝向不同，旋转（rotation）本身不循环。你可以有非常大的旋转值，远超出 范围，大数值只代表非常快的旋转速度；
  • 代码：
class Align:
    # 角色自身的运动学数据
    character
    # 目标的运动学数据
    target

    # 最大角加速度
    maxAngularAcceleration
    # 最大旋转速度（角速度）
    maxRotation

    # 目标半径：角度差小于此值，视为已对齐
    targetRadius

    # 减速半径：角度差进入此范围，开始减速
    slowRadius

    # 达到目标旋转速度的时间
    timeToTarget = 0.1

    def getSteering(target):
        # 创建转向输出结构体
        steering = new SteeringOutput()

        # 1. 先直接计算朝向差（未处理循环）
        rotation = target.orientation - character.orientation

        # 2. 将角度映射到 (-π, π) 区间（关键！解决角度环绕问题）
        rotation = mapToRange(rotation)
        # 取角度差的绝对值
        rotationSize = abs(rotation)

        # 3. 如果已经足够接近，不需要转向
        if rotationSize < targetRadius:
            return None

        # 4. 根据角度差计算目标旋转速度
        # 超出减速半径 → 最大转速
        if rotationSize > slowRadius:
            targetRotation = maxRotation
        # 否则按比例减速
        else:
            targetRotation = maxRotation * rotationSize / slowRadius

        # 5. 给目标旋转速度加上方向（正负）
        targetRotation *= rotation / rotationSize

        # 6. 计算角加速度：在 timeToTarget 时间内达到目标转速
        steering.angular = targetRotation - character.rotation
        steering.angular /= timeToTarget

        # 7. 限制角加速度不超过最大值
        angularAcceleration = abs(steering.angular)
        if angularAcceleration > maxAngularAcceleration:
			# 对标量（单个数值）做 “归一化” —— 保留旋转方向，把大小变成1.
			# 和向量归一化（`vector.normalize()`）逻辑一致，只是向量归一化是把长度变 1，这里是把标量大小变 1，保留正负方向
            steering.angular /= angularAcceleration
            
            # 把 “归一化后的方向” 乘以最大角加速度，得到 “方向正确、大小刚好等于上限” 的角加速度
            steering.angular *= maxAngularAcceleration

        # 8. 该行为不产生移动加速度
        steering.linear = 0
        return steering

Velocity Matching

概念：Velocity Matching

  • 概念：
    • 这个行为单独使用时几乎没什么实际价值，它虽然能让角色模仿目标的运动状态，但这一用途并不实用；其真正发挥关键作用的场景是与其他行为组合使用时；
    • 例如，它是群体聚群（Flocking）转向行为的核心组成部分之一。其实已经实现过一个 “匹配速度” 的核心逻辑：Arrive（到达）行为会先根据与目标的距离计算出目标速度，再试图让角色达到这个目标速度。而速度匹配（Velocity Matching）本质上就是把 Arrive 行为中 “根据距离算目标速度” 的步骤剥离，只保留 “让角色速度逼近目标速度” 的核心逻辑；
    • 即：速度匹配不是 “独立可用” 的行为，而是组合型转向行为的基础模块（比如 Flocking 聚群行为中的 “速度对齐（Velocity Alignment）” 规则，就是基于这个行为实现）
  • 代码：
class VelocityMatch:
    # 角色自身的运动学数据（包含velocity等核心属性）
    character
    # 目标的运动学数据（包含velocity等核心属性）
    target

    # 角色能承受的最大线性加速度（限制加速的上限）
    maxAcceleration

    # 达到目标速度的时间步（默认0.1秒，控制加速的“平滑度”）
    timeToTarget = 0.1

    # 计算转向输出（核心函数）
    def getSteering(target):
        # 1. 创建转向输出结构体（存储线性/角加速度）
        steering = new SteeringOutput()

        # 2. 计算需要的线性加速度：目标是在timeToTarget时间内追上目标速度
        # 公式逻辑：加速度 = (目标速度 - 当前速度) / 时间（匀加速运动公式变形）
        steering.linear = target.velocity - character.velocity
        steering.linear /= timeToTarget

        # 3. 限制加速度不超过最大值（避免加速过快、运动不自然）
        if steering.linear.length() > maxAcceleration:
            steering.linear.normalize()       # 归一化：保留方向，将长度变为1
            steering.linear *= maxAcceleration # 缩放：让加速度大小等于最大值

        # 4. 速度匹配仅影响移动，不影响旋转，因此角加速度为0
        steering.angular = 0
        # 5. 返回最终的转向输出
        return steering
  • 补充：
    • 核心公式 (目标速度 - 当前速度) / 时间 来自 “匀加速运动” 的物理逻辑；
    • 算法仅涉及向量减法、除法、长度计算、归一化等固定次数的操作，无论角色 / 目标的数量多少，计算量都不会增加，因此是常数级 复杂度；

Delegated Behavior And Pursue

概念：Delegated Behavior - 委托行为

  • 概念：
    • 追逐（Seek）、逃离（Flee）、到达（Arrive）和对齐（Align）为许多其他行为提供了转向计算的底层支持。接下来所有的行为都具有相同的基本结构：
    • 它们先计算出一个目标，可以是位置或朝向（虽然也可以使用速度，但我们接下来要讲的行为都不使用），然后委托给前面讲过的某个基础行为来计算最终的转向力；
    • 目标的计算可以基于很多种输入：
      • 追击（Pursue） 根据另一个目标的运动，为 Seek 计算一个预测目标；
      • 避障（Collision Avoidance） 根据障碍物的距离，为 Flee 创建一个目标；
      • 漫游（Wander） 会创建一个随时间不断漂移的自己的目标；
    • 事实上，结果表明：Seek、Align、Velocity Matching 是仅有的三个基础行为（按类比还存在一个旋转匹配行为，但我们从未见过它的实际应用）
    • 正如我们在上一个算法中看到的，Arrive 可以被拆分成两步：创建一个（速度）目标、应用速度匹配算法；
    • 这是很常见的结构，下面许多委托行为，反过来又可以作为另一个委托行为的基础。Arrive 可以作为 Pursue 的基础，Pursue 可以作为其他算法的基础，依此类推；
  • 概念：Pursue and Evade 追击与躲避
    • 到目前为止，我们的移动都只基于当前位置。如果我们追逐的是一个移动目标，那么不断朝它当前位置移动是不够的。等我们到达它现在的位置时，它早就已经走开了；
    • 当目标很近，并且我们每一帧都在重新计算位置时，这问题不算太大，我们最终还是能追上。但如果角色离目标很远，它就会朝着明显错误的方向冲出去，如图所示；

    • 我们不应该瞄准它当前的位置，而是需要预测它未来一段时间会在哪里，并朝那个点移动。我们可以使用各种复杂算法来做预测，但大多数都过于冗余；
    • 关于最优预测与被追角色的最优策略已经有很多研究（这是军事研究里的热门话题，比如躲避来袭导弹）。Craig Reynolds 的原始方法要简单得多：
    • 假设目标会以当前速度继续保持运动，这在短距离内是合理的假设，即使在较长距离下看起来也不会太蠢；
  • 实现：算法流程
      1. 计算角色与目标之间的距离；
      1. 计算以最大速度到达那里需要多长时间，用这个时间作为预测时长；
      1. 计算目标如果继续以当前速度移动，会到达什么位置，把这个新位置作为普通 Seek 行为的目标；
      1. 如果角色移动很慢，或者目标非常远，预测时间可能会非常大。目标不可能永远沿直线走，所以我们希望给预测的时长设置一个上限；
      • 因此算法里有一个 maxPrediction（最大预测时间）参数，如果预测时间超过这个值，就使用最大值；
  • 图示：
    • Seek 和 Pursue 追逐同一个移动目标，Pursue 明显更高效、更直接；

# 追击行为，继承自 Seek
class Pursue (Seek):

    # 最大预测时间（不会预测比这更久的未来）
    maxPrediction

    # 覆盖（重名）Seek 中的 target
    # 也就是说：这个类有两个叫 target 的变量
    # Seek.target 是父类的目标，会自动计算，不应该手动设置
    # Pursue.target 是我们真正要追击的目标
    target

    # ... 其他数据从父类继承 ...

    def getSteering():

        # ================ 1. 计算要委托给 Seek 的预测目标 ================

        # 计算到目标的方向
        direction = target.position - character.position
        # 计算距离
        distance = direction.length()

        # 当前速度
        speed = character.velocity.length()

        # 如果速度太小，无法给出合理的预测时间
        # 直接使用最大预测时间
        if speed <= distance / maxPrediction:
            prediction = maxPrediction
        # 否则计算正常的预测时间：时间 = 距离 / 速度
        else:
            prediction = distance / speed

        # 设置 Seek 要使用的预测目标
        Seek.target = explicitTarget
        # 预测目标未来的位置：当前位置 + 速度*预测时间
        Seek.target.position += target.velocity * prediction

        # ================ 2. 委托给 Seek 执行 ================
        return Seek.getSteering()

Face

概念：Face

  • 概念：
    • Face 行为让角色看向它的目标。它委托给 Align（对齐）行为来执行旋转，但会先计算出目标朝向；
    • 目标朝向是根据目标相对于角色的位置计算出来的，这个过程和我们在运动学移动的 getOrientation 函数中使用的方法完全相同；
  • 代码：
# Face 行为，继承自 Align
class Face (Align):

    # 覆盖 Align 中的 target 成员
    target

    # ... 其他数据从父类继承 ...

    # 与 Pursue 类似的实现方式
    def getSteering():

        # ================== 1. 计算要委托给 Align 的目标朝向 ==================

        # 计算指向目标的方向
        direction = target.position - character.position

        # 如果方向向量为零（目标就在身上），直接返回，不做任何改变
        if direction.length() == 0:
            return target

        # 设置 Align 要使用的目标
        Align.target = explicitTarget
        # 通过 atan2 计算目标朝向：从方向向量转成角度
        Align.target.orientation = atan2(-direction.x, direction.z)

        # ================== 2. 委托给 Align 执行旋转 ==================
        return Align.getSteering()

Wander

概念：Wander

  • 概念：
    • 漫游行为控制角色无目的地随意移动。在运动学漫游行为中，我们每一帧都会给移动方向叠加一个随机扰动，这会让角色平滑前进，但旋转会很不自然，移动时左右乱抖；
    • 最初的漫游思路（直接随机改方向）会造成线性抖动。运动学版本增加了一层间接性，但又产生了旋转抖动；
    • 因此可以通过再增加一层间接性来平滑这种抖动，让角色的朝向不直接依赖随机数，而是间接依赖：你可以把运动学漫游理解成一种委托式 Seek 行为，在角色周围有一个圆，目标点被限制在圆上。每次执行行为时，我们把目标点在圆上随机移动一小段，然后让角色去 Seek 这个目标。每一帧，角色都会尝试用 Face 行为朝向目标，然后增加一步：沿着当前朝向全速前进；
    • 也可以这样实现：让角色 Seek 目标，同时用 Look Where You’re Going 来修正朝向。无论哪种方式，角色的朝向会在帧之间被保留，因此旋转变化会被平滑；
  • 代码：
# 漫游行为，继承自 Face 行为
class Wander (Face):

    # 存储漫游圆的半径（wanderRadius）和向前偏移量（wanderOffset）
    # （向前偏移：圆的中心在角色前方的距离）
    wanderOffset
    wanderRadius

    # 存储漫游目标朝向的最大变化率（控制转向的快慢）
    wanderRate

    # 存储漫游目标当前的朝向（相对角色自身朝向的偏移角度）
    wanderOrientation

    # 角色的最大线性加速度
    maxAcceleration

    # 同样不需要新增 target 成员变量
    # ... 其他数据从父类继承 ...

    def getSteering():

        # ================== 1. 计算要委托给 Face 的目标点 ==================

        # 更新漫游目标的朝向：叠加随机偏移
        # randomBinomial() 返回 [-1,1] 之间的随机数，保证转向方向随机且均值为0
        wanderOrientation += randomBinomial() * wanderRate

        # 计算综合目标朝向：漫游偏移朝向 + 角色当前朝向
        # （把相对偏移转成世界坐标系的绝对朝向）
        targetOrientation = wanderOrientation + character.orientation

        # 计算漫游圆的圆心位置：
        # 角色当前位置 + 向前偏移量 * 角色朝向的向量形式
        target = character.position + wanderOffset * character.orientation.asVector()

        # 计算最终的漫游目标点：
        # 圆心位置 + 漫游圆半径 * 综合目标朝向的向量形式
        target += wanderRadius * targetOrientation.asVector()

        # ================== 2. 委托给 Face 行为，让角色朝向目标点 ==================
        steering = Face.getSteering()

        # ================== 3. 设置线性加速度：沿当前朝向全速前进 ==================
        # 让角色以最大加速度，朝着自身当前朝向移动
        steering.linear = maxAcceleration * character.orientation.asVector()

        # 返回最终的转向输出（包含角加速度+线性加速度）
        return steering

Path Following

概念：Path Following

  • 概念：
    • 到目前为止，我们见过的行为要么使用单个目标，要么完全没有目标。路径跟随（Path Following）是一种以整条路径作为目标的转向行为，拥有路径跟随行为的角色应当沿着路径朝一个方向持续移动；
    • 路径跟随是一种委托行为，它根据角色当前位置和路径形状，计算出一个目标点，然后把这个目标交给 Seek 行为去处理。我们不需要使用 Arrive，因为目标点会一直沿着路径向前移动，不用担心角色会追上并停在原地；
    • 目标位置的计算分为两步：
      1. 首先，将角色当前位置映射到路径上最近的点。这个过程可能比较复杂，尤其是当路径是曲线或由多条线段组成时；
      1. 其次，在映射点沿路径向前一段固定距离处选择一个新目标点。如果要改变沿路径运动的方向，只需将这段距离取反即可；
  • 图示：
    • 图中显示了角色在路径上的当前投影点，以及前方一小段距离处的目标点
  • 分析：
    • 有些实现生成目标的方式略有不同：它们先预测角色一小段时间后的位置，再将这个预测位置映射到路径上的最近点，作为候选目标。如果新候选目标没有比上一帧更靠前，就强制让它更靠前。我们把这种方式称为预测式路径跟随，如图所示；
    • 后一种实现方式在方向突变的复杂路径上会显得更平滑，但缺点是：当两条路径靠得很近时，角色会抄近路（切角）；

    • 路径参数（Param）：可以理解为 “从路径起点走到该点的总长度”（单调递增），比如路径是 S→A→B，S 到 A 长 4 米，A 到 B 长 6 米，那么 S 的参数 = 0，A=4，B=10；
    • 这部分代码就是先把角色的世界坐标转成路径参数，再算目标参数，最后把目标参数转回世界坐标；
    • 而 pathOffset 可以理解为代表的就是每次走的路径长度，但是 pathOffset 是「目标点与角色映射点之间的沿路径长度」，而非「角色每帧实际移动的路径长度」；
    • pathOffset 可以理解为：你希望角色的追逐目标，始终在自己当前路径映射点的 “前方 / 后方” 多少米（沿路径），这个 “多少米” 就是 pathOffset，它本质是「沿路径的长度值」，但不是角色 “每次走的实际步长”；

Separation

概念：Separation

  • 概念：
    • 分离行为在人群模拟中非常常见，即大量角色大致朝同一方向移动的场景。它的作用是让角色之间不要靠得太近、避免拥挤；
    • 当角色互相横穿路径时，分离行为的效果并不好。这种情况下应该使用下面介绍的避障行为（Collision Avoidance），在大多数情况下，分离行为的输出为零，即不会产生任何移动建议。
    • 只有当该行为检测到其他角色小于某个距离阈值时，才会像 逃离行为（Evade） 那样，让角色远离对方。但与基础逃离行为不同的是：移动的强度与离目标的距离相关；
    • 分离强度可以按任意公式衰减，最常见的是线性衰减或平方反比衰减；

class Separation:

    # 角色的运动数据（位置、速度等）
    character

    # 可能需要避开的目标列表
    targets

    # 分离行为的作用阈值：小于这个距离才会产生排斥
    threshold

    # 平方反比力的衰减系数
    decayCoefficient

    # 最大加速度
    maxAcceleration

    def getSteering():

        # 存储最终输出的转向力
        steering = new Steering

        # 遍历每一个需要检测的目标
        for target in targets:

            # 计算指向目标的方向与距离
            direction = target.position - character.position
            distance = direction.length()

            # 只有距离小于阈值才进行排斥
            if distance < threshold:

                # 计算排斥强度：平方反比，并限制不超过最大加速度
                strength = min(decayCoefficient / (distance * distance), maxAcceleration)

                # 方向归一化
                direction.normalize()

                # 累加到线性加速度上（排斥 = 朝反方向推）
                steering.linear += strength * direction

        # 遍历完所有目标，返回最终转向
        return steering

概念：吸引力与独立性

  • 概念：吸引力
    • 使用平方反比公式时，我们可以把衰减系数设为负值，从而得到吸引力。角色会被半径内的其他角色吸引，但这种用法很少有用；
    • 有些开发者尝试在场景中使用大量吸引器和排斥器，让角色运动主要由这些力控制。比如让角色被目标吸引、被障碍物排斥。虽然表面上简单，但这种方式暗藏很多坑，不注意就会出问题；
    • 下一节 “组合转向行为” 会说明：为什么单纯用大量吸引 / 排斥器会让角色经常卡住，为什么一开始用更复杂的算法，长期来看反而更省事；
  • 概念：独立性
    • 分离行为单独使用没什么用。角色只会挤开一点，然后就不再移动了。分离行为与本章剩余行为一样，设计目的是与其他转向行为组合使用。我们将在下一节讨论这种组合如何工作；

Collision Avoidance

概念：Collision Avoidance

  • 概念：
    • 在城区等场景中，经常会有大量角色在同一空间内移动。这些角色的移动轨迹会相互交叉，它们需要避免与其他移动中的角色不断发生碰撞。一种简单的思路是使用逃离或分离行为的变种：只在目标出现在角色前方的锥形区域内时才生效；

    • 对于躲避群体角色，对位置和速度取平均的方法在这套算法中效果不好。正确做法是：找到最先发生最近逼近的那个角色，只对它做出反应。一旦避开这次即将发生的碰撞，再去处理更远的角色；

class CollisionAvoidance:

    # 角色的运动数据（位置、速度、朝向等）
    character

    # 最大加速度
    maxAcceleration

    # 可能发生碰撞的目标列表
    targets

    # 角色的碰撞半径（这里假设所有角色半径相同）
    radius

    def getSteering():

        # 1. 找出【最先会发生碰撞】的目标

        # 记录最早的碰撞时间，初始化为无穷大
        shortestTime = infinity

        # 存储最先碰撞的目标 以及相关数据，避免重复计算
        firstTarget = None
        firstMinSeparation
        firstDistance
        firstRelativePos
        firstRelativeVel

        # 遍历每一个目标
        for target in targets:

            # 计算相对位置：目标 - 自己
            relativePos = target.position - character.position

            # 计算相对速度：目标速度 - 自己速度
            relativeVel = target.velocity - character.velocity

            # 相对速率（速度的模）
            relativeSpeed = relativeVel.length()

            # 计算到达【最近逼近时刻】的时间
            timeToCollision = (relativePos . relativeVel) / (relativeSpeed * relativeSpeed)

            # 当前距离
            distance = relativePos.length()

            # 最近时刻的最小间距
            # 当前距离 - 相对速度*最小逼近时间 = 不发生碰撞的最小间距
            minSeparation = distance - relativeSpeed * timeToCollision

            # 如果最小间距大于两倍半径，说明不会碰撞，直接跳过
            if minSeparation > 2 * radius:
                continue

            # 只保留【时间为正、且最早】的那次碰撞
            if timeToCollision > 0 and timeToCollision < shortestTime:

                shortestTime = timeToCollision
                firstTarget = target
                firstMinSeparation = minSeparation
                firstDistance = distance
                firstRelativePos = relativePos
                firstRelativeVel = relativeVel

        # 2. 计算转向力

        # 如果没有需要躲避的目标，直接返回空
        if not firstTarget:
            return None

        # 如果已经相撞 或 马上正中相撞，直接用当前位置躲避
        if firstMinSeparation <= 0 or firstDistance < 2 * radius:
            relativePos = firstTarget.position - character.position

        # 否则用【未来最近时刻】的相对位置躲避
        else:
            relativePos = firstRelativePos + firstRelativeVel * shortestTime

        # 归一化方向
        relativePos.normalize()

        # 沿这个方向施加最大加速度（远离目标）
        steering.linear = relativePos * maxAcceleration

        return steering
  • 分析：算法性能
    • 内存复杂度：O (1)
    • 时间复杂度：O (n)，n 为需要检测的目标数量；
    • 和上一个行为一样，如果能在进入算法前用空间划分等方法快速过滤掉远处目标，整体性能会大幅提升。但算法本身对它检查的目标数量仍然是线性复杂度；

Obstacle and Wall Avoidance

概念：Obstacle and Wall Avoidance

  • 概念：
    • 碰撞避免行为假设目标是球形的，它关心的是避免靠近目标中心点。这种方式也可以用于游戏中任何能轻松用包围球表示的障碍物，比如箱子、桶等小物体。但更复杂的障碍物就不适合用这种方式表示了，比如楼梯这类大型物体的包围球可能会占满整个房间，我们肯定不希望角色仅仅为了避开角落的楼梯，就一直贴着房间外围走；
    • 而游戏中最常见的障碍物：墙壁，更是根本无法简单用包围球表示。障碍物与墙壁躲避行为使用了另一种完全不同的避障思路：移动的角色会沿运动方向发射一条或多条射线，如果这些射线碰到障碍物，就创建一个用于躲避的目标点，然后让角色对这个目标点执行基础的 Seek（追逐）。通常射线不是无限长的，只会向前延伸一小段距离（一般对应角色几秒的移动距离）。
  • 图示：展示了角色发射一条射线碰到墙壁，利用碰撞点和墙壁的法线，在距离墙面固定距离的位置创建一个目标点
  • 代码：

    collisionDetector

    # 与墙壁保持的最小距离（即避障距离）
    # 应大于角色自身半径
    avoidDistance

    # 向前检测碰撞的距离（即射线长度）
    lookahead

    # ... 其他数据继承自父类 ...

    def getSteering():

        # 1. 计算要委托给 Seek 的目标点

        # 计算碰撞射线方向
        rayVector = character.velocity
        rayVector.normalize()       # 归一化方向
        rayVector *= lookahead      # 拉长到检测距离

            character.position, rayVector)

        if not collision:
            return None

        # 否则根据碰撞点和法线生成目标点
        target = collision.position + collision.normal * avoidDistance

        # 2. 委托给 Seek 执行
        return Seek.getSteering()

组合转向行为

概念：组合转向行为基础概念

  • 概念：
    • 通过将多个转向行为组合在一起，可以实现更复杂的移动效果；
    • 组合转向行为有两种核心方法：混合（blending）和仲裁（arbitration）。两种方法都会接收一组各自带有输出的转向行为，并生成单一的最终转向输出。混合法会执行所有转向行为，然后通过一组权重或优先级组合它们的结果；
    • 这种方式足以实现一些非常复杂的行为，但当角色的移动受到大量约束时，就会出现问题。仲裁法则会选择一个或多个转向行为，让其完全控制角色的移动；
    • 有各种各样的仲裁策略可以决定哪种行为能够生效。不过，混合和仲裁并非互斥的方案，混合法的权重或优先级可能会随时间变化，而调整这些权重的过程可能是响应游戏场景或角色内部状态的结果，进而导致某些转向行为的权重可能被设为零，相当于被实际关闭；
    • 同时，仲裁架构也并非必须返回单个待执行的转向行为，它也可以返回一组混合权重，用于组合多个不同的行为；
  • 总结：
    • 一个通用的转向系统需要结合混合和仲裁的元素，理想的实现方式是将两者的元素结合起来，共同完成组合转向行为；

加权混合

概念：加权混合

  • 概念：
    • 组合转向行为最简单的方式是通过权重混合它们的结果，假设游戏中有一群移动的角色，这些角色需要作为一个整体移动，同时确保不会持续相互碰撞。每个角色需要与其他角色保持靠近，又要维持安全距离；
    • 因此，它们的整体行为是两种行为的混合：
        1. 向群体质心移动的抵达（arrive）行为；
        1. 与附近角色分离（separation）的行为；
    • 角色在任何时候都不会只执行其中一种行为，而是始终同时考虑这两种需求；
  • 算法：
    • 一组转向行为可以被混合成一个单一的行为。组合中的每个转向行为都会被请求输出其加速度（假设它是唯一运行的行为），这些加速度会通过加权线性和的方式组合，每个行为对应特定的系数；
    • 混合权重没有强制约束。例如，它们不需要求和为 1，而且通常也不会（也就是说，这并非加权平均值），求和得到的最终加速度可能超出角色的能力上限，因此需要根据角色的最大可行加速度进行截断（也可以使用更复杂的执行步骤）；
    • 在上述群体移动示例中，我们可以为分离和聚合（cohesion）行为都设置权重 1，此时请求的加速度会被求和并截断到最大可行加速度，这就是该算法的输出；
  • 图示：与所有参数化系统一样，权重的选择往往需要凭经验猜测或反复试验。有研究尝试通过遗传算法或神经网络演化转向权重，但结果并不理想，手动调试仍然是最合理的方式
  • 代码：
// 加权混合转向行为类
public class BlendedSteering
{
    // 存储转向行为与对应权重的结构体
    public struct BehaviorAndWeight
    {
        public ISteeringBehavior Behavior; // 转向行为接口实例
        public float Weight;               // 该行为的混合权重
    }

    // 转向行为-权重列表
    public List<BehaviorAndWeight> Behaviors;

    // 角色的最大加速度和最大旋转量
    public float MaxAcceleration;
    public float MaxRotation;

    // 计算并返回最终的转向输出（加速度）
    public SteeringOutput GetSteering()
    {
        // 创建用于累加的转向输出结构体
        SteeringOutput steering = new SteeringOutput();

        // 累加所有转向行为的加权加速度
        foreach (var behaviorItem in Behaviors)
        {
            // 获取单个行为的转向输出
            SteeringOutput behaviorSteering = behaviorItem.Behavior.GetSteering();
            // 加权后累加到最终结果
            steering.Linear += behaviorItem.Weight * behaviorSteering.Linear;
            steering.Angular += behaviorItem.Weight * behaviorSteering.Angular;
        }

        // 截断线性加速度到最大值（防止超出角色能力）
        if (steering.Linear.magnitude > MaxAcceleration)
        {
            steering.Linear = steering.Linear.normalized * MaxAcceleration;
        }
        // 截断角加速度（旋转量）到最大值
        steering.Angular = Mathf.Clamp(steering.Angular, -MaxRotation, MaxRotation);

        // 返回最终转向输出
        return steering;
    }
}

    public Vector3 Linear;  // 线性加速度
    public float Angular;   // 角加速度（旋转）

概念：Flocking and Swarming

  • 概念：
    • 群集 Flocking 行为是最常见的转向行为组合，它依赖于三种简单行为的加权混合：
        1. 远离过近的个体（分离）
        1. 与群体保持相同的移动方向和速度（对齐与速度匹配）
        1. 向群体质心移动（聚合）
    • 聚合转向行为会先计算群体的质心作为目标，然后将该目标交给常规的抵达行为处理。对于简单的群集行为，使用相等的权重可能就足够了。但通常来说，分离的优先级高于聚合，聚合又高于对齐（后两者的优先级有时也会颠倒）；
    • 在大多数实现中，群集行为会被修改为忽略远处的个体：每种行为都有一个 “邻域”，只有邻域内的其他个体才会被考虑。分离行为仅避开附近的个体，聚合和对齐行为也只基于邻域内的个体计算目标位置、朝向和速度；
    • 邻域最常用的实现方式是简单的半径阈值，不过 Reynolds 建议使用角度阈值；

概念：加权混合行为分析

  • 在实际游戏中，加权混合转向行为存在几个重要问题；
    1. 混合转向行为的演示通常使用非常稀疏的户外环境，而非室内或城市场景；
    • 因为在更真实的场景中，角色常常会以难以调试的方式被困在环境中。与所有 AI 技术一样，关键是在需要时能够获取有效的调试信息，至少要能可视化混合中每个转向行为的输入和输出。这些问题中的一部分（但绝非全部）可以通过在转向系统中引入仲裁来解决；
    1. 稳定平衡：当两个转向行为想要执行相互冲突的操作时，混合转向行为就会出现问题，这可能导致角色完全不动，陷入平衡状态；
    • 在图中，角色试图抵达目标同时避开敌人，追逐（seek）和逃离（evade）行为的输出恰好相互抵消；
    • 这种平衡状态很快会自行解除：只要敌人静止，数值不稳定性会给角色一个微小的横向速度，使其逐渐绕开敌人，最终冲向目标，这就是不稳定平衡；
    • 但下张图展示了更严重的情况：即使角色因数值误差稍微脱离平衡状态，也会立即回到平衡状态，完全无法脱身。角色会停在原地，显得笨拙且犹豫不决，这就是稳定平衡。稳定平衡存在 “吸引域”：场景中角色会落入该平衡状态的区域；
    • 如果吸引域很大，角色被困的概率就会极高。图中的吸引域在一条走廊中无限延伸，而不稳定平衡的吸引域大小实际上为零；
    • 吸引域不仅由位置决定，还可能只吸引沿特定方向移动或具有特定朝向的角色，因此它们往往难以可视化和调试；
    1. 受限环境：无论是单个还是混合的转向行为，在约束较少的环境中都能良好工作
    • 开放的 3D 空间中的移动约束最少，但大多数游戏都发生在受限的 2D 世界中，室内环境、赛道和编队移动都会大幅增加角色移动的约束条件；
    • 下图展示了追逐转向行为给出的反常移动建议：仅追逐（pursue）行为会导致角色撞墙，而添加墙壁躲避行为后，移动方向反而离捕获敌人的正确路线更远；
    • 这种问题常见于角色试图以锐角穿过狭窄门口的场景：障碍物躲避行为触发后，可能会让角色错过门口，偏离原本的路线。由于穿过狭窄通道的导航问题一直存在，许多开发者会特意要求关卡设计师在 AI 角色需要导航的区域设计较宽的通道；
    1. 短视性：
    • 转向行为是局部性的，仅基于即时的周围环境做决策。人类会预判行为的结果并评估是否值得执行，但基础转向行为无法做到这一点，因此它们常常为了达成目标而选择错误的行动路线；
    • 图展示了角色使用标准墙壁躲避技术避开墙壁，但移动路线恰好卡在拐角的错误一侧，因此角色永远无法追上敌人，且在一段时间内都不会意识到这一点；
    • 无法通过增强转向行为来解决这个问题：任何不具备前瞻能力的行为，都会被超出其感知范围的问题所阻碍。解决该问题的唯一方法是将寻路（pathfinding）整合到转向系统中，这种整合将在下文讨论，而寻路算法本身会在接下来的章节中介绍；

优先级

概念：优先级

  • 概念：
    • 我们接触过许多仅在特定条件下才会请求加速度的转向行为。与始终会产生加速度的追逐（seek）或逃离（evade）不同，碰撞避免（collision avoidance）、分离（separation）和抵达（arrive）在游戏的大多数时候不会产生任何加速度。但当这些行为确实提出加速度请求时，忽视它们是不明智的；
    • 例如，碰撞避免行为的请求应立即被执行，以防止角色与其他对象相撞。当行为被混合在一起时，它们的加速度请求会被其他行为的请求稀释；
    • 比如，追逐行为总会在某个方向上返回最大加速度，如果将其与碰撞避免行为等权混合，那么碰撞避免行为对角色移动的影响永远不会超过 50%。不过这可能不足以让角色摆脱危险；
  • 实现：
    • 行为混合的一种变体是用优先级替代权重。在基于优先级的系统中，行为会被分成多个组，每组内的行为使用常规的混合权重；
    • 这些组再按优先级排序，转向系统会依次处理每个组：
      • 先像之前一样混合组内的所有转向行为，如果混合后的总结果非常小（小于某个可调整的小参数），则忽略该结果并处理下一个组；
      • 最好不要直接与零值比较，因为计算中的数值不稳定性可能导致某些转向行为永远无法返回零值。使用一个小的常量（通常称为 epsilon 参数）可以避免这个问题；
    • 当找到一个结果不太小的组时，该组的结果就会被用于控制角色的转向。例如，一个团队中执行追逐任务的角色可能有三个组：碰撞避免组、分离组和追逐组；
      • 碰撞避免组包含障碍物躲避、墙壁躲避和避开其他角色的行为；
      • 分离组仅包含分离行为，用于避免与追逐队伍中的其他成员靠得太近；
      • 追逐组包含用于逼近目标的追逐转向行为；
    • 如果角色远离任何干扰物，碰撞避免组会返回无期望加速度的结果，接着系统会处理分离组（同样无动作），最后处理追逐组，并使用继续追逐所需的加速度；
    • 如果角色当前的移动状态完全符合追逐需求，该组也可能返回无加速度的结果，此时没有更多组需要处理，角色将不会获得任何加速度，就像完全由追逐行为控制一样；
    • 在另一种场景下，如果角色即将撞墙，第一个组（碰撞避免组）会返回用于避撞的加速度，角色会立即执行该加速度，而其他组的转向行为将不会被考虑；
  • 代码：
// 基于优先级的转向行为类
public class PrioritySteering
{
    // 存储混合转向行为组的列表（每组内是带权重的行为混合）
    public List<BlendedSteering> Groups;

    // 极小值参数（用于判断加速度是否可忽略），通常设为很小的正数（如0.001）
    public float Epsilon;

    // 计算并返回最终的转向输出
    public SteeringOutput GetSteering()
    {
        // 依次遍历每个优先级组
        foreach (var group in Groups)
        {
            // 计算当前组内混合后的转向输出
            SteeringOutput steering = group.GetSteering();

            // 检查线性加速度模长或角加速度绝对值是否超过epsilon
            // 超过则说明该组需要生效，直接返回该结果
            if (steering.Linear.magnitude > Epsilon || Mathf.Abs(steering.Angular) > Epsilon)
            {
                return steering;
            }
        }

        // 如果所有组的加速度都可忽略，返回最后一个组的结果（通常接近零）
        // 注：原文逻辑为遍历完所有组后返回最后一次计算的steering
        // 补充：实际工程中可优化为返回零加速度，避免空引用
        return Groups.LastOrDefault()?.GetSteering() ?? new SteeringOutput();
    }
}

基于流水线架构的转向系统

概念：基于流水线架构的转向系统

  • 概念：
    • 转向流水线方案由马尔钦・查迪（Marcin Chady）首创，是介于简单融合或按优先级排序转向行为，与实现一套完整运动规划方案（寻路算法）之间的中间方案；
    • 它属于协同仲裁方案，能够让不同转向行为之间产生建设性交互，在诸多常规方案容易出问题的场景中都能实现出色表现，包括狭窄通道通行、以及将转向行为与寻路无缝结合等。目前仅有少数开发者使用该方案；
    • 整个流水线分为四个阶段：
      • 目标生成器（Targeter）：确定角色的移动目标；
      • 分解器（Decomposer）：生成通往主目标的子目标；
      • 约束器（Constraint）：限制角色达成目标的可行方式；
      • 执行器（Actuator）：约束角色的物理运动能力；
    • 除最后一个阶段外，其余每个阶段都可以包含一个或多个组件；
    • 流水线中的每个组件都有不同的职责，它们本质上都是转向行为，但协作方式由所处阶段决定；
  • 概念：目标生成器 - Targeters
    • 目标生成器为角色生成顶层目标。目标可以分为多种：位置目标、朝向目标、速度目标、旋转目标。这些要素称为目标的通道（如位置通道、速度通道）；
    • 算法中的所有目标都可以指定任意一个或全部通道，未指定的通道则视为 “无关项”。不同通道可以由不同行为提供（例如追击敌人的目标生成器负责生成位置目标，而朝向目标生成器负责生成朝向目标），也可以由单个目标生成器请求多个通道；
    • 当使用多个目标生成器时，每个通道最多只能由一个目标生成器赋值。本节实现的算法默认目标生成器会按此规则协作，不会额外处理目标生成器覆盖已设置通道的情况。转向系统会尽最大可能满足所有通道的要求，尽管某些目标组合可能无法同时达成。关于这一点，我们会在执行阶段再做说明。
    • 乍看之下，为转向指定单一目标似乎有些奇怪。逃离、避障这类行为的逻辑是远离某个对象，而非趋近。但转向流水线要求你从角色目标的角度思考：如果目标是逃离，目标生成器就需要选定一个逃往的位置。即便追击的敌人不断迂回追逐，这个目标会逐帧变化，但每一时刻仍只有一个目标；
    • 其他 “远离类” 行为（如障碍物规避），在转向流水线中不视为目标，而是作为角色移动的限制条件，放在约束器阶段处理。

碰撞避免 - Force-Based 方法

概念：碰撞避免 - Force-Based 方法

  • 概念：
    • 基于力的模型将每个行人视为一个受力体，其运动由多种 “力” 共同驱动，这些力既包括物理层面的碰撞排斥，也包含社会心理层面的行为倾向；
    • 模型将影响人群行为的因素抽象为两类力：
      • 社会心理力（Socio-psychological forces）：如个人希望向目标移动的驱动力、与他人保持社交距离的排斥力等；
      • 物理力（Physical forces）：如与墙壁、障碍物或其他行人发生实际接触时产生的碰撞力、摩擦力等；
    • 每个个体的实际运动是两种因素共同作用的结果：
      • 期望速度（Desired velocity）：个体想要前往的方向和速度，通常由目标（如出口）决定；
      • 环境交互（Interaction with the environment）：个体与其他行人、墙壁等障碍物的避让和碰撞反应；
      • 这两种因素通过力的合成，最终决定了个体的实际运动轨迹；
