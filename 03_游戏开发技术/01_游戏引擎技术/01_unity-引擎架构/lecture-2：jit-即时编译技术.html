Lecture 2：JIT 即时编译技术

JIT 的作用

概念：JIT 的作用

  • 概念：
    • 编写的 C# 代码在经过编译后得出的 exe 或 dll 里面包含的并不是机器代码，而是一种中间代码, 也称为 MSIL (简称 IL)；
    • MSIL 可以在不同的系统或者平台上执行，CLR 中执行它们的模块就是这篇要讲的 JIT；
    • CoreCLR 中的 JIT 代号是 RyuJIT，RyuJIT 可以把 MSIL 翻译为 X86、X64 或者 ARM 的机器代码；
  • 特点：
    • 优点：
      • 同一个程序集可以在不同平台上运行；
      • 减少编译时间 (编译到 MSIL 的时间比编译到机器代码的时间要短很多)；
      • 可以根据目标平台选择最优的代码 (例如只在支持 AVX 指令的 CPU 使用 AVX 指令)
    • 缺点：
      • 部分平台上无法使用 (例如 iOS)
      • 不能执行过多的优化 (否则将会增加更多的运行负担)；
    • 为了解决这些坏处而出现的技术有 NGEN, AOT, CoreRT 等, 但是使用它们以后同时也就失去了使用 JIT 的好处；

注意：IOS 与 JIT
  • Apple 出于安全考虑禁止应用动态生成可执行代码，JIT 需要内存写入+执行权限，违反 iOS 沙盒规则；
  • 并且，App Store 及 iOS 使用 No eXecute bit 技术，审核要求所有代码必须静态打包，禁止运行时修改代码，所以在 Unity 当中，使用 AOT 编译来规避此问题，提前将 IL 代码编译为 Native 代码，iOS 上默认使用这种方式；
  • 那么，为什么 IOS 可以支持 Lua 热更新呢？因为 Lua 代码作为文本资源由 Lua 虚拟机解释执行，无需编译为本地代码，因此不被视为"可执行代码更新"。Lua 脚本可作为 AssetBundle 资源打包，通过资源热更新流程动态加载，与纹理、音效等资源更新方式一致，因此规避了代码编译的问题；
  • 具体而言，通过预生成所有可能的桥接代码，避免在 iOS 的 AOT 环境下动态生成代码；

JIT 流程

JIT 流程概述

概念：JIT 流程概述

  • 概念：
    • 总体上来说，RyuJIT 可以分为两个部分：
    • 前端: 也就是图上的第一行，负责把 MSIL 转换为 JIT 中的内部表现 (IR)并且执行优化；
    • 后端: 也就是图上的第二行，负责准备生成机器代码，分配寄存器等与平台相关的处理；
  • 概念：导入器
    • 将线性的、基于堆栈的 MSIL 指令转换为 RyuJIT 内部使用的、更复杂的中间表示（IR），这是一种树形结构（或基于图的）的 IR，更精确地表达了操作之间的依赖关系和操作数的来源（而不仅仅是堆栈操作），包含了丰富的类型信息和控制流信息；
  • 概念：RyuJIT 阶段
    • 这是 JIT 编译的核心和主体（图中用浅蓝色高亮区域表示）；
    • 它包含一系列连续的、相互依赖的优化和分析阶段（Passes），对 GenTrees 进行反复的转换和精炼。这些阶段的目标是生成尽可能高效的 GenTrees IR；
    • 常见的优化包括函数内联、常量传播、循环优化、移除非执行代码、控制流优化；
  • 概念：指令描述符 - InstrDescs
    • 将抽象的、平台无关的 GenTrees IR 转换为特定于目标 CPU 架构的中间表示，称为 InstrDescs (Instruction Descriptors) ，它代表了目标 CPU 的机器指令或指令模板，它包含了操作码、操作数类型、寻址模式、指令长度等具体信息。但是它仍然是一个描述性的结构，还不是最终的二进制字节码，充当了抽象 IR 与具体硬件指令之间的桥梁；
  • 概念：发射器
    • 将 InstrDescs 转换（“发射”）为真正的、可执行的本地机器码 (Native Code)​​ 二进制指令流。这涉及到解析 InstrDescs 中的信息，生成对应的操作码字节、操作数字节，计算相对跳转偏移量等；
    • 同时生成方法运行所需的额外信息，这些信息与机器码一起存储在内存中，供 .NET 运行时使用。最终的可执行本地机器码 (Native Code)​​ 块，以及与之关联的 Debug, GC, EH Info​ 数据结构；

using System;
using System.Runtime.InteropServices;

namespace ConsoleApplication
{
    public class Program
    {
        public static void Main(string[] args)
        {
            for (int x = 0; x < 3; ++x)
            {
                Console.WriteLine(x);
            }
        }
    }
}

IL to import:
// 无操作 (No Operation)。这是一条空指令，通常由编译器插入用于调试目的（例如设置断点）或对齐。在发布版本中，编译器通常会优化掉这些指令
IL_0000  00                nop

// 将 32 位整数常量 0 压入操作数栈顶。
// ​C# 对应：​​ int x = 0;中的 0
IL_0001  16                ldc.i4.0                  ; 运行堆栈 [ 0 ]

// 将操作数栈顶的值弹出，并存储到索引为 0 的局部变量中
// stloc.0 = store local to index 0
IL_0002  0a                stloc.0                   ; 运行堆栈 [ ], 保存到本地变量0 (x = 0)

// 无条件短跳转 (Branch Short)。立即跳转到偏移量 0x0d(十进制 13) 处的指令，即 IL_0012
// for循环初始化完成后 (x = 0)，​直接跳转到循环条件检查部分​ (x < 3)。这是 for循环结构在 IL 中的典型实现方式
IL_0003  2b 0d             br.s         13 (IL_0012) ; 跳转到IL_0012

// 无操作 (同上 IL_0000)。可能标记循环体开始或用于调试
IL_0005  00                nop

// 将索引为 0 的局部变量的值加载到操作数栈顶
// ldloc.0 = load local from index 0
IL_0006  06                ldloc.0                   ; 运行堆栈 [ x ]

// 调用由指定元数据令牌 (0xA00000C) 标识的方法。这个令牌指向 Console.WriteLine(int)方法
IL_0007  28 0c 00 00 0a    call         0xA00000C    ; 运行堆栈 [ ], 调用Console.WriteLine, 这里的0xA00000C是token

IL_000c  00                nop
IL_000d  00                nop         

// 再次将索引为 0的局部变量 x的值加载到操作数栈顶.加载变量 x的值，准备进行 ++x操作
IL_000e  06                ldloc.0                   ; 运行堆栈 [ x ]
IL_000f  17                ldc.i4.1                  ; 运行堆栈 [ x, 1 ]
IL_0010  58                add                       ; 运行堆栈 [ x+1 ]
IL_0011  0a                stloc.0                   ; 运行堆栈 [ ], 保存到本地变量0 (x = x + 1)
IL_0012  06                ldloc.0                   ; 运行堆栈 [ x ]
IL_0013  19                ldc.i4.3                  ; 运行堆栈 [ x, 3 ]
IL_0014  fe 04             clt                       ; 运行堆栈 [ x<3 ]
IL_0016  0b                stloc.1                   ; 运行堆栈 [ ], 保存到本地变量1 (tmp = x < 3)
IL_0017  07                ldloc.1                   ; 运行堆栈 [ tmp ]
IL_0018  2d eb             brtrue.s     -21 (IL_0005); 运行堆栈 [ ], 如果tmp为true则跳转到IL_0005
IL_001a  2a                ret                       ; 从函数返回

  • 分析：
    • IL 被分成了好几组 (BB02~BB05), 这里的 BB 是 BasicBlock 的缩写, 一个 BasicBlock 中有多个语句 (Statement), 一个语句就是一棵树 (GenTree)；
    • 上面的文本对应了以下的结构 (又称 HIR 结构)：
    • BasicBlock: 保存了一组语句, BasicBlock 内原则上跳转指令只会出现在最后一个语句；
    • Statement: 一个语句就是一棵树, 在内部 Statement 也是一个 GenTree 的子类 (GenTreeStmt)；
    • GenTree: 组成树的节点, 有很多不同的类型例如 GenTreeUnOp(unary op), GenTreeIntCon(int constant)；
    • 有人可能会好奇为什么上面的 BasicBlock 从 02 开始, 这是因为 01 是内部用的 block, 里面会保存函数开始时运行的内部处理；
  • 代码：RyuJIT 的前端会不断的修改 HIR 结构, 做出各种变形和优化

JIT 的数据结构

HIR 高层中间表示

概念：JIT 的数据结构

  • 图示：HIR 的数据结构
  • 概述：
    • HIR 的核心思想是将代码表示为 ​基本块 (Basic Blocks)​​ 的序列。每个基本块内部包含一系列 ​语句 (Statements)​。而每个语句本身则由一棵 ​表达式树 (Expression Tree)​​ 来表示其计算逻辑；
    • 同时，编译器维护一个关于方法中所有 ​局部变量 (Local Variables)​​ 信息的表；
  • 概念：Compiler (编译器对象)​
    • 这是整个 JIT 编译过程的中心控制器和状态持有者。它包含了编译当前方法所需的所有全局信息和上下文；
    • fgFirstBB:​​
      • 这是一个指针，指向方法控制流图 (Control Flow Graph, CFG) 中的 ​第一个基本块 (BasicBlock)​。这是遍历整个方法代码的入口点；
    • IvaTable (或类似结构，如 lvaTable)​：
      • 这是一个数组或表的 ​索引​ 或 ​基址；
      • 它指向 ​局部变量描述符表 (LclVarDsc 数组)​​ 的起始位置，Iva 能代表 "Indexed Variable Address"
  • 概念：局部变量描述符 - LclVarDsc
    • 描述方法中一个局部变量的所有关键属性；
    • 编译器为方法中声明的每个局部变量（包括编译器生成的临时变量）创建一个 LclVarDsc 条目，表达式树 (GenTree) 中引用该变量的节点（如 LCL_VAR 加载变量，STORE_LCL_VAR 存储变量）会通过变量索引 (lvIndex) 链接回对应的 LclVarDsc 条目；
  • 概念：Basic Block
    • 表示代码中一个线性执行的指令序列，它有一个唯一的入口点​（第一条指令）和一个唯一的出口点​（最后一条指令）。基本块是控制流分析、数据流分析和优化的基本单位。循环、条件分支等结构通过基本块之间的跳转关系来表示；
    • bbNext, bbPrev：
      • 指向控制流图中下一个和上一个基本块的指针；
      • 这些指针通常按照代码的布局顺序 (Layout Order)​​ 连接基本块，形成一个双向链表。这方便了顺序遍历基本块（例如按源码顺序）；
    • bbStmtList：
      • 这是一个指针，指向属于这个基本块的第一条语句 (GenTreeStmt)​；
      • 通过这条语句，可以遍历该基本块内的所有语句；
  • 概念：GenTreeStmt 语句节点 - Statement Node
    • ​表示一个完整的语句。在 HIR 中，语句是位于基本块内、表达式树之上的一个层级。它对应着源代码中的一个语句或 IL 中的一组有明确语义边界的指令（例如：一个赋值、一个方法调用、一个 return）
    • ​gtStmtExpr：
      • 这是该语句的核心，是一个指向 ​表达式树根节点 (GenTree)​​ 的指针。这棵表达式树完整地描述了该语句所执行的计算和操作；
  • 概念：GenTree 表达式树
    • 表示计算、操作和值；
    • 这是 HIR 最核心的数据结构，用于构建表达式树。每个节点代表一个操作（如加法 +、加载变量、方法调用）或一个值（如常量、局部变量引用）
    • 节点通过操作数指针 (gtOp1, gtOp2等) 连接起来形成树形结构，精确地表达了计算的依赖关系和操作数的来源；

JIT 的触发

概念：JIT 的触发

  • 概念：Stub 与 JIT
    • 当方法首次被调用时，CLR 检查其机器码是否已生成。若未生成，JIT 编译流程启动；
    • 这一过程通过 ​方法表（Method Table）​​中的函数入口指针实现：初始在方法表中，方法的指针指向一个 ​桩 Stub 而非实际代码；
    • 若多个线程同时首次调用同一方法，CLR 通过 ​锁机制（如 SpinLock）​​ 确保仅一个线程触发 JIT 编译，其他线程等待编译完成后再执行缓存代码；
    • CLR 使用自旋锁（SpinLock）​​ 或其他同步原语来实现这种保护。自旋锁的特点是当线程获取锁失败时，不会立即阻塞，而是循环尝试获取锁，这在锁竞争不激烈的场景下性能较好；
  • 分析：多线程下的 JIT 编译
    • 第一个到达的线程获取锁并开始 JIT 编译，其他后续线程在尝试获取锁时会发现锁已被占用，于是它们会自旋等待直到编译完成，然后直接使用已编译好的机器代码；
  • 概念：Compiler 实例
    • JIT 会为准备的函数创建一个 Compiler 实例，Compiler 实例储存了 BasicBlock 列表等编译时需要的信息。一个正在编译的函数对应一个 Compiler 实例，函数编译后 Compiler 实例会被销毁；
    • Compiler 实例负责将 IL 代码逐行读取，并将其转换为特定平台的机器指令。这个过程包括解析 IL 指令、进行各种优化（如内联、循环展开和死代码消除），最终生成高效的本地机器代码；
    • 编译完成后，生成的机器代码会被缓存到动态分配的内存块中，同时方法表中的方法指针会被更新为指向这个内存地址。此后，该方法的调用将直接执行缓存中的机器代码，不再需要 JIT 编译；

补充：对类型方法的 JIT 编译
  • 在 CLR 为每个类型创建的内部数据结构中，每个方法都有一个对应的记录项，其中包含一个地址字段指向方法的实现。当方法首次被调用时，这个地址字段初始指向一个特殊的未编档函数：JITCompiler 函数，而不是实际的方法代码；
  • 在方法首次被调用前，这个指针指向的是一个桩（Stub）​，即指向 JITCompiler 函数的地址。这个桩的作用是拦截方法的首次调用，触发 JIT 编译过程。只有当 JIT 编译完成后，这个指针才会被修改为指向新编译生成的本地机器代码的内存地址；
