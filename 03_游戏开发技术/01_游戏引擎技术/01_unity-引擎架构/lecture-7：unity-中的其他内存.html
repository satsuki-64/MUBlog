Lecture 7：Unity 中的其他内存

C# 非托管内存

C# 非托管内存基础概念

概念：C# 非托管内存

  • 概念：
    • C# ​非托管内存​ (Unmanaged Memory) 是指不由 .NET 垃圾回收器 (GC) 管理的内存区域；
    • 它让开发者能够手动控制内存的分配和释放，从而避免 GC 带来的性能开销，特别适用于需要频繁分配和释放内存的场景；
    • 可以使用 Unity 核心 API 中的 Unity.Collections 命名空间（包括 NativeArray），以及 Unity Collections 软件包中的数据结构来访问 C# 非托管内存；
    • 如果使用 Unity 的 C# Job 系统或 Burst，则必须使用 C# 非托管内存；
  • 实现：
    • 在 Unity 中，非托管内存主要通过 System.Runtime.InteropServices.Marshal 类进行管理。该类提供了一系列方法，如 AllocHGlobal 和 FreeHGlobal，用于从全局堆中分配和释放内存；
    • 其核心原理是绕过 .NET 的垃圾回收机制，直接通过系统调用申请和释放内存块，从而实现对内存的精确控制；
  • 作用：
    • （1）与原生插件交互​：在调用 C/C++ 编写的原生插件时，非托管内存可用于数据交换，避免额外的数据复制和转换开销；
    • （2）减少 GC 压力​：对于需要频繁创建和销毁的大型对象，使用非托管内存可以显著减少垃圾回收的频率，提升运行时性能；

使用方法

概念：使用方法

  • 流程：
    • （1）分配内存​：调用 Marshal.AllocHGlobal 方法，传入需要分配的字节数，返回一个 IntPtr 指针，指向分配的内存块；
    • （2）使用内存​：通过指针读写内存数据，常用方法包括 Marshal.Read/Write 系列方法或 Copy 方法进行数据复制；
    • （3）​释放内存​：使用 Marshal.FreeHGlobal 方法释放之前分配的内存，并将指针设置为 IntPtr.Zero，避免野指针；
  • 示例：
using System;
using System.Runtime.InteropServices;

public class UnmanagedMemoryExample
{
	// 1. 使用一个IntPtr指针来存储分配得到的非托管内存
    private IntPtr unmanagedMemory;
    
    public void AllocateMemory(int size)
    {
        // 2. 分配非托管内存，参数用于指定分配的内存大小
        unmanagedMemory = Marshal.AllocHGlobal(size);
        Console.WriteLine($"Allocated {size} bytes of unmanaged memory.");
    }
    
    public void FreeMemory()
    {
        // 3. 如果当前指针非空，则释放非托管内存
        if (unmanagedMemory != IntPtr.Zero)
        {
	        // 3.1 Free 当前分配的内存
            Marshal.FreeHGlobal(unmanagedMemory);
            // 3.2 将指针置为空，放置野指针
            unmanagedMemory = IntPtr.Zero;
            Console.WriteLine("Unmanaged memory freed.");
        }
    }
    
    public void UseMemory(byte[] data)
    {
        // 将数据从托管数组复制到非托管内存
        Marshal.Copy(data, 0, unmanagedMemory, data.Length);
        Console.WriteLine("Data copied to unmanaged memory.");
    }
}

原生内存

原生内存基础概念

概念：原生内存 - Native Memory

  • 概念：
    • 原生内存是 Unity 引擎底层（C++）管理的内存区域，用于存储引擎内部对象和第三方插件分配的内存，它与 C# 托管内存完全分离，不受 .NET 垃圾回收器 (GC) 的管理；
  • 作用：
    • 原生内存主要用于存储 Unity 引擎运行所需的各类核心资源，包括：
      • 纹理资源​ (Textures)：包括各种纹理格式的像素数据，通常占用较大的内存空间；
      • 网格数据​ (Meshes)：存储模型的顶点、法线、UV 等几何信息；
      • 音频数据​ (Audio Clips)：存储未压缩或压缩的音频采样数据；
      • 渲染资源​：包括帧缓冲区、渲染目标、着色器程序等图形 API 对象；
      • 物理引擎数据​：如物理碰撞体、刚体、关节等物理状态信息；
      • 第三方插件分配的内存​：例如原生插件通过 malloc 或 new 分配的内存块；
    • 与原生内存相关的 Unity 对象（如 Texture2D、Mesh、AudioClip 等）实际上是混合对象​,它们在托管堆中有一个 C# 包装对象，但实际数据存储在原生内存中；
    • 这种设计使得实际数据的生命周期由引擎管理，而不是 .NET GC；
  • 分类：原生内存的五种内存分配器
    • Unity 有五种分配器类型，每种类型都有不同的算法来将分配拟合到内存块中，因此它们对不同的分配很有用。分配之间的重要区别通常是持久性或分配寿命，它决定了分配应该放在哪里；
    • 长期（持久）分配进入堆和桶分配器，而短期分配进入线程安全的线性分配器和 TLS 分配器；
  • 区别：
  • 分析：托管内存和原生内存内存交互机制
    • UnityEngine.Object 对象（如 Texture2D、GameObject）在托管堆中存储轻量级的包装对象和元数据，并包含一个指向本机内存中实际资源的指针；
    • 当托管包装对象被 GC 回收时，并不意味着本机资源会被立即释放，后者仍需通过 Unity 的引用计数机制来管理；

内存分配器算法

动态堆分配器

概念：动态堆分配器 - DynamicHeapAllocator

  • 概念：
    • 动态堆分配器是 Unity 主要的内存分配器之一，其设计目标是：
      • （1）高效处理动态内存请求​：适应不同大小和生命周期的内存分配需求；
      • （2）减少内存碎片​：通过特定算法优化内存布局；
      • （3）提供性能保障​：力求使分配和释放操作在恒定时间（)）​​ 内完成；
    • 当分配请求的数量小于半个内存块时，动态堆分配器会使用其管理的块进行处理。如果请求的内存达到或超过半个块的大小，分配器无法处理，Unity 会回退（Fallback）到使用虚拟内存 API（如 VirtualAlloc 或 mmap）​​ 来直接分配大块内存；
    • 这种机制设计能防止超大分配请求破坏分配器内部的结构和效率，但也意味着这些分配不再受动态堆分配器的优化管理；
    • 最棘手的部分是设置块的大小。根据不同平台，更大的块效率更高，碎片更少，但对于内存有限的平台来说灵活性差。最大块为 256 M，最小块为 128 k，如果 64 位架构，使用更大的 Region 来保存多个块 MEMORY_USE_LARGE_BLOCKS，如果分配失败会会退到虚拟内存分配，效率更差；
    • Log 中 [ALLOC_DEFAULT_MAIN] 下关注 Peakusageframecount 字段，查看大多数帧分配内存的范围，关注内存分配峰值 Peak Allocated memory 字段，以及 Peak Large allocation bytes 没有使用 TLSF 分配的内存大小；
  • 图示：动态堆分配示例使用报告
    • Requested Block Size 16.0 MB：表示 TLSF 的内存块大小设置为 16 MB；
    • Peak Block count 2：峰值时分配了 2 个这样的内存块，总计 32 MB 原始空间；
    • Peak Allocated memory 54.2 MB：峰值时总共分配了 54.2 MB 的内存。这超过了 32 MB，说明有大量内存是通过回退机制（虚拟内存 API）​​ 分配的；
    • Peak Large allocation bytes 40.2 MB：在峰值分配的内存中，有 40.2 MB 属于大分配​（单个分配 >= 8MB），它们直接来自虚拟内存，而非 TLSF 管理的内存块；
    • Peak usage frame count：大多数帧（497 帧）的内存使用峰值在 16-32 MB 之间，可能有一帧（如加载帧）的峰值达到了 32-64 MB；

概念：TLSF - Two-Level Segregated Fit

  • 概念：
    • Unity 的动态堆分配器采用了 ​ TLSF (Two-Level Segregated Fit)​​ 算法，这是一种专为实时系统设计的内存分配算法，以其恒定的分配/释放时间和低碎片化特性而闻名；
  • 特点：
    • （1）无论堆的状态如何，malloc 和 free 操作都能在 ​O (1) 时间复杂度内完成，满足了实时系统的苛刻要求；
    • （2）通过立即合并（Immediate Coalescing）​​ 释放的块（当相邻块空闲时立即合并）和 Good-Fit 策略​（寻找足够大但又不会大浪费的空闲块）来有效减少内存碎片；
    • （3）每次内存分配仅产生约 ​ 4 字节的额外开销，用于管理元数据；
  • 概念：TLSF 算法的核心数据结构
    • TLSF 通过两级位图（Two-Level Bitmap）​​ 和分级空闲块链表来管理动态内存池中的空闲块：
    • （1）内存块大小分类：两级索引
      • TLSF 将空闲内存块按其大小划分到不同的“容器”中，这些容器组织成两级数组；
      • 第一级索引：
        • 根据内存请求大小的大小级（Size Class）进行划分，通常按 2 的幂次方​（如 16, 32, 64, 128, ...）进行粗粒度分类；
        • 第一级索引 f 的计算方法是找到请求大小 r 的二进制最高位（MSB）的位置。例如，对于 r=460（二进制 111001100，最高位是第 9 位，从 0 开始计数），f 就是 8，因为 ；
      • 第二级索引：
        • 在每个第一级的大小范围内，再进行更细粒度的线性划分；
        • 第二级索引 s 的计算公式通常为：，其中 SLI 表示 Second Level Index，是第二级划分的数量，这是一个可配置的参数；
        • 例如，SLI=4 表示将每个第一级区间分为 4 份；
    • （2）位图标记空闲链表：
      • 每个大小类别（由 f 和 s 共同确定）都对应一个双向链表，链接着该特定大小的所有空闲内存块；
      • 系统使用两个位图（第一级位图和第二级位图）来标记哪些大小类别的链表非空​（即有空闲块可用），这允许分配器通过几次位操作快速找到满足请求的空闲链表，而无需遍历所有链表；
    • 图示：两级索引
  • 实现：算法的分配过程
    • 当收到大小为 r的内存分配请求时：
    • （1）计算索引​：根据上述公式计算出第一级索引 f 和第二级索引 s；
    • （2）查找空闲块：检查对应的位图，找到能满足 r 大小的最小空闲块链表。如果该链表非空，则从链表中取出第一块进行分配；
    • （3）分割块（可选）​​：如果取出的空闲块远大于请求大小，可能会将其分割，剩余部分放回相应的空闲链表中；
  • 实现：算法的释放与合并
    • 当内存被释放时：
    • （1）立即合并​：TLSF 会立即检查与该释放块物理相邻的内存块是否也是空闲的，如果是，就将它们合并成一个更大的空闲块；
    • （2）​更新链表和位图​：合并后的新空闲块会根据其新的大小被重新插入到对应的空闲链表中，并更新相应的位图状态；

桶分配器

概念：桶分配器 - BucketAllocator

  • 概念：
    • 桶分配器是一种快速无锁分配器，用于执行小型分配。通常，桶分配器用作加速小型分配的第一步，然后再转到堆分配器；
    • 分配器为分配保留内存块，每个块被划分为大小为 16KB 的子部分。此设置不可配置，并且不会显示在用户界面中。每个子部分又进一步划分为分配。分配大小是配置的固定大小（称为粒度）的倍数；
    • 核心思想是预先分配好多个不同大小的内存块（即“桶”）​，每个桶专门负责一种固定大小的内存分配请求，从而避免通用分配器在处理小对象时的碎片化和性能开销；
  • 概念：
    • 每个 bucket 由固定大小粒度 granularity 表示，如果 granularity 为 16 字节大小，则用于分配 16/32/48/64/ 字节的内存；
    • 如果是 development 版本，granularity 在设置大小的基础上增加 40 个字节；
    • 分配器为分配保留内存块 Block，每个块被划分为 16kb 的子段 (subsections)，并且不可配置。Block 块只能增长, 并且需要是固定 16kb 大小的整数倍。分配是固定大小无锁的，速度快，通常作为进入堆分配器之前用来加速小内存分配的分配器
    • Log 日志中可以通过查看 [ALLOC_BUCKET] 字段来看是否有 FailedAllocations.Bucketlayout 字段以及 PeakAllocatedbytes 字段与 LargeBlocksize 字段的利用率，来判定大小是否分配合适，分配失败会会退到 DynamicHeapAllocator 或 UnityDefaultAllocator 分配器上，效率变差；

  • 概念：桶分配器的核心数据结构
    • （1）分桶策略：
      • 系统会预先定义一系列固定大小（称为大小类别，Size Classes）​，例如 16B、32B、64B、128B 等，每个大小类别对应一个独立的“桶”；
      • 当一个内存分配请求到来时，分配器会向上取整到最近的标准大小类别，然后交由对应的桶来处理；
      • 例如，请求分配 18B 的内存，可能会被分配在 32B 的桶中；
    • （2）内存结构：
      • 每个桶在内存中的管理结构通常包含以下部分：
        1. 内存块 (Memory Chunk)​​：从系统申请的一大块连续内存，被划分为多个大小相等的单元 (Cell)​；
        1. 空闲链表 (Free List)​​：一个将所有这些空闲单元连接起来的单向链表，链表的每个节点实际上就是空闲单元本身（或一个指向下一个空闲单元的指针）。链表头指针指向第一个可用的空闲单元；
  • 实现：算法的分配过程
    • （1）确定大小类别​：根据请求的分配大小，确定应使用哪个大小的桶；
    • （2）​检查空闲链表​：从该桶的空闲链表头部获取第一个空闲单元；
    • （3）调整链表​：将链表头指针指向下一个空闲单元；
    • （4）​返回内存指针​：将步骤 2 中获取的空闲单元的内存地址返回给调用者；
  • 实现：算法的释放过程
    • （1）​确定所属桶​：根据要释放的内存地址，确定它来自于哪个桶（分配器通常通过元数据或地址范围来跟踪）
    • （2）放回空闲链表​：将该内存单元放回其所属桶的空闲链表头部；
    • （3）更新链表​：让该单元指向原来的链表头，并更新链表头指针指向这个新释放的单元；
  • 示例：引擎内部管理与《奥日 2》的优化案例
    • Unity 引擎底层在使用桶分配器管理小型、高频分配的对象，例如 GameObject 和 Component 的内部数据结构、事件参数、渲染命令等，这在《奥日 2》的 Switch 移植优化中得到了很好的体现；
    • Moon 工作室与 Unity 企业技术支持团队合作时发现，游戏在开始前就填满了部分桶内存。通过适当增加内存桶的数量，为游戏期间分配的小内存提供了额外空间。这使得这些小内存块能够由桶分配器高效处理，而不再交给速度较慢的动态堆分配器。动态堆分配器因此可以更专注于处理大型内存块，其碎片化程度降低，游戏也能更长时间地流畅运行，最终实现了 60 帧的目标；

双线程分配器

概念：双线程分配器

  • 概念：
    • 它是将 2 个 DynamicHeapAllocator 实例与 1 个 BucketAllocator 封装到一起，其中 BucketAllocator 负责小的共享内存分配，1 个无锁的 DynamicHeapAllocator 用于主线程分配，另外一个 DynamicHeapAllocator 负责其他线程的共享分配，但是分配与回收时需要加锁；
  • 概念：
    • 双线程分配器是一个包装器，它将动态堆和桶分配器结合起来。更具体地说，它结合了：
    • （1）两个动态堆分配器：
      • 一个用于主线程的无锁分配器；
      • 一个用于所有其他线程的分配器，它在分配和释放时锁定；
      • Unity 使用这些分配器来进行对于桶分配器来说太大的分配；
      • 动态堆分配器使用内存块等于或大于半个块的分配将转到虚拟内存系统，而不是动态堆分配器；
    • （2）用于小分配的桶分配器：
      • 如果桶分配器已满，则分配将溢出到动态堆分配器中；
  • 使用：
    • Log 中 [ALLOC_DEFAULT] 下分别关注 [ALLOC_BUCKET]、[ALLOC_DEFAULT_MAIN]、[ALLOC_DEFAULT_THREAD] 字段下一个 BucketAllocator 与 2 个 DynamicHeapAllocator 的分配信息，其中尤其关注 Peak main deferred allocation count 字段，这个字段代表需要在主线程回收的分配队列；
    • 负责其他线程的共享分配的 DynamicHeapAllocatord 对应 C# 非托管内存的 Allocator.Persistent；

基于栈的分配器

概念：基于栈的分配器 - TLS Stack Allocator

  • 概念：
    • 用于快速临时分配的堆栈分配器。它是最快的分配器，几乎没有开销，并且可以防止内存碎片，对应 C# 非托管内存的 Allocator.Temp 分配类型；
    • 它是基于后进先出 LIFO 的算法，分配的内存生命周期在一帧内，如果分配器使用超过了配置的大小，Unity 会增加分配块大小，但会是配置设置的 2 倍大小；
    • 如果线程堆栈分配器满了，分配器将会退到线程安全线性 job 分配器。这时一帧内允许有 1-10 分配，甚至如果是加载期间可以允许几百次。但如果每帧的分配数字增加可以考虑增加块大小；
  • 使用：
    • Log 日志 [ALLOC_TEMP_TLS] 下关注 [ALLOC_TEMP_MAIN] 与 [ALLOC_TEMP_Job.Worker xx] 中的 Peakusage frame count于Peak Allocated Bytes 字段，也要关注 Overflow Count 字段，是否发生了分配器溢出会退的情况；

线程安全的线性分配器

概念：线程安全的线性分配器 - Thread SafeLinearAllocator

  • 概念：
    • 用于快速无锁分配 Job 线程的缓冲区，并在 Job 完成后释放缓冲区，对应 C# 非托管内存的 Allocator.TempJob；
    • 采用循环先进先出 FIFO 算法，先分配内存块，然后在这些内存块内进行线性分配内存；
    • 所有可用块都会存到一个池中，当一个块满时再从池中提取一个新的可用块。当分配器不再需要块中内存时，清理该块并放回到可用池中；
    • 快速清理分配可用块非常重要，因此 Job 的分配尽量在几帧内完成；
    • 如果所有块都在使用中，或者一个分配对于一个块太大的情况下，则会退到主堆分配器，分配效率会下降；
  • 使用：
    • Log 日志 [ALLOC_TEMP_JOB_4_FRAMES] 中我们需要关注 Overflow Cout(toolarge) 与 OverflowCount(full) 字段，判断是否发生了分配器溢出会退；
