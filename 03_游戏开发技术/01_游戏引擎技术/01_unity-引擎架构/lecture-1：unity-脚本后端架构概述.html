Lecture 1：Unity 脚本后端架构概述

什么是脚本后端

两类脚本后端

概念：什么是脚本后端

  • 概念：
    • 脚本后端 (Scripting Backend)是 Unity 引擎将 C# 代码转换为可执行代码的核心组件，它决定了代码如何被编译和执行，直接影响项目的性能、平台兼容性和开发体验；
    • 脚本后端可以理解为脚本运行时环境，它是 Unity 引擎与 C# 代码之间的桥梁，负责将开发者编写的 C# 代码转换为目标平台可执行的机器码；
  • 概念：Unity 中的脚本后端分类
    • Unity 目前提供两种脚本后端选项：Mono 和 IL2CPP，它们采用不同的编译技术和执行策略；
    • Mono 使用即时 (JIT) 编译，并在运行时按需编译代码；
    • IL2CPP 使用预先 (AOT) 编译，并在应用程序运行前编译整个应用程序；
  • 概念：Mono
    • Mono 是基于开源 .NET 实现的脚本后端，是 Unity 最早采用的脚本运行时环境；
    • 工作流程：
        1. C# 代码通过 Roslyn 编译器编译为 .NET IL (中间语言)
        1. 运行时 Mono 虚拟机加载 IL 代码；
        1. 在方法首次执行时，JIT 编译器将其编译为当前平台的机器码；
        1. 后续调用直接执行已编译的机器码；
  • 概念：IL2CPP
    • IL2CPP (Intermediate Language To C++) 是 Unity 开发的替代 Mono 的脚本后端，旨在解决 Mono 的局限性；
    • 工作流程：
        1. C# 代码通过 Roslyn 编译器编译为 .NET IL (中间语言)；
        1. Unity 应用代码剥离，移除未使用的代码；
        1. IL2CPP 将 IL 转换为标准 C++ 代码；
        1. 使用平台原生编译器 (如 Clang、GCC) 将 C++ 编译为机器码；
        1. 生成目标平台的可执行文件或动态库；
  • 总结：
  • 概念：设置方法
    • 在 Unity Editor 中：Edit > Project Settings > Player > Other Settings > Scripting Backend 此路径进行设置；

Unity 脚本后端与 .NET 兼容性

概念：Unity 脚本后端与 .NET 兼容性

  • 概念：
    • Unity 的脚本后端与 .NET 兼容性密切相关，两者共同决定了开发者能够使用的 API 范围、代码执行方式以及跨平台能力；
  • 概念：两类 .NET API
      1. .NET Standard 2.0
      • .NET Standard 2.0 是微软推出的跨平台 API 规范，旨在统一不同 .NET 实现（如 .NET Core、Mono、Xamarin 等）的 API 集合；
      • 它通过定义一套标准化的接口，确保库开发者编写的代码能在所有支持该标准的平台上运行；
      • 包含超过 32,000 个 API，比 .NET Standard 1.6 增加 140%，覆盖了 .NET Framework 4.6.1 中常用的 API（如 AppDomain、TimeZoneInfo 等）
      • 新增对 System 命名空间的增强支持，包括垃圾回收控制（GC 类）、字符串处理（String 类）和异常序列化功能；
      • 支持 .NET Core 2.0+、Mono 5.4+、Xamarin.iOS 10.14+ 等实现，覆盖 Windows、Linux、macOS 和移动端；
      1. .NET Framework 4.x
      • .NET Framework 4.x 是微软 Windows 专属的开发框架，提供 CLR 运行时和类库，用于构建桌面、Web 和服务端应用；
      • 其最新版本（如 4.8.1）仍广泛用于企业级 Windows 开发；
      • 引入并行计算支持（TPL 库）、增量垃圾回收（减少 GC 卡顿）和动态语言运行时（DLR），集成 Windows API（如 COM+、DirectX），适合开发 Windows 桌面应用和 ASP.NET MVC 项目;

基础类库 BCL

概念：基础类库 BCL

  • 概念：什么是 BCL
    • BCL（Base Class Library）是 .NET 平台的核心类库集合，为所有 .NET 应用程序提供基础功能支持。它包含数千个预定义的类、接口和值类型，涵盖字符串处理、集合操作、文件 I/O、网络通信、多线程编程等核心功能；
    • 核心类型​：System.Object（所有类型的基类）、System.String、System.ValueType（值类型基类）等；
    • BCL 可被 C#、VB .NET、F# 等 .NET 语言共用；
  • 概念：Mono 与 BCL
    • Unity 使用的 Mono 版本对 BCL 进行裁剪，移除不必要模块（如 System.Web）以减小运行时体积，对游戏开发常用类型（如 Vector3）进行特殊优化；
    • 不同的 .NET 规范下使用不同的 BCL，但是它们三者共享核心 BCL 的 API 设计：
    • 其中，.NET Framework BCL 功能最全，但仅限 Windows。Mono BCL 兼容性较弱，但支持更多非 Windows 平台特性；
    • 在跨平台项目​中优先使用 .NET Standard 兼容的 BCL API（如 System.IO.Path 而非 System.IO.FileInfo）
  • 关系：
  • 概念：数据流图
总结：BCL 和 CLR 的关系
  • BCL 中的代码同样是托管代码，需要 CLR 来加载、编译（JIT）和执行。当你调用 BCL 中的类和方法时，CLR 负责将其编译成机器码并管理其运行时的内存、安全性和线程等；
  • BCL 封装了许多基础功能，但其底层实现有时会依赖于 CLR 提供的更核心的服务。例如，BCL 中的 File 类进行文件读写时，最终会通过 CLR 与操作系统交互并接受其安全策略（如代码访问安全性 CAS）的检查；
注意：关于 BCL、CLR 和跨平台
  • 从 .NET Core​（包括现在的 .NET 5 及以上）开始，BCL 和 CLR 都被重新设计为跨平台的。这意味着它们可以在 Windows、Linux 和 macOS 上运行，并为这些平台提供一致的基础 API 和运行时服务；

核心概念

托管执行

概念：托管执行

  • 概念：
    • 托管执行环境（Managed Execution Environment）是指代码在运行时由“托管”系统（如 .NET 的 CLR）控制和管理的执行方式；
    • CLR 提供内存管理、类型安全、异常处理、安全性等一系列高级服务，使开发者能专注于业务逻辑而非底层资源管理；
    • 托管代码与非托管代码的核心区别在于：前者运行在 CLR 的监管下，享受自动化服务，后者则需开发者自行管理内存、类型与异常，易出错且安全性低。CLR 的设计目标是降低编程复杂度，提高开发效率和系统健壮性；
  • 概念：两类代码
    • 托管代码（Managed Code） ：经 CIL 编译、包含元数据、由 CLR 执行和管理的代码；
    • 非托管代码（Unmanaged Code） ：如原生 C/C++，直接运行于操作系统与硬件上，不受 CLR 控制；
  • 概念：CLR 系统架构
  • 概念：托管程序的执行流程
    • 托管程序的执行流程包括：加载程序集 → 验证元数据和 CIL → JIT 编译为本地代码 → 运行时服务（GC、异常、安全等）→ 程序退出时资源释放；

即时编译 JIT

概念：即时编译 JIT 与预先编译 AOT

  • 概念：即时编译 JIT
    • 即时编译 (Just-In-Time Compilation) 是 CLR (公共语言运行时)的核心执行机制，它实现了 .NET 平台"一次编译，到处运行"的跨平台愿景；
    • JIT 编译的本质是在程序运行时将中间语言 (IL) 代码动态转换为目标平台的本地机器码，兼具解释执行的灵活性和原生代码的高性能；
    • 其具体架构分为：
  • 概念：JIT 执行流程
      1. 方法调用触发：
      • 当首次调用某个方法时，CLR 检查方法表发现该方法的实现尚未编译，此时调用 JIT 编译器；
      1. 加载元数据：
      • JIT 编译器从程序集的元数据中查找该方法的 IL 代码，同时加载相关的类型信息和依赖项；
      1. 代码验证：
      • 验证器对 IL 代码进行严格检查，确保类型签名正确，内存访问合法，不会执行危险操作 (如直接内存访问)；
      1. 编译与优化：
      • JIT 编译器将验证通过的 IL 代码转换为机器码，期间进行多级优化；
      1. 代码缓存与执行：
      • 编译完成的机器码被存入动态内存区域，同时更新方法表的调用入口；
      • 后续调用直接执行缓存代码，跳过编译过程；
  • 概念：JIT 分层编译策略
    • 快速 JIT (Tier 0)​​：生成未优化的机器码，快速完成初始编译，减少启动延迟；
    • 优化 JIT (Tier 1)​​：对热点方法进行深度优化，牺牲更多编译时间换取更好的运行时性能；
    • PGO 引导优化​：基于 Profile-Guided Optimization 收集的运行时信息 (如分支预测、调用频率) 进行针对性优化；

预先编译 AOT

概念：预先编译 AOT

  • 概念：
    • AOT (Ahead-of-Time) 编译是一种在程序执行前将中间语言 (IL) 完全编译为目标机器码的技术，与 JIT 的运行时编译形成对比；
    • 在 .NET 生态中，AOT 编译生成的代码可以直接在目标平台上运行，无需依赖实时编译组件；
    • AOT 并非 .NET Core 之后才出现的技术，早在 .NET Framework 时代就已存在。但在 .NET 5+ 中，AOT 技术得到了显著增强，形成了独立的编译体系；
  • 区别：
    • 与传统 CLR 不同，AOT 运行时(CoreRT)不是虚拟机，它移除了 JIT 组件，但保留了 GC 和有限的反射能力；
  • 概念：AOT 编译流程
      1. 源码编译​：Roslyn 编译器将 C# 代码编译为 IL 中间语言，生成托管 DLL；
      1. AOT 转换​：ILC (Intermediate Language Compiler) 工具将 IL 转换为目标文件 (.obj)；
      1. 链接​：使用 NativeAOT 的引导程序 (Bootstrap)调用链接器生成最终可执行文件；
      1. 执行​：生成的本地代码直接在目标平台运行，无需 JIT 参与；
  • 架构：
  • 分析：AOT 和 JIT 代码
// JIT代码
call JIT_TrialAllocSFastMP_InlineGetThread
mov qword ptr [rbp+20h],rax
mov rcx,qword ptr [rbp+20h]
call Program..ctor()

// AOT代码
// AOT代码直接使用虚函数表指针作为参数调用运行时快速分配函数，跳过了JIT的通用分配逻辑
lea rcx,[repro_Program::`vftable']
call RhpNewFast

类型系统与元数据

概念：类型系统与元数据

  • 概念：
    • 类型系统定义了如何在运行时表示和管理类型以及类型间的交互规则，而元数据则是描述这些类型及其关系的数据集合；
  • 概念：类型系统 - CTS
    • 类型系统的主要目标是提供跨语言集成的能力，它规定了所有. NET 语言都必须遵守的类型定义、使用和交互规则；
    • 无论开发者使用 C#、VB.NET 还是 F#，最终都会通过 CTS 统一为相同的运行时表示形式。这种设计使得不同语言编写的组件可以无缝互操作，共享相同的基类库和运行时服务；
  • 概念：元数据
    • 元数据作为数据的数据​，以结构化的方式记录了程序集中包含的所有类型信息、成员签名、依赖关系和安全属性等；
    • 每个 .NET 程序集都嵌入了完整的元数据，这使得 CLR 能够在运行时了解代码结构而无需依赖外部信息源。元数据与 IL 代码紧密绑定，始终保持同步，避免了传统 C/C++ 开发中头文件与实现文件可能不同步的问题；
  • 概念：类型系统的运行时实现
    • 类型系统在 CLR 内部通过一系列相互关联的数据结构实现：
      1. MethodTable：
      • 每个加载的类型都有一个 MethodTable，包含类型的方法表、静态字段和同步块索引等核心信息。它是对象实例头部的一部分，垃圾回收器和 JIT 编译器都依赖它；
      1. EEClass：
      • 与 MethodTable 配合工作，存储类型的元数据信息、字段布局和继承关系等；
      • 这种分离设计减少了工作集，因为多个泛型实例可能共享相同的 EEClass 但拥有不同的 MethodTable；
      1. MethodDesc/FieldDesc：
      • 描述类型中的方法和字段，包含调用约定、参数类型等详细信息。JIT 编译时会将这些描述符转换为可直接调用的入口点；
      1. TypeDesc​：
      • 表示泛型参数、数组等特殊类型的描述符，参与类型推导和验证；
  • 概念：元数据的格式
    • 定义表 (Definition Tables) ​​：
      • 包括 TypeDef (类型定义)、MethodDef (方法定义)、FieldDef (字段定义)等，记录模块中定义的元素；
    • 引用表 (Reference Tables)​​：
      • 如 TypeRef (类型引用)、MemberRef (成员引用)，记录对外部程序集中类型的依赖；
    • 清单表 (Manifest Tables)​​：
      • 包含 Assembly (程序集)、File (文件)、ExportedType (导出类型) 等，描述程序集的身份和组成；

概念：类型加载完整流程

    1. 定位程序集​：
    • 根据引用信息 (名称、版本、公钥等)，在对应的应用程序域 (AppDomain) 中查找或加载目标程序集；
    1. 解析元数据​：
    • 从程序集的元数据中读取类型定义，验证其完整性和正确性；
    1. 分配数据结构​：
    • 创建并初始化 EEClass、MethodTable 等运行时数据结构；
    1. ​布局计算​：
    • 确定类型的实例大小、字段偏移量和虚方法表布局；
    1. ​安全检查​：
    • 验证当前安全上下文是否允许加载和使用该类型；
    1. 初始化​：
    • 执行类型的静态构造函数(如存在)并设置初始状态；
