Mono 基础概念

什么是 Mono

概念：Mono 基础概念

  • 概念：
    • Mono 是一个由 Xamarin 公司（先前是 Novell，最早为 Ximian）所主持的自由开放源代码项目；
    • 该项目的目标是创建一系列匹配 ECMA 标准（Ecma-334 和 Ecma-335）的 .NET 工具，包括 C# 编译器和通用语言架构。与微软的 .NET Framework（共通语言运行平台）不同，Mono 项目不仅可以运行于 Windows 系统上，还可以运行于 Linux，FreeBSD，Unix，OS X 和 Solaris，甚至一些游戏平台，例如：Playstation 3，Wii 或 XBox 360；
  • 概念：Mono 脚本后端
    • Unity 使用称为即时编译 (JIT) 的技术在运行时编译代码，Unity 使用开源 Mono 项目的一个分支；
    • 某些平台不支持 JIT 编译，因此 Mono 后端并非在所有平台上都适用。其他平台支持 JIT 和 Mono，但不支持提前编译 (AOT)，因此无法支持 IL2CPP 后端；
    • 当平台能够支持这两种后端时，Mono 为默认后端；
  • 概念：使用 Mono 构建项目
    • 在项目设置当中进行配置；

Mono 与 .NET 其他框架

概念：Mono 与 .NET 其他框架

  • 图示：Mono 与 .NET 其他框架的关系
  • 概念：Mono 与 .NET 其他框架的目标
    • Mono：
      • 最初由社区主导的开源项目，支持 Linux、macOS、iOS、Android 等非 Windows 平台，基于 .NET 标准但独立演进，通过 Xamarin 提供原生移动开发能力，直接调用平台 API，性能接近原生；
    • .NET (Core/Framework)​​：
      • .NET Framework ​：Windows 专属，成熟稳定，支持桌面软件 WPF/WinForms、企业级应用 WCF/WF 等；
      • ​ .NET Core ​：微软主导的跨平台、模块化框架，专注云原生、微服务和高性能 Web，如 ASP.NET Core
  • 概念：技术实现
    • Mono 支持 JIT/AOT/解释器，尤其依赖 AOT 解决 iOS 等平台限制；
    • .NET Core 使用 CoreCLR（JIT）​​ 和 CoreRT（AOT）​，优化服务器端场景；
  • 概念：应用场景
注意：Unity 的技术转型
  • Unity 引擎正在经历一个重要的技术转型期。传统上，Unity 一直使用 Mono 作为其主要的脚本运行时环境，这使得开发者能够用 C# 等 .NET 支持的语言开发应用程序，并在支持的平台上运行；
  • 然而，随着技术发展，Unity 开始将运行时从 Mono 迁移到 .NET Core CLR*（现已统一为 .NET），这主要是因为 .NET Core 提供了完全重新设计的跨平台基础，比 Mono 的跨平台策略更加统一和现代化，并且提供了更现代化的 API，而 Mono 主要兼容旧版 .NET Framework；

Mono 类库

概念：Mono 类库

  • 概念：
    • 类库为应用程序开发提供了一套全面的功能设施。虽然主要用 C# 编写，但由于通用语言规范（Common Language Specification, CLS）​的支持，它可以被任何 .NET 语言调用；
    • 类库通过命名空间（Namespaces）​组织，并以程序集（Assemblies）​的形式部署。通常所说的“.NET 框架”核心即指此类库；
  • 概念：命名空间
    • 命名空间是一种逻辑分组机制，将功能相似的类按层次结构组织，避免命名冲突。其结构采用点分隔的单词表示；
    • 顶级命名空间​：大多数 .NET 框架类库以 System 为根，例如 System.IO（文件与流操作）、System.Threading（多线程支持）等；
  • 概念：程序集
    • 程序集是类库的物理部署单元，通常为 .dll 文件，比如 mscorlib.dll、System.Data.dll，但与 Win32共享库 有本质区别；
    • 一个程序集可包含多个文件（如主模块、资源文件等），同时一个命名空间可能分散在多个程序集中，如 System.Drawing 功能分布在 System.Drawing.dll 和 System.Windows.Forms.dll 中；
    • 其中：
      • mscorlib.dll：核心类型（如基础数据类型）
      • System.Web.dll：ASP.NET 功能支持；
    • Mono 作为 .NET 的开源实现，其类库结构与微软 .NET 框架高度兼容，但部分命名空间（如 Mono.Security）为 Mono 特有；

Mono 与 CLR

概念：Mono 与 CLR

  • Mono 的运行时流程分为以下阶段：
    1. ​编译阶段​：
    • C# 等语言通过 Mono 编译器（如 mcs）生成符合 CLS 的 CIL 代码，存储为 .exe 或 .dll 程序集；
    1. 执行阶段：
    • 第一步：Mono 运行时解析程序集依赖（搜索 GAC、MONO_PATH 路径等）
    • 第二步：进行 JIT 编译，将 CIL 转换为目标平台的本地代码（如 x86/ARM 指令）
    • 第三步：托管执行，在运行时管理内存、线程和异常，确保类型安全

Mono 与 AOT

概念：Mono 与 AOT

  • 概念：
    • AOT 在运行前将程序集预编译为原生代码，默认 AOT 仅编译高频方法，保留部分 IL 供 JIT 补充，平衡优化与灵活性。完全 AOT ​则强制编译所有 IL 代码，禁用 JIT 引擎（通过 --full-aot 参数）
  • 优点：
      1. 启动时间短，避免运行时 JIT 编译开销；
      1. 生成的位置无关代码（PIC），通过 mmap 在多个进程间共享；
      1. 支持 JIT 无法启用的高级优化（如 -O=all 开启全部优化）；
  • 概念：完全 AOT
    • 完全 AOT 编译是 Mono 运行时的一种模式，​在程序运行前将所有中间语言（IL）代码静态编译为目标平台的原生机器码，生成的文件（如 .so 或 .dll）直接由操作系统加载执行；
    • 与 JIT（即时编译）不同，完全 AOT 模式下运行时禁止动态生成代码，适用于 IOS，因为 iOS 内核禁止内存页动态修改可执行权限，因此无法使用 JIT 编译，必须依赖 AOT；
    • 具体而言，通过 mono --aot=full 命令将程序集（如 mscorlib.dll）的 IL 代码转换为原生机器码；
  • 代码：
# 完全编译程序集
mono --aot=full mscorlib.dll
# 运行时不启用 JIT
mono --full-aot program.exe

分析：AOT 下泛型实例化的限制与解决方案

  • 分析：
    • 完全 AOT 对泛型的支持存在显著限制，主要源于静态编译无法动态推断类型参数​；
  • 概念：具体限制
    • 静态分析无法确定运行时调用的具体方法（如 virtual void PrintValues<T>(T[] values)）
    • 泛型类中的平台调用（如 [DllImport]）无法编译；
    • 通过 Property.SetInfo 对 Nullable<T> 赋值会失败；
    • Dictionary<TKey, TValue> 默认使用 EqualityComparer<TKey>.Default，后者依赖反射创建比较器，导致运行时崩溃；
    • System.Reflection.Emit 不可用，无法运行时生成 IL 或动态类型；
  • 解决：显式实例化泛型
static void ForceAOTCompilation() {
    var dummy1 = new NodeStack<int>(); // 显式实例化
    var dummy2 = new NodeStack<string>();
}

Mono 如何查找程序集

分析：Mono 如何查找程序集

  • 概念：
    • 应用程序代码通常分散在多个程序集中。除了应用程序自身的程序集外，所有 Mono 应用都会引用包含运行时核心类库的 mscorlib 程序集。应用程序可能使用任意数量的程序集，部分由运行时提供，部分为可选组件，其余可能由第三方开发者编写；
    • 运行时，Mono 会从以下三个位置查找运行程序所需的程序集：
      1. 执行程序集所在目录​：
      • 若应用程序使用的程序集非运行时提供，可将它们放在同一目录下以便 Mono 查找；
      1. MONO_PATH 环境变量​：
      • 若程序集未在第一步找到，Mono 会搜索 MONO_PATH 变量指定的路径（以冒号分隔的路径列表）；
      • 例如，在 Linux 的 bash shell 中设置：
      • export MONO_PATH=/path/to/assemblies:/another/path/to/assemblies
      1. 全局程序集缓冲 - GAC：
      • 若仍未找到，Mono 会搜索 GAC，这是一个存储共享程序集的中央仓库；
      • Mono 提供的所有程序集均存储在 GAC 中；
  • 概念：安装程序集到 GAC
# 1. 安装
gacutil -i MyLibrary.dll

# 2. 列出GAC内容
gacutil -l

# 3. 通过 -package 选项将程序集链接到编译器可访问的目录
gacutil -i MyLibrary.dll -package DEMO

# 4.1 多 GAC 支持
export MONO_GAC_PREFIX=~/.mono  # 指向自定义 GAC 目录
