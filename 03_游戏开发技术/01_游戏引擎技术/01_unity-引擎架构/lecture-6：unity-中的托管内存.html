值类型的装箱拆箱

概念：值类型的装箱拆箱

  • 概念：
    • 在 C# 和 .NET 中，​值类型​（如基本数据类型 int、float，结构体 struct）通常分配在线程栈上，它们存储的是实际的数据值。而引用类型​（如类、数组、接口）分配在托管堆上，变量本身存储的是指向堆内存地址的引用；
    • 而装箱是将值类型转换为引用类型的过程，拆箱则是将引用类型转换回值类型的过程。它们连接了值类型和引用类型，使得值类型能够被视为对象进行操作，但同时也带来了一定的性能开销；
  • 示例：
  • 概念：值类型的装箱
    • 当将一个值类型赋值给一个 object 类型的变量时，CLR 会自动进行装箱操作，具体步骤如下：
    • （1）在托管堆上分配内存​：CLR 计算值类型所有字段所需的内存总量，并额外添加托管堆对象必需的两个成员​，即类型对象指针和同步块索引所占用的空间；
    • （2）字段复制​：将栈上的值类型实例的所有字段逐位复制到新分配的堆内存中；
    • （3）返回地址​：返回新创建的托管堆对象的地址​（引用），该地址被存储到目标引用类型变量中；
  • 概念：引用类型的拆箱
    • 拆箱不是简单地逆转装箱过程，它主要包括获取指向已装箱对象中值类型数据部分的指针，并可选地将值复制回栈上的值类型变量中；
    • （1）类型检查​：CLR 首先检查进行拆箱操作的引用（对象）是否为 null，若是则抛出 NullReferenceException。接着，检查该引用是否指向与被拆箱目标值类型相匹配的已装箱对象。如果类型不匹配（例如，试图将装箱的 int 拆箱为 long），会抛出 InvalidCastException；
    • （2）获取地址​：类型验证通过后，CLR ​获取已装箱对象中值类型字段的起始地址。这一步是真正的“拆箱”，它只是一个指针操作，​不会复制任何数据，因此开销相对较小；
    • （3）字段复制​（通常紧接着发生）：在 C# 等语言中，拆箱操作通常后接一个将值从堆复制到栈上值类型实例的动作。例如 int j = (int)o; 会将已装箱 int 的值复制到栈变量 j 中；
  • 分析：装箱拆箱的性能影响
    • 每次装箱都在托管堆上创建一个新对象，增加了内存压力；
    • 频繁装箱产生的短期对象会增加垃圾回收器的负担，可能因此触发更频繁的 GC 操作（尤其是第 0 代回收），进而影响程序性能；
  • 分析：优化策略
    • 方式一：使用泛型集合
      • 这是避免大量装箱/拆箱最有效的方法，优先使用 List<T>, Dictionary<TKey, TValue> 等泛型集合，而不是 ArrayList，Hashtable 这样的非泛型集合；
    • 方式二：重写值类型的 ToString 方法
      • 在需要将值类型与字符串连接时，直接调用其 ToString 方法，比如 i.ToString()，而不是直接使用值类型本身（"" + i），后者会导致装箱；
    • 方式三：使用 ref 结构体
      • C# 7.2 引入了 ref struct，它们只能分配在栈上，无法被装箱，这可以在某些场景中彻底避免装箱的可能性；

托管堆对象的构成

概念：托管堆对象的构成

  • 概念：类型对象指针
    • 每个托管堆上的对象都包含一个指向其类型对象​（Type Object）的指针，这个类型对象本身也是一个在托管堆上分配的对象，它包含了关于该类型的元数据信息；
    • 主要包含以下内容：
      • （1）方法表 (Method Table)​​：包含该类型所有方法的地址（包括 虚方法表/vtable），这是实现多态的基石；
      • （2）静态字段​：该类型的所有静态字段都存储在类型对象上，而不是各个实例上；
      • （3）​类型元数据​：基类型信息、实现的接口、字段的布局和偏移量等；
      • （4）指向基类型的类型对象的指针；
    • 比如，当调用一个对象的方法（如 myObject.ToString()）时，CLR 会通过该对象的类型对象指针找到其类型对象，然后在方法表中找到 ToString 方法的实际地址并执行。这确保了虚方法调用能够正确分发到重写的方法上；
  • 概念：同步块索引
    • 同步块索引是一个指向同步块​（Sync Block）的整数索引，同步块并不直接存储在对象本身的内存中，而是由 CLR 维护在一个单独的表中；
    • 主要包含以下内容：
      • （1）线程同步锁​：当该对象被用作 lock 语句（或 Monitor.Enter/Exit）的参数时，同步块会记录哪个线程拥有了该锁以及锁的递归计数，这是实现 C# lock 关键字的核心机制；
      • （2）对象的哈希码​：如果你重写了 GetHashCode() 方法并显式调用了它，计算出的哈希值可能会被缓存到同步块中，以避免重复计算。如果未重写，系统也可能在需要时生成并存储一个；
      • （3）​应用程序域信息​：对于跨应用程序域访问的对象，会存储相关信息；
    • 最初，对象的同步块索引为 0，表示它未被使用。当你第一次 lock 这个对象时，CLR 会从同步块表中分配一个同步块，并将其索引值设置到该对象的同步块索引字段中。后续对该对象的同步操作，都会通过这个索引来查找对应的同步块；
  • 概念：实例数据
    • 这部分存储了对象定义的所有实例字段​（非静态字段），字段的存储顺序和布局由 CLR 优化决定，可能与你在代码中声明的顺序不同；
  • 概念：填充字节
    • CLR 可能会在对象的末尾添加一些额外的字节（Padding），以确保对象的总大小是字长（4字节或8字节）的倍数。这主要是为了内存对齐，优化 CPU 的访问效率；

Unity 内存类型

基础概念

概念：Unity 内存类型

  • 概念：
    • Unity 主要分为三类内存，分别是托管内存、C# 非托管内存和原生内存；
    • 托管内存由 .NET/Mono 运行时自动管理，主要对象包括 C# 对象，类实例、数组、字符串等，由垃圾回收器 (GC) 自动分代回收；
    • C# 非托管内存由开发者手动管理，使用 Unity.Collections 等分配的数据，需调用 Dispose() 等方法显式释放；
    • 原生内存 Native Memory 由 Unity 引擎的 C++ 内部管理，主要存储引擎资源，回收方式采用引用计数或显式销毁​；

三类内存

概念：三类内存

  • 概念：托管内存
    • 托管内存是由 .NET/Mono 运行时（如 Mono 或 IL2CPP）自动管理的内存区域，用于存储所有 C# 脚本中创建的引用类型对象，如类实例、数组、字符串；
    • GC 发生时通常会暂停主线程​（Stop-the-World），经历标记​（从根对象出发标记所有可达对象）、清除​（回收未被标记的垃圾对象内存）和压缩​（可选，整理内存碎片）三个阶段；
    • 在 Update() 中频繁创建临时字符串或数组：string debugText = "Score: " + score.ToString(); 这种方式不推荐，会产生 GC Alloc；
  • 概念：C# 非托管内存
    • 这部分内存虽然由 C# 代码分配，但不受 .NET 垃圾回收器 (GC) 管理，需要使用特定的 API（如 Unity.Collections 命名空间）来手动分配和释放它；
    • 通过调用如 UnsafeUtility.Malloc()、Marshal.AllocHGlobal() 或 NativeArray<T> 的构造函数来直接向操作系统申请内存，必须在使用完毕后调用相应的 Dispose() 方法或 Free() 函数来显式释放内存，否则会造成内存泄漏；
    • C# 非托管内存常用于需要与 Unity 底层 Job System 和 Burst Compiler 配合的高性能计算场景：
      • 分配一个非托管的 NativeArray
        • NativeArray<float> data = new NativeArray<float>(100,Allocator.Persistent);
      • 释放：
        • data.Dispose();
  • 概念：原生内存 Native Memory
    • 这是由 ​Unity 引擎自身（C++ 层）​​管理的内存，用于存储引擎运行所需的绝大多数核心资源；
    • Unity 引擎主要通过引用计数 (Reference Counting)​​ 来管理这些资源，当一个资源被加载或被游戏对象引用时，其引用计数增加；当不再被引用时，计数减少。当计数归零时，资源被标记为可回收；
    • 在引擎中加载得到的纹理 (Textures)、网格 (Meshes)、音频剪辑 (AudioClips)、动画片段 (AnimationClips)、着色器 (Shaders) 以及所有 UnityEngine.Object 派生对象的底层 C++ 实现都存储在这里，比如：
      • Texture2D tex = Resources.Load<Texture2D>("MyTexture");// 加载纹理到原生内存
      • GameObject enemy = Instantiate(enemyPrefab);// 实例化的对象数据在原生内存
      • AudioSource.PlayClipAtPoint(mySoundClip, position);// 播放的音频数据在原生内存

Unity 中的托管内存

托管内存基础概念

概念：托管内存基础概念

  • 概念：
    • Mono 和 IL2CPP 的脚本虚拟机 (VM) 实现 托管内存 系统，有时被称为脚本内存系统。这些 VM 提供了一个受控的内存环境，分为以下不同类型：
      1. 托管堆：VM 使用垃圾收集器 (GC) 自动控制的内存部分。因此，分配在托管堆上的内存被称为 GC 分配。性能分析器记录了这种分配的任何出现，作为 GC.Alloc 样本；
      1. 脚本堆栈：这是在应用程序进入和退出任何代码范围时建立和展开的；
      1. 本机 VM 内存：包含与 Unity 的脚本层相关的内存。大多数情况下，开发者无需操作本机 VM 内存，但了解它包含与代码生成的执行代码相关的内存很有用，特别是围绕泛型、反射使用的类型元数据以及运行 VM 所需的内存；
    • 由于托管内存系统使用 VM，因此它具有一个受控的环境，该环境自动跟踪分配的引用以管理其生命周期。这意味着应用程序不太可能过早地释放内存，而其他代码正在尝试访问它；
  • 特点：
    • 优点：
      • （1）自动内存管理，开发者无需手动分配和释放内存，降低了开发门槛；
      • （2）通过 GC 自动回收无引用对象，避免了野指针和重复释放等内存错误；
    • 缺点：
      • （1）GC 操作（尤其是堆内存较大时）会暂停主线程，引起帧率下降和游戏卡顿，影响体验；
      • （2）频繁分配和释放不同大小的对象会导致内存碎片化，可能产生足够的总空闲内存却无法分配大对象的情况，从而触发不必要的堆扩展或 GC；
      • （3）垃圾收集器使用起来很方便，但它在释放和分配内存的方式上也很不可预测，这可能会导致性能问题；

Mono 和 IL2CPP 中的托管堆

概念：Mono 和 IL2CPP 中的托管堆

  • 概念：
    • Mono ​ 是一个完整的、独立的跨平台 .NET 实现，它包含自己的虚拟机、即时编译器​​ 和垃圾回收器。在 Unity 中使用 Mono 时，C# 代码首先被编译成中间语言（IL）​。在运行时，Mono VM 会加载这些 IL 代码，并通过 JIT 编译器将其即时编译为目标平台的原生代码执行；Mono 的托管堆由其自身的内存管理器进行管理；
    • IL2CPP ​则是一个提前（AOT）编译的解决方案，它的工作流程是：首先将 C# 代码编译成 IL，然后通过 ​ il2cpp.exe ​ 工具将 IL 代码整体转换为标准的 ​ C++ 代码，最后使用目标平台的原生编译器将 C++ 代码编译为原生机器码；为了保持 C# 的语义（如垃圾回收、异常处理、虚方法调用等），IL2CPP 包含一个用 C++ 编写的轻量级运行时库（libil2cpp），这个运行时负责管理托管堆、提供垃圾回收等服务；

内存碎片处理机制

概念：内存碎片处理机制

  • 引入：内存碎片的产生
    • 对象在托管堆上分配时，需要占用一块连续的内存空间​，当对象被垃圾回收器回收后，它所占用的内存空间被释放，​但该空间的位置和大小保持不变。如果新对象的大小小于或等于某个空闲碎片，则可以放入该碎片。如果新对象的大小大于所有现有的空闲碎片，即使所有碎片的总和远大于该对象所需，分配也会失败，因为找不到一块足够大的连续空间；
  • 概念：
    • 由于 GC 是非压缩的，两者都无法主动解决已经产生的内存碎片，它们的处理方式本质上是被动的；
    • （1）在分配失败时触发 GC，希望这次回收能创造出足够的连续空间；
    • （2）如果 GC 后仍无法分配，则扩展堆​（通常是将当前堆大小加倍）获取一大块新的连续空间；
    • （3）对于长期运行的游戏（如服务器、持续在线游戏），碎片化会逐渐累积，可能最终导致大对象分配失败。在这种情况下，最彻底的解决方案是重启游戏进程，或者精心设计内存分配策略以尽量减少碎片的产生；

托管堆的扩容机制

概念：托管堆的扩容机制

  • 概念：
    • 当托管堆中的空闲内存（包括碎片）无法满足当前内存分配请求时，就会触发堆扩展；
  • 过程：
    • （1）分配请求到来，内存管理器在当前堆中找不到足够大的连续空闲空间；
    • （2）触发一次垃圾回收（GC），尝试释放内存；
    • （3）GC 完成后再次尝试分配，如果仍然失败，则向操作系统申请更大的内存空间；
    • （4）在大多数平台上，Unity 的托管堆会采用加倍策略进行扩展（例如，从 64MB 扩展到 128MB）
    • （5）新分配的内存块会附加到现有托管堆的末尾，提供新的连续空间用于分配；
  • 注意：扩容的注意事项
    • （1）Unity 不会定期释放分配给托管堆的内存；相反，它会保留扩展的堆，即使其中很大一部分是空的。这样做是为了防止在进一步发生大型分配时需要重新扩展堆；
    • （2）在大多数平台上，Unity 最终会将托管堆空闲部分使用的内存释放回操作系统。发生这种情况的间隔没有保证且不可靠；
    • （3）垃圾收集器不会清除原生内存对象或其他本机分配；
    • （4）​警惕非主线程分配​：在其他线程分配的内存不会被 Unity Profiler 的 "GC Alloc" 列追踪，需额外注意；

垃圾回收实现

垃圾回收器概述

概念：垃圾回收器概述

  • 概念：
    • Unity 使用垃圾回收器来回收应用程序和 Unity 不再使用的对象的内存，当脚本尝试在托管堆上进行分配，但没有足够的空闲堆内存来容纳该分配时，Unity 会运行垃圾回收器。垃圾回收器运行时，它会检查堆中的所有对象，并标记应用程序不再引用的任何对象以供删除。然后，Unity 删除未引用的对象，从而释放内存；
    • 垃圾回收器以相同的方式处理后续请求，直到没有足够大的空闲区域来分配所需的块大小。在这种情况下，不太可能所有分配的内存仍在使用。Unity 的脚本后端只能访问堆上的引用项，只要仍然存在可以定位它的引用变量。如果所有指向内存块的引用都丢失，则垃圾回收器可以重新分配它占用的内存；
    • 为了确定哪些堆块不再使用，垃圾回收器会搜索所有活动引用变量，并将它们引用的内存块标记为“活动”。搜索结束时，垃圾回收器会将“活动”块之间的任何空间视为为空，并将其标记为供后续分配使用。查找和释放未使用内存的过程称为垃圾回收（GC）
  • 分类：
    • 在 Unity 中，垃圾回收器具有以下模式：
    • （1）增量垃圾回收：默认情况下启用（项目设置 > 播放器 > 配置），此模式将垃圾回收过程分散到多个帧中；
    • （2）禁用增量垃圾回收：如果禁用增量 GC 播放器设置，则垃圾回收器会停止运行应用程序以检查和处理堆上的对象；
    • （3）禁用自动垃圾回收：使用 GarbageCollector.GCMode API 完全控制 Unity 何时运行垃圾回收器；
  • 设置：
    • Edit > Project Settings > Player > Other Settings > Configuration > Use Incremental GC

Boehm GC 算法

概念：Boehm GC 算法

  • 概念：
    • Unity 历史上长期使用 ​ Boehm-Demers-Weiser Garbage Collector (Boehm GC)​​ 作为其主要的托管内存管理器，这是一个保守式（Conservative）、非移动式（Non-Moving）的标记-清除（Mark-Sweep）​​垃圾回收器。近年来，Unity 也在向更先进的 ​.NET CoreCLR / Unity Burst GC ​（精确式、分代式、压缩式）迁移；
  • 区别：
  • 图示：算法处理流程
  • 概念：算法核心流程
    • Boehm GC 的核心工作流程基于 ​标记-清除算法，并包含两个关键特性：​保守式和 ​非移动式；
    • （1）触发时机：
        1. 分配请求失败时；
        1. 手动调用 System.GC.Collect()；
        1. GC 根据自身策略（如自上次回收后的分配总量、时间间隔等）决定触发；
    • （2）暂停程序：
      • 这是 Boehm GC 引起卡顿（GC Spike）​​ 的主要原因。一旦触发，GC 会暂停所有托管线程；
    • （3）准备阶段：
      • 在此阶段，GC 会重置所有对象的标记位（MarkBit）​，为接下来的标记阶段做准备；
    • （4）标记阶段：
        1. GC 会从所有根对象（Roots）​​，比如静态变量、当前执行栈上的局部变量、寄存器等出发
        • 常见的 GC 根包括当前线程的栈，其中存储了局部变量和方法参数等。还包括当前 CPU 寄存器当中的值，以及全局变量或静态变量；
        1. 保守式扫描：Boehm GC ​不依赖精确的类型信息。它会将上述区域中的每一个字长（32 位系统为 4 字节，64 位为 8 字节）的数据都当作一个潜在的指针来检查，检查条件包括以下：
        • 该值是否落在 GC 管理的堆地址范围内；
        • 该值是否指向某个已知分配对象的起始地址​；
        • 如果满足条件，GC 就“保守地”认为这是一个有效指针，并将它指向的对象标记为可达；
        1. 递归标记：从根直接指向的对象开始，GC 会递归地遍历这些对象所引用的所有其他对象，并将它们也标记为“可达”。这个过程持续下去，直到所有从根出发可达的对象都被标记；
    • （5）清除阶段：
      • 标记阶段完成后，堆中的所有对象都已被分类为“可达”或“不可达”；
      • GC 线性遍历整个托管堆，对于被标记为“可达”的对象，​清除其标记位，为下一次 GC 做准备，该对象存活。对于未被标记的对象，则认为它们是垃圾，将其占用的内存块回收；
      • 回收的内存会被加入到相应的“空闲列表（Free List）​”中，以供后续分配使用；
    • （6）终结器处理 Finalization Phase：
      • 这是一个特殊且相对独立的阶段，用于处理拥有终结器（Finalizer）​​ 的对象，例如在 C# 中重写了 Finalize 方法的对象；
      • 所有注册了终结器的对象会被存储在一个单独的 FinalizableList 中，在标记阶段结束后，GC 会检查这个列表。将所有未被标记​（即不可达）但含有终结器的对象，​重新推入标记栈，并默认它们“复活”​，然后以它们为根重新进行一轮标记遍历。这是为了确保在执行终结器的过程中，该对象所依赖的其他对象不会被意外回收；
      • 在这轮特殊的标记结束后，仍然未被标记的终结器对象会被移入一个等待执行终结器的队列 F-Queue，并在一个单独的线程中执行其终结器；
      • 执行完终结器的对象会在下一次 GC时，如果确认不可达，才会被真正回收；
    • （7）恢复阶段：
      • 当所有阶段完成后，GC 会恢复所有被暂停的托管线程，程序继续执行‘’
  • 总结：
      1. 带有终结器的对象至少需要经历两次 GC 才能被完全释放，应避免不必要的使用；
      1. Boehm GC ​为保守式的扫描，无需精确的运行时类型信息，简化了 Unity 的 C++ 引擎与 C # 脚本之间的集成，但可能因为非指针数据恰好看起来像指针​（如一个整数等于某个已死亡对象的地址），导致该死亡对象无法被回收，进而导致保守性内存泄漏；
      1. 传统的 Boehm GC 算法会在标记和清理阶段会暂停主线程，而 Unity 当中使用的增量式 GC 通过将标记工作分到多帧执行，显著缓解了此问题；

Unity 中的增量式垃圾回收

概念：增量式垃圾回收

  • 概念：
    • 增量式垃圾回收 (GC) 将垃圾回收过程分散到多个帧中，这是 Unity 中默认的垃圾回收行为；
    • Unity 的垃圾回收器使用 Boehm–Demers–Weiser 垃圾回收器。默认情况下，Unity 在增量模式下使用它，这意味着垃圾回收器将其工作量分散到多个帧中，而不是停止主 CPU 线程来处理托管堆上的所有对象，这意味着 Unity 对应用程序的执行进行了较短的中断，而不是长时间中断以让垃圾回收器处理托管堆上的对象；
    • 增量模式不会使垃圾回收总体上更快，但由于它将工作量分散到多个帧中，因此减少了与 GC 相关的性能峰值。这些中断被称为 GC 峰值，因为它们在性能分析器窗口的帧时间图中显示为较大的峰值；
    • 如果禁用增量模式，则垃圾回收器在执行收集过程时必须检查整个堆。这被称为 停止世界 垃圾回收，因为每当垃圾回收器运行时，它都会停止主 CPU 线程。它只会在处理完托管堆上的所有对象后恢复执行，这可能会导致 GC 峰值影响应用程序的性能；
    • 垃圾回收器也是非压缩的，这意味着 Unity 不会在内存中重新分配任何对象以关闭对象之间的间隙；
    • 如果程序使用 VSync 或 Application.targetFrameRate，Unity 会根据剩余的可用帧时间调整分配给垃圾回收的时间。这样，Unity 可以在原本花费在等待的时间内运行垃圾回收，并以最小的性能影响执行垃圾回收；
  • 图示：禁用增量式 GC 的性能分析会话
    • 禁用增量式 GC，并且可以看到一个清晰的 GC 峰值。此峰值会中断原本平滑的 60fps 帧速率，并将垃圾回收发生的帧推迟到超过保持 60fps 所需的 16 毫秒限制；
  • 图示：启用增量式 GC 的性能分析会话
    • 启用之后，应用程序具有稳定的 60fps 帧速率，因为垃圾回收器将垃圾回收操作分散到多个帧中，并使用每个帧的一小段时间片 (黄色 VSync 轨迹正上方的深绿色边缘)；

分析：在 Unity 中精确控制增量式垃圾回收行为

  • 分析：
    • 在 Unity 精确控制增量式垃圾回收的行为，主要依赖于 Scripting.GarbageCollector 类提供的 API；通过它可以手动触发回收步骤并精细调整时间片，从而更好地管理 GC 对游戏帧率的影响；
  • 概念：incrementalTimeSliceNanoseconds
    • 类型为静态的 ulong 成员，默认为 3ms，设置增量 GC 每个帧步骤的目标持续时间​（以纳秒为单位）；
  • 概念：CollectIncremental(ulong nanoseconds)
    • 手动触发一个增量 GC 步骤，并尝试在指定的纳秒数内执行回收工作。返回 true 表示还有更多工作要做；
  • 示例：计算本帧已用时间和剩余时间，在本帧的剩余时间内手动执行增量 GC 
using UnityEngine;
using System.Diagnostics; // 使用Stopwatch获取高精度时间

public class IncrementalGCController : MonoBehaviour
{
    public float targetFrameTime = 1f / 60f; // 例如：60FPS的目标帧时间
    private Stopwatch frameTimer = new Stopwatch();
    private float frameStartTime;

    void Start()
    {
        frameTimer.Start();
    }

    void Update()
    {
        // 记录帧开始时间
        frameStartTime = (float)frameTimer.Elapsed.TotalSeconds;

        // ... 你的游戏逻辑 ...

        // 在帧结束前，计算已用时间和本帧的剩余时间
        float currentFrameTime = (float)frameTimer.Elapsed.TotalSeconds - frameStartTime;
        float timeLeftInFrame = targetFrameTime - currentFrameTime;

        // 将剩余时间转换为纳秒，并预留一部分安全余量（例如0.5ms）
        ulong nsAvailable = (ulong)((timeLeftInFrame - 0.0005f) * 1e9f);

        // 确保时间片为正且合理
        if (nsAvailable > 1000) // 至少1微秒
        {
            // 手动执行增量GC，使用本帧剩余时间
            bool workRemaining = Scripting.GarbageCollector.CollectIncremental(nsAvailable);
            // 如果workRemaining为true，表示GC工作尚未完成，下一帧继续
        }

        frameTimer.Restart();
    }
}
  • 分析：
    • 在帧末尾等空闲时段，使用 CollectIncremental() ​手动触发 GC 步骤，可以更精确地利用空闲 CPU 时间；
