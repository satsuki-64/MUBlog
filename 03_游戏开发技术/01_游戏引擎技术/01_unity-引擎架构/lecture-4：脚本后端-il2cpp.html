Lecture 4：脚本后端 - IL2CPP

什么是 IL2CPP

基础概念

概念：IL2CPP

  • 概念：
    • IL2CPP 表示中间语言到 C++ 的脚本后端，是 Mono 后端的一种替代方案；
    • IL2CPP 为更广泛的平台上的应用程序提供了更好的支持，IL2CPP 后端将 MSIL（Microsoft 中间语言）代码（例如，脚本中的 C# 代码）转换为 C++ 代码，然后使用 C++ 代码为所选平台创建本地二进制文件；
    • 这种编译类型称为提前（AOT）编译，其中 Unity 在构建本地二进制文件时针对目标平台编译代码。Mono 后端使用称为即时编译（JIT）的技术在运行时编译代码；
  • 特点：
    • 解决 JIT 限制​：
      • iOS 等平台禁止动态代码生成（JIT），IL2CPP 通过 AOT 编译生成平台原生二进制文件；
    • 性能优化​：
      • C++ 代码经本地编译器（Clang/MSVC/GCC）优化后，执行效率较 Mono 提升；
    • ​代码保护​：
      • C++ 二进制比 IL 代码更难反编译，增强安全性；
  • 总结：Mono 和 IL2CPP 区别

IL2CPP 实现原理

概念：IL2CPP 实现原理

  • 阶段一：Roslyn 编译 - C# 到 .NET DLL 的转换
    • C# 代码通过 ​ Roslyn 编译器​（Unity 2017+ 默认）转换为 .NET 中间语言（IL）​，进而生成托管程序集（如 Assembly-CSharp.dll）
    • 编译过程在编辑器中实时触发，输出文件存储在 Library/ScriptAssemblies 目录下。 Library/ScriptAssemblies 目录存储编译生成的 ​托管程序集（.dll 文件）​
  • 阶段二：托管代码剥离
    • Unity 分析 IL 代码，移除未使用的类、方法、字段（如未被场景引用或反射标记的代码），具体可看正文下方的托管代码剥离介绍；
  • 阶段三：IL 到 C++ 的代码生成
    • IL2CPP 后端将所有托管程序集转换为标准 C++ 代码，具体而言分为以下几个步骤：
    • （1）IL 解析：读取 DLL 中的 IL 指令与元数据，构建内存中的类型系统模型；
      • IL2CPP 读取 C# 编译后的 IL 代码（.dll 文件），解析其中的元数据（类、方法、字段等）和指令流。元数据包括类型继承关系、方法签名、泛型约束等关键信息；
    • （2）C++ 代码生成：进行类型映射、方法转换、泛型处理等步骤
      • 每个方法转换为独立的 C++ 函数，方法名按规则编码，参数和返回值类型直接映射；
      • 对于虚方法，需要生成虚方法表（vtable），存储函数指针，支持多态调用。子类 vtable 会覆盖父类条目；
  • 阶段四：C++ 编译器使用原生平台编译器编译生成的 C++ 代码和 IL2CPP 的运行时部分
  • 阶段五：Unity 创建可执行文件或 DLL

IL2CPP 启用运行时检查

基础概念

概念：IL2CPP 启用运行时检查

  • 概念：
    • 使用 IL2CPP 脚本后端时，可以控制 il2cpp.exe 生成 C++ 代码的方式，可以使用 Il2CppSetOption 属性启用或禁用以下运行时检查：
    • （1）空值检查：如果启用此选项，则 IL2CPP 生成的 C++ 代码将包含空值检查，并根据需要抛出托管 NullReferenceException 异常。如果禁用此选项，则 IL2CPP 不会将空值检查发出到生成的 C++ 代码中；
      • 默认启用；
    • （2）数组边界检查：如果启用此选项，则 IL2CPP 生成的 C++ 代码将包含数组边界检查，并根据需要抛出托管 IndexOutOfRangeException 异常。如果禁用此选项，则 IL2CPP 不会将数组边界检查发出到生成的 C++ 代码中；
      • 默认启用；
    • （3）除以零检查：如果启用此选项，则 IL2CPP 生成的 C++ 代码将包含针对整数除法的除以零检查，并根据需要抛出托管 DivideByZeroException 异常。如果禁用此选项，则 IL2CPP 不会将整数除法的除以零检查发出到生成的 C++ 代码中
      • 默认禁用；
  • 方法：如何使用 IL2CPP 启用运行时检查
    • 第一步：在安装 Unity 版本的目录中，导航到 Windows 上的 Data\il2cpp 目录，或 OS X 上的 Contents/Frameworks/il2cpp 目录；
    • 第二步：查找 Il2CppSetOptionAttribute.cs 源文件
    • 第三步：将源文件复制到项目的 Assets 目录中

Il2CppSetOption 属性实现及其使用

代码：Il2CppSetOption 属性实现及其使用

  • 代码：Il2CppSetOptionAttribute.cs 
using System;

namespace Unity.IL2CPP.CompilerServices
{
    /// <summary>
    /// IL 到 C++ 转换过程中的代码生成选项。
    /// 请谨慎启用或禁用这些选项。
    /// </summary>
    public enum Option
    {
        /// <summary>
        /// 启用或禁用空引用检查的代码生成。
        ///
        /// 当 il2cpp.exe 从 Unity 编辑器启动时，默认启用全局空引用检查支持。
        ///
        /// 禁用此选项将阻止在生成的代码中抛出 NullReferenceException 异常。
        /// 在大多数情况下，解引用空指针的代码会直接崩溃。
        /// 有时崩溃发生的位置会比原本应该抛出异常的位置更靠后。
        /// </summary>
        NullChecks = 1,
        
        /// <summary>
        /// 启用或禁用数组越界检查的代码生成。
        ///
        /// 当 il2cpp.exe 从 Unity 编辑器启动时，默认启用全局数组越界检查支持。
        ///
        /// 禁用此选项将阻止在生成的代码中抛出 IndexOutOfRangeException 异常。
        /// 这将允许读写数组边界之外的内存而不进行任何运行时检查。
        /// 请极其谨慎地禁用此检查。
        /// </summary>
        ArrayBoundsChecks = 2,
        
        /// <summary>
        /// 启用或禁用除零检查的代码生成。
        ///
        /// 当 il2cpp.exe 从 Unity 编辑器启动时，默认禁用全局除零检查支持。
        ///
        /// 启用此选项将在生成的代码中抛出 DivideByZeroException 异常。
        /// 大多数代码不需要处理此异常，因此保持禁用状态通常是安全的。
        /// </summary>
        DivideByZeroChecks = 3,
    }

    /// <summary>
    /// 将此特性应用于程序集、结构体、类、方法或属性，以指示 IL2CPP 代码转换工具覆盖某些运行时检查的全局设置。
    ///
    /// 示例：
    ///
    ///     [Il2CppSetOption(Option.NullChecks, false)]
    ///     public static string 禁用空检查的方法()
    ///     {
    ///         var tmp = new Object();
    ///         return tmp.ToString();
    ///     }
    /// </summary>
    [AttributeUsage(AttributeTargets.Assembly | AttributeTargets.Struct | AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property | AttributeTargets.Delegate, Inherited = false, AllowMultiple = true)]
    public class Il2CppSetOptionAttribute : Attribute
    {
        public Option Option { get; private set; }
        public object Value { get; private set; }

        public Il2CppSetOptionAttribute(Option option, object value)
        {
            Option = option;
            Value = value;
        }
    }
}
  • 示例：使用 Il2CppSetOption 属性
// 禁用整个类的空引用检查（适用于性能关键且能确保非空的代码）
[Il2CppSetOption(Option.NullChecks, false)]
public class TypeWithNullChecksDisabled
{
    public static string AnyMethod()
    {
        // 此方法内所有对象访问不会生成空检查代码
        // 若tmp为null，将直接解引用导致崩溃（而非抛出NullReferenceException）
        var tmp = new object();
        return tmp.ToString();
    }

    // 单独启用此方法的空引用检查（覆盖类级别的设置）
    [Il2CppSetOption(Option.NullChecks, true)]
    public static string MethodWithNullChecksEnabled()
    {
        // 此方法会生成空检查代码，若tmp为null会抛出NullReferenceException
        var tmp = new object();
        return tmp.ToString();
    }
}

public class SomeType
{
    // 禁用整个属性的空检查（包括getter和setter）
    [Il2CppSetOption(Option.NullChecks, false)]
    public string PropertyWithNullChecksDisabled
    {
        get
        {
            // 无空检查，直接解引用
            var tmp = new object();
            return tmp.ToString();
        }
        set
        {
            // 无空检查，若value为null会崩溃
            value.ToString();
        }
    }

    public string PropertyWithNullChecksDisabledOnGetterOnly
    {
        // 仅禁用getter的空检查（setter保持默认或类级设置）
        [Il2CppSetOption(Option.NullChecks, false)]
        get
        {
            // 无空检查
            var tmp = new object();
            return tmp.ToString();
        }
        set
        {
            // 此处仍会执行空检查（因未显式禁用）
            value.ToString();
        }
    }
}

托管代码剥离

基础概念

概念：托管代码剥离

  • 概念：什么是托管代码剥离
    • Unity 通过称为托管代码剥离的过程删除未使用的或无法访问的代码，这可以显著减小应用程序的最终大小;
    • 托管代码剥离会从托管程序集中删除代码，包括从 C# 脚本构建的程序集、包和插件的一部分的程序集以及 .NET Framework 中的程序集；
    • Unity 使用一个称为 Unity 链接器 的工具来执行项目程序集代码的静态分析，静态分析识别在执行期间无法访问的任何类、类的部分、函数或函数的部分。此分析仅包括构建时存在的代码，因为 Unity 执行静态分析时，运行时生成的代码不存在；
  • 概念：设置代码剥离级别
    • 有两种方法可以用于设置代码的剥离级别：
    • 方法一：在项目中进行配置
      • 在 Edit > Project Setting 中找到以下选项
    • 方法二：使用代码注释来指示 Unity 链接器应保留的代码库的哪些部分
  • 概念：Managed Stripping Level
    • Managed Stripping Level 属性用于确定 Unity 链接器在分析和剥离应用程序代码时遵循的规则集；
    • 当将设置从 最小 提高到 高 时，这些规则使链接器能够搜索更多程序集以查找无法访问的代码。Unity 链接器在较高设置下删除更多代码，从而减小最终构建的大小，尽管扩展搜索意味着每个构建需要更长的时间才能生成；

使用注释保留代码

概念：使用注释保留代码

  • 概念：
    • 通过代码注释（如 [Preserve] 特性）标记关键代码，可以用于阻止 Unity 链接器在静态分析阶段将其误判为未使用代码而剥离；
    • 此机制特别适用于 ​动态生成的运行时代码​，比如反射、序列化、依赖注入等场景，因为这些代码的调用关系无法通过静态分析识别；
  • 分类：两种主要方法来注释代码以将其保留在托管代码剥离过程中
    • 方法一：根注释
      • 根注释将代码的某些部分标识为根，Unity 链接器不会剥离标记为根的任何代码；
      • 根注释使用起来不太复杂，但也可能导致 Unity 链接器保留一些它应该剥离的代码；
    • 方法二：依赖项注释
      • 依赖项注释定义代码元素之间的连接，与根注释相比，依赖项注释可以减少代码过度保留的数量；

方法：根注释 - [Preserve]

  • 概念：
    • 使用 Preserve 属性单独排除代码的特定部分，以使其不受 Unity 链接器的静态分析；
    • 要使用此属性注释一段代码，在要保留的代码的第一部分之前立即添加 [Preserve]；
  • 分类：使用 [Preserve] 时 Unity 链接器会保留的实体
  • 示例：
using UnityEngine.Scripting;

[Preserve]
public class DynamicLoader 
{
    public void LoadType() 
    {
        // 动态反射调用示例
        Type type = Type.GetType("MyGame.ReflectedClass");
    }
}
注意：关于序列化
  • 如果使用 Unity 内置序列化（如 [Serializable]类），且所有字段都在编译时可见，通常不需要 [Preserve]；

方法：根注释 - Link XML 文件注释根

  • 概念：
    • 可以在项目中包含一个名为 link.xml 的 .xml 文件以保留特定程序集或程序集部分的列表；
    • link.xml 文件必须位于项目的 Assets 文件夹或 Assets 文件夹的子目录中，并且必须在文件中包含 <linker> 标记。Unity 链接器将 link.xml 文件中保留的任何程序集、类型或成员视为根类型；
    • 可以在项目中使用任意数量的 link.xml 文件，可以为每个插件提供单独的保留声明；
  • 示例：
<linker>
  <!--Preserve types and members in an assembly-->
  <assembly fullname="AssemblyName">
    <!--Preserve an entire type-->
    <type fullname="AssemblyName.MethodName" preserve="all"/>

    <!--No "preserve" attribute and no members specified means preserve all members-->
    <type fullname="AssemblyName.MethodName"/>

    <!--Preserve all fields on a type-->
    <type fullname="AssemblyName.MethodName" preserve="fields"/>

    <!--Preserve all fields on a type-->
    <type fullname="AssemblyName.MethodName" preserve="methods"/>

    <!--Preserve the type only-->
    <type fullname="AssemblyName.MethodName" preserve="nothing"/>

    <!--Preserving only specific members of a type-->
    <type fullname="AssemblyName.MethodName">
        
      <!--Fields-->
      <field signature="System.Int32 FieldName" />

      <!--Preserve a field by name rather than signature-->
      <field name="FieldName" />
      
      <!--Methods-->
      <method signature="System.Void MethodName()" />

      <!--Preserve a method with parameters-->
      <method signature="System.Void MethodName(System.Int32,System.String)" />

      <!--Preserve a method by name rather than signature-->
      <method name="MethodName" />

      <!--Properties-->

      <!--Preserve a property, its backing field (if present), 
          getter, and setter methods-->
      <property signature="System.Int32 PropertyName" />

      <property signature="System.Int32 PropertyName" accessors="all" />

      <!--Preserve a property, its backing field (if present), and getter method-->
      <property signature="System.Int32 PropertyName" accessors="get" />

      <!--Preserve a property, its backing field (if present), and setter method-->
      <property signature="System.Int32 PropertyName" accessors="set" />

      <!--Preserve a property by name rather than signature-->
      <property name="PropertyName" />

      <!--Events-->

      <!--Preserve an event, its backing field (if present), add, and remove methods-->
      <event signature="System.EventHandler EventName" />

      <!--Preserve an event by name rather than signature-->
      <event name="EventName" />

    </type>
  </assembly>
</linker>

Unity 链接器

概念：Unity 链接器

  • 概念：
    • Unity 构建过程使用一个名为 Unity 链接器 的工具来剥离托管代码；
    • Unity 链接器是 IL 链接器的一个定制版本，专门用于与 Unity 协同工作，Unity 链接器中 Unity 引擎特有的定制部分不公开提供；
    • Unity 链接器负责托管代码剥离和引擎代码剥离过程的一部分，后者是一个独立的过程，可通过 IL2CPP 脚本后端使用，该后端会删除未使用的引擎代码；
  • 概念：Unity 链接器的工作原理
    • Unity 链接器会分析项目中的所有程序集；
    • 第一步：标记根类型、方法、属性和字段；
      • Unity 链接器会扫描项目中所有托管程序集，包括用户代码​，比如 Assembly-CSharp.dll，Unity 引擎模块​，如 UnityEngine.CoreModule.dll。以及​第三方库​，通过 Package Manager 或插件引入的程序集，如 Newtonsoft.Json.dll；
    • 第二步：Unity 链接器会分析它标记的根，识别并标记这些根所依赖的任何托管代码；
      • 根类型是代码保留的起点，标记规则包括：
      • （1）场景中的 MonoBehaviour ​：所有挂载到场景 GameObject 的脚本类及其公开字段/方法；
      • （2）Resources 文件夹引用​：通过 Resources.Load 加载的脚本或资产关联的类型；
      • （3）​特殊标记​：通过 [Preserve] 特性或 link.xml 显式声明的类型/成员；
    • 第三步：删除
      • 完成第一步和第二步的静态分析后，任何剩余的未标记代码都无法通过应用程序代码的任何执行路径访问，Unity 链接器会将其从程序集中删除；
      • 离级别（Managed Stripping Level）越高，删除的代码越多，但误删风险也越高；
  • 补充：public static bool stripEngineCode
    • 从项目构建中移除未使用的引擎代码（仅限 IL2CPP）；
    • 如果启用此选项，则在 IL2CPP 构建中将移除 Unity 引擎代码库中未使用的模块和类，这将导致二进制文件大小更小；
    • 建议使用此设置，但是如果你怀疑这会导致项目出现问题，则可能需要禁用它；
    • 注意：对于 IL2CPP 脚本后端，始终启用托管程序集的字节码剥离；

概念：依赖项标记规则
