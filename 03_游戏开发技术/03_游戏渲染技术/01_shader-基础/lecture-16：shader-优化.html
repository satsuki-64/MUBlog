Lecture 16：Shader 优化

通用优化规则

概念：Unity 的 Shader 当中的通用优化规则

  • 概念：精度优化
    • Shader 当中有三种精度，分别为 fixed/half/float；
    • （1）位置坐标、纹理坐标类使用 float；
      • Float (32 bit)（位置与纹理坐标信息）
    • （2）HDR 颜色、方向向量类使用 half；
      • Half (16 bit)(纹理坐标与颜色信息）
    • （3）普通纹理、颜色类使用 fixed；
      • Fixed (11 bit)(颜色信息） (SPR 下已不支持)
    • 实际上，使用的精度取决于目标平台和 GPU；
    • 比如，现在桌面级 GPU 都是直接采用 float，shader 中的 fixed/half/float 最后都是用的 float；
    • 而现代移动端 GPU 大多仅支持 half 与 float，所以能用 half 的就用 half。fixed 仅用于较旧的移动版本的 GPU；
  • 概念：能放顶点就不要放片段
    • 能将计算量放到顶点着色器当中，就尽量不要放到片段着色器当中；
    • 如果遇到了在顶点着色器中、插值效果不好的情况，可以考虑将顶点的数量增多；
  • 概念：少用多个 Pass
    • 一个 SubShader 内如果含有多个 Pass (不包含 Unity 中特殊作用的)，则会渲染多次，同时将不能进行合批；
    • 一般情况就是一个 Pass；
  • 概念：小心使用 AlphaTest 和 ColorMask
    • AlphaTest：也就是在 shader 中使用了 clip() 函数，通常在大多数平台上使用 AlphaTest 会有些性能优势，但是在 IOS 和某些使用 PowerVRGPU 的 Android 设备上性能很低；
    • ColorMash：在当前只输出某些通道。在 IOS 和部分 Android 设备上同样性能很低；
  • 概念：NoScaleOffset
    • 在定义贴图的时候，可以定义此符号：[NoScaleOffset]_MainTex("MainTex",2D)="white"{}
    • 在不需要调节 Tiling 和 Offset 的贴图属性中加入 [NoScaleOffset]，同时 Shader 不做相关计算；
  • 概念：DisableBatching - 禁用批处理
    • Tags{"DisableBatching"="true"}
    • true 表示不进行合批，false 表示能合批就合批，默认值；
    • 如果顶点上的计算需要在模型的本地空间下进行，则需要开启禁止合批（因为合批会将很多个模型一起放置在同一个本地空间下，因此此时原本的本地空间计算可能会有问题），否则其他情况下最好不要开启禁止合批；
    • 如图是动态合批，两个模型动态合批到了一起；
  • 概念：关于 GrabPass
    • GrabPass 是抓取当前缓冲区中的数据，并将其存储到贴图；
    • 其有两种方式，分别是默认的 GrabPass{}，其 Grabpass 如果不指定贴图的名称，则每个对象的 Grabpass 都会每帧生成一次；
    • Grabpass 如果指定贴图名称，则所有对象的 Grabpass 在一帧内只会生成一次；
    • Grabpass 一般用在水体或者扭曲效果时；
  • 概念：Surface Shader
    • 没有必要多去了解 Surface Shader，其只是 Unity 当中，为初学者提供的一个封装了片段和顶点着色器的 Shader；
    • 能不用就不要用 Surface Shader (表面着色器)，尽量去使用 vertex/fragment 顶点片断着色器；
  • 概念：ShaderLod
    • 利用 ShaderLod 来实现高中低配硬件的区分；
  • 概念：Overdraw
    • 假设当前场景当中有大量的半透明渲染，由于半透明渲染是从后往前渲染，这意味着同一个像素点、其可以会经过很多次的绘制，因此产生性能问题；
    • 可以使用 Tags{"RenderType"="Transparent"} 来看当前场景中的 Overdraw 问题；
    • 图示的场景中，黑色的部分是没有 Overdraw 的，而颜色越来越亮的部分是 Overdraw 最严重的地方；
    • 自己写的 Shader 为了能够看到以上图示的效果，需要将其渲染类型设置为透明，即：
      • Tags{"RenderType"="Transparent"}；
  • 概念：变体优化
    • 变体的数量直接影响 ShaderLab 内存的占用，能少则少；
    • 尽量不要去用内置的 Standard 材质，会生成大量的变体，可以自己修改定制一个；
注意：数值精度的支持

变体优化

变体的组合与控制

概念：变体与变体爆炸

  • Unity 的 Shader 并非单一的 GPU 代码，它包含了渲染状态、属性、Pass 以及基于不同关键字（Keywords）​​ 组合编译出的多个版本，每个版本就是一个变体（Variant）；
  • 变体通过 #pragma multi_compile 或 #pragma shader_feature 等指令定义，允许同一份 Shader 代码根据不同的条件（如平台、图形质量、功能开关）生成不同的代码路径，以适应不同的渲染需求；
  • 变体爆炸（Variant Explosion）​​ 是指当 Shader 中定义了多个关键字组合时，变体的数量会呈指数级增长。而因为每个变体都需要编译，因此会导致体积变大、构建时间长等问题；

概念：变体的组合与控制

  • 概念：变体如何组合生成
    • 图示：查看变体数量
      • 点击要查看的 Shader 文件，在 Inspector 视图中可以看到；
    • 图示：当前 Shader 的变体
  • 概念：优化方式一 - 自定定义需要的变体
    • 使用 #pragma multi_compile _ FOG_LINEAR 来自己指定变体
    • 其中下划线代表空变体
    • 在 Pass 中声明后，接下来只有两个变体；
  • 概念：优化方式二 - 跳过变体
    • 使用 #pragma sikp_variants FOG_EXP FOG_EXP2 来将指定的变体剔除；
    • 结果同样还是只剩下两个变体；
  • 概念：变体的组合
    • 代码：
      • #pragma multi_compile_fog //不跳过任何变体
      • #pragma multi_compile A B C
    • 结果：
      • 一共有 12 个变体；
    • 代码：
      • #pragma multi_compile_fog //不跳过任何变体
      • #pragma multi_compile A B C
      • #pragma shader_feature D E F
    • 结果：
      • 36 个变体，但是点击 Skip unused shader_feature 后只剩下三个；
    • 分析：
      • 之所以只会有 D 变体被使用上，是因为 shader_feature 的特点是如果材质球上没有使用 D、E、F，则默认只会使用第一个；
  • 结果：
    • （1）多使用 shader_feature，少使用 multi_compile；
    • （2）但是如果假设 ABC 三个变体的内容都是程序运行时动态更改的，则需要使用 multi_compile；
    • （3）如果 Shader 是一个美术资源使用开关控制的，程序不会进行更改，此时用 shader_feature 很好；

补充：dynamic_branch

  • 概念：
    • dynamic_branch 属于动态分支的一种实现方式，与在着色器编译时就决定代码路径的静态分支不同，动态分支的条件（比如由纹理采样结果、顶点属性或 uniform/constant 变量计算出的值）需要等到 GPU 真正执行着色器程序时才能确定；
    • 它的主要特点是不会产生额外的着色器变体，无论条件在运行时如何变化，GPU 执行的都是同一份编译好的着色器程序，Unity 无需为每种可能的条件组合预先编译多个变体；
  • 分析：
    • 理想情况：
      • 如果一个线程束​内的所有线程（例如 32 个）都满足相同的条件（都走 if 分支或都走 else 分支），GPU 可以高效地让整个线程束执行同一个路径，性能损耗相对较小。当分支条件基于 Uniform 变量（在整个绘制调用中恒定）时，通常就是这种情况；
    • 最差情况：
      • 如果一个线程束内的线程需要走不同的路径（一部分走 if，另一部分走 else），GPU 的处理就会变得复杂；
      • 现代 GPU 通常采用 “分支串行化”​ 策略：
        • 先执行其中一个分支的代码（比如 if 块），此时不走这个分支的线程会被掩蔽（Masked）而处于等待状态；
        • 执行完后，再切换去执行另一个分支的代码（else 块），之前被掩蔽的线程开始工作，而刚才活跃的线程被掩蔽；
      • 最终，总执行时间几乎是两个分支时间之和，并行效率大打折扣；

Shader 的变体收集器

概念：Shader 的变体收集器

  • 概念：
    • 变体收集器核心目的是精准控制哪些变体需要被包含在最终构建中，其重要作用体现在提前编译并加载关键的 Shader 变体，避免在运行时首次遇到这些变体时才进行编译，从而有效消除因实时编译导致的渲染卡顿和帧率下降；
    • 并且，变体收集器确保那些运行时必需但未被任何场景材质直接引用的变体不会被构建剥离（Stripping），例如通过代码动态启用关键字组合的变体，或者在某些特定条件下才会出现的变体；
    • 不过如果变体数量是趋近于无限的，那么变体收集也不可能完成 -> 只能尽量保证变体尽量剔除、变体尽量收集；
  • 图示：创建变体收集器
    • 可以在变体收集器中，点击 Add Shader 来选择收集哪些变体；
  • 图示：显示并保存当前 Unity 自动收集的变体
    • 可以点此按钮来自动将场景中会使用到的变体生成到变体收集器当中；
  • 图示：选择剔除掉一些 Unity 中自带、但是开发者不选择使用的 Shader
    • 可以将其设置为 No Support，使得一些 Unity 中的 Shader 不被加载进来；

Shader 编绎目标级别

概念：Shader 编绎目标级别

  • 概念：Shader Model
    • Shader Model 由微软提出，要求显卡厂商按 SM 级别提供对应的功能与指令支持；
    • 不同的 SM 包含不同的指令集与 Shader 规范，同时高版本的 SM 是低版本的 SM 的超集；
  • 概念：Shader 编译目标级别
    • 方式一：指定 #pragma target 3.0
    • 方式二：指定需要什么特性 - #pragma require xxx
  • 代码：使用 #pragma target x.x
Shader "Unlit/Course_09_Fog"
{
	CGINCLUDE
	struct appdata
	{
		float4 vertex : POSITION;
	};

	struct v2f
	{
		float4 vertex : SV_POSITION;
		// 相当于是在雾效开启时，定义一个float类型的变量 fogCoord
		UNITY_FOG_COORDS(1)
	};

	fixed4 _Color;

	v2f vert (appdata v)
	{
		v2f o;
		o.vertex = UnityObjectToClipPos(v.vertex);

		// 传入的 o.vertex 表示裁剪空间下的坐标
		UNITY_TRANSFER_FOG(o,o.vertex);
		return o;
	}

	fixed4 frag (v2f i) : SV_Target
	{
		fixed4 c = _Color;
		// 3.0 代表的就是 30
		#if (SHADER_TARGET >= 30)
			// 高配
			return fixed4(1,1,0,1)
		#else
			// 低配
			return fixed4(1,0,1,1)
		#endif

		// 第二个参数传入物体的颜色
		UNITY_APPLY_FOG(i.fogCoord,c);
		return c;
	}
	ENDCG

    Properties
    {
        _Color("Color",Color) = (1,0,1,1)
    }
    
    // 高配置使用的SubShader
    SubShader
    {
        Tags { "RenderType"="Opaque" }
        LOD 600

        Pass
        {
            CGPROGRAM
            #pragma vertex vert
            #pragma fragment frag
            #pragma multi_compile_fog
			#pragma target 3.0
            #include "UnityCG.cginc"

            ENDCG
        }
    }
    
    // 低配置使用的SubShader
    SubShader
    {
        Tags { "RenderType"="Opaque" }
        LOD 200

        Pass
        {
            CGPROGRAM
            #pragma vertex vert
            #pragma fragment frag
            #pragma multi_compile_fog
			#pragma target 2.0
            #include "UnityCG.cginc"

            ENDCG
        }
    }
}

设置 Shader 只编译到目标平台

概念：设置 Shader 只编译到目标平台

  • 概念：
    • 当前写好一个 Shader 之后，其可能会被编译到很多个平台，其会编译成很多种不同的语言，比如 metal、glcore、gles3 等；
    • 而 Unity 帮开发者写好了将 Shader 编译到不同平台的工具。如果开发者的 Shader 只需要编译到部分平台，此时可以进行配置，使得当前 Shader 只编译到目标平台；
  • 方法：当前 Shader 只编译到目标平台
    • #pragma exclude_renderers：排除掉某个渲染器；
    • #pragma only_renderers：只渲染到某个渲染器；
  • 分类：平台的内部标识符
  • 注意：
    • 如果不写上述指令，则默认是全平台都渲染；
  • 代码：
Shader "Unlit/Course_09_Fog"
{
    Properties
    {
        _Color("Color",Color) = (1,0,1,1)
    }
    
    SubShader
    {
        Tags { "RenderType"="Opaque" }
        LOD 100

        Pass
        {
            CGPROGRAM
            #pragma vertex vert
            #pragma fragment frag
            #pragma multi_compile_fog

            #include "UnityCG.cginc"
            
            // 只渲染到特定平台
            // 比如当前是只渲染到 Direct3D 11
            // #pragma only_renderers d3d11
            
            // 渲染到OpenGL的几个平台
            #pragma only_renderers gles gles3 metal glcore

            struct appdata
            {
                float4 vertex : POSITION;
            };

            struct v2f
            {
                float4 vertex : SV_POSITION;
                // 相当于是在雾效开启时，定义一个float类型的变量 fogCoord
                UNITY_FOG_COORDS(1)
            };

            fixed4 _Color;

            v2f vert (appdata v)
            {
                v2f o;
                o.vertex = UnityObjectToClipPos(v.vertex);

                // 传入的 o.vertex 表示裁剪空间下的坐标
                UNITY_TRANSFER_FOG(o,o.vertex);
                return o;
            }

            fixed4 frag (v2f i) : SV_Target
            {
                fixed4 c = _Color;

                // 第二个参数传入物体的颜色
                UNITY_APPLY_FOG(i.fogCoord,c);
                return c;
            }
            ENDCG
        }
    }
}
  • 图示：点击 Compile and code 之后，可以看到为各个平台生成的代码
  • 总结：
    • 手动设置 Shader 只编译到目标平台后，可以减少游戏启动时 Shader 的编译时间，以及 Shader 文件所占的体积。但是并不会减少运行时 GPU 的性能消耗；

GPU 逻辑管线

概念：GPU 逻辑管线

  • 概念：渲染管线
  • 概念：NVIDIA 的 GPU 逻辑管线
  • 解释：
    • 第一步：图像 API
      • 这是管线的起点，指代如 DirectX、OpenGL、CUDA 等应用程序接口。开发者通过它们向 GPU 下发渲染指令 DrawCall，CPU 调用这些 API，将顶点数据、纹理、着色器程序、状态设置等命令打包，通过驱动程序传递给 GPU；
    • 第二步：图形处理集群 - GPC
      • 工作被分配交叉开关（Work Distribution Crossbar）分配到不同的 GPC 上进行并行处理，多个 GPC 协同工作，共同完成渲染任务
      • GPC 是 GPU 的一个主要分区单元，一个 GPU 芯片由多个 GPC 组成。每个 GPC 都具备完整的光栅化（Raster Engine）、着色（SM）和光栅操作（ROP）能力，可以看作是 GPU 中的一个“自治”单元；
    • 第三步：SM - Streaming Multiprocessor
      • SM 是 Fermi 架构的核心计算单元，所有顶点着色、片元着色、曲面细分、几何着色以及通用计算（CUDA）都在 SM 中执行，一个 GPC 内包含多个 SM；
      • SM 接收由分配器分配来的线程组（Warps），并利用其内部的大量核心（Cores）以单指令多线程（SIMT）的方式并行执行它们；
    • 第四步：光栅化
      • 对处理完的数据进行光栅化，光栅化完成之后、还是利用 SM 单元，此时是使用 SM 处理片元的数据；
    • 第五步：ROP + FB
      • 进行颜色回合、画面合成，并输出到缓冲区 FB 中；
  • 解释：GPU 处理架构
    • 图示：
    • 解释：
      • 顶点拾取 (Vertex Fetch)
        • 这是管线的起点，GPU 从这里读取描述 3D 模型形状的原始顶点数据（如位置坐标、法线方向、纹理坐标、颜色等）从显存（DRAM）中；
        • 这是一个固定功能阶段，GPU 根据 DrawCall 指令，知道要去哪里获取哪些数据。它会高效地将这些数据从慢速的 DRAM 批量读取到高速的 L1/L2缓存 中，为后续处理做准备；
      • 顶点着色器 (Vertex Shader)
        • 这是第一个可编程阶段，每个顶点都会独立执行一次顶点着色器程序；
        • 其主要任务是进行坐标变换，将顶点从各自的模型空间 (Model Space)​​ 转换到统一的屏幕空间 (Screen Space)​，这通常通过乘以一系列矩阵（模型矩阵、视图矩阵、投影矩阵，合称 MVP 矩阵）来实现。此外，它还可以计算并输出后续阶段所需的数据，如光照信息、纹理坐标等；
      • 曲面细分与几何着色器 (Tessellation & Geometry Shaders)
        • 这两个是可选的可编程阶段，用于在原有图元基础上动态生成或修改几何图形；
        • ​曲面细分着色器 (Hull/Domain Shader)​​：
          • 将一个粗糙的原始图元（如一个三角形面片）细分成大量更小的三角形，从而增加模型细节，常用于让表面看起来更平滑或更粗糙；
        • ​几何着色器 (Geometry Shader)​​：
          • 以整个图元（点、线、三角形）为单位，可以生成新的图元或丢弃现有图元;
      • 光栅操作单元 - ROP
        • 是管线的最后一步，也是一个固定功能阶段。它负责将像素着色器计算出的颜色与帧缓冲区中现有的颜色进行合并，处理深度和透明度，最终决定像素的结果；
      • 缓存与内存：
        • DRAM 是主存储器，容量大但速度慢，用于存储所有纹理、顶点缓冲区和帧缓冲区；
        • L1/L2缓存 是位于 GPU 芯片上的高速存储器，用于缓存从 DRAM 读取的数据，极大减少访问延迟；

编译内容解析

概念：常量缓冲区

  • 概念：
    • 常量缓冲区用于存储 Shader 执行过程中所需的恒定数据，由 CPU 设置，GPU 只读；
  • 分类：
    • （1）UnityPerDraw：
      • 通常包含顶点着色器的数据，如变换矩阵；
    • （2）UnityPerFrame：
      • 包含每帧全局数据（如 VP 矩阵、摄像机位置、光照信息）；
  • 代码：
// 常量缓冲区"UnityPerDraw"（176字节），绑定到槽位0
// 存储与当前绘制对象相关的变换矩阵
Constant Buffer "UnityPerDraw" (176 bytes) on slot 0 {
  Matrix4x4 unity_ObjectToWorld at 0 // 模型空间到世界空间的变换矩阵
}
// 常量缓冲区"UnityPerFrame"（368字节），绑定到槽位1
// 存储与当前帧相关的全局变量，如视图投影矩阵
Constant Buffer "UnityPerFrame" (368 bytes) on slot 1 {
  Matrix4x4 unity_MatrixVP at 272 // 世界空间到裁剪空间的视图投影矩阵
}

概念：顶点着色器

  • 概念：
    • 顶点着色器阶段主要负责几何变换，顶点着色器对每个顶点执行，负责顶点变换；
  • 代码：
// 以下是顶点阶段进行的实际计算指令
vs_4_0                              // 顶点着色器模型 4.0
dcl_constantbuffer CB0[4], immediateIndexed  // 声明常量缓冲区CB0（对应slot 0）
dcl_constantbuffer CB1[21], immediateIndexed // 声明常量缓冲区CB1（对应slot 1）
dcl_input v0.xyz                    // 输入：模型空间的顶点位置（POSITION语义）
dcl_output_siv o0.xyzw, position    // 输出：裁剪空间中的顶点位置（SV_POSITION语义）
dcl_temps 2                         // 声明使用2个临时寄存器

// 以下汇编指令执行顶点变换：model -> world -> clip space
// 将顶点从模型空间变换到世界空间 (mul & mad 指令相当于矩阵乘法)
0: mul r0.xyzw, v0.yyyy, cb0[1].xyzw // r0 = v0.y * unity_ObjectToWorld[1]
1: mad r0.xyzw, cb0[0].xyzw, v0.xxxx, r0.xyzw // r0 = v0.x * unity_ObjectToWorld[0] + r0
2: mad r0.xyzw, cb0[2].xyzw, v0.zzzz, r0.xyzw // r0 = v0.z * unity_ObjectToWorld[2] + r0
3: add r0.xyzw, r0.xyzw, cb0[3].xyzw // r0 = r0 + unity_ObjectToWorld[3] (加上平移分量)

// 将世界空间位置变换到裁剪空间 (与unity_MatrixVP相乘)
4: mul r1.xyzw, r0.yyyy, cb1[18].xyzw // r1 = r0.y * unity_MatrixVP[18]
5: mad r1.xyzw, cb1[17].xyzw, r0.xxxx, r1.xyzw // r1 = r0.x * unity_MatrixVP[17] + r1
6: mad r1.xyzw, cb1[19].xyzw, r0.zzzz, r1.xyzw // r1 = r0.z * unity_MatrixVP[19] + r1
7: mad o0.xyzw, cb1[20].xyzw, r0.wwww, r1.xyzw // o0 = r0.w * unity_MatrixVP[20] + r1

8: ret                              // 返回

常见寄存器及其命名

概念：常见寄存器及其命名

  • 概念：临时寄存器 - r0
    • GPU 核心内部的高速寄存器文件，位于 SM 内部，访问延迟极低（通常 1-2 个时钟周期），数量有限；
  • 概念：常量寄存器 - cb0
    • 存储于 GPU 的常量缓存/一级缓存当中，数据来源于显存中的常量缓冲区，GPU 会将其缓存到片上的高速缓存中，以减少访问显存的延迟；
  • 概念：输入寄存器 - v0
    • 用于顶点着色器，接收来自顶点缓冲区的输入数据（如位置、法线、纹理坐标等）
  • 概念：纹理寄存器 - t0
    • 绑定纹理资源，在像素着色器中用于纹理采样操作；
  • 概念：输出寄存器 - o0
    • 着色器的输出；
    • 其中：顶点着色器通常输出位置(SV_Position)，像素着色器输出颜色(SV_Target)；

常见优化方法

常量优化

概念：常量优化

  • 概念：
    • 在 Shader 外部或编译时预先计算好一些值，可以避免在 GPU 上进行昂贵的运行时计算；
  • 分类：
    • （1）CPU 端预计算：
      • 对于一些复杂的计算或倒数等操作，可以在 C# 脚本中计算好结果，然后通过 material.SetFloat 等命令传递给 Shader 变量；
      • 例如，预先计算 1.0f / _Glossiness 并传入 Shader，避免在 Shader 中进行昂贵的除法运算；
    • （2）编译时常量折叠：
      • Shader 编译器在编译时会自动识别并计算常量表达式，例如 return 2 * 3;；
      • 最终，生成的指令中只会包含计算结果，比如 mov o0.xyzw, l(6,6,6,6)，而不会包含原始的乘法指令，进而节省指令数；
  • 示例：
// 以下代码生成的编译指令会进行常量折叠，因此在片元着色器阶段不会存在计算
fixed4 frag(v2f i) : SV_Target
{
	fixed e0 = 5*4 + sin(1/5);
	float e1 = e0;
	return e1; 
}
注意：关于常量优化
  • 并不是所有的平台都会进行编译时的常量折叠，因此最好在遇到常量计算时，自己将计算好的结果直接填入；

概念：无法常量优化的清空情况

  • 示例：

Properties
{
	_Value("Value",vector) = (0,0,0,0)
}

float4 _Value;

// 以下代码生成的编译指令存在一条运算

{
	float a = _Value.x;
	float b = _Value.y;
	float c = _Value.z;
	float d = _Value.w;
	
	fixed e0 = 5*4 + a;
	return e0; 
}
  • 代码：生成的编译代码
ps_4_0

// 声明一个常量缓冲区 (Constant Buffer)，命名为 CB0。
// 该缓冲区绑定在特定的寄存器槽位，包含至少 3 个向量（或矩阵行）的数据。
// `immediateIndexed` 表明其索引方式是直接索引
dcl_constantbuffer CB0[3], immediateIndexed

// 声明着色器的输出变量 o0，它是一个 float3 类型的向量 (xyz)。
// 在像素着色器中，这通常对应着渲染目标的颜色输出（例如 RGB）
dcl_output o0.xyz

// 指令 0: 加法操作 (add)
// 将常量缓冲区 CB0 中第 2 个索引位置（即第 3 个常量，索引从0开始）的 x 分量 (cb0[2].xxxx)，
// 与一个立即数向量 l(5.000000, 5.000000, 5.000000, 5.000000) 相加。
// 结果存储到输出寄存器 o0 的所有分量中
0: add o0.xyzw, cb0[2].xxxx, l(5.000000, 5.000000, 5.000000, 5.000000)

// 指令 1: 返回 (ret)
// 结束像素着色器的执行。
1: ret
总结：编译后的指令
  • 加法和减法在编译之后，都是使用 add 指令。除法使用 div 指令，乘法使用 mul；
  • 但是如果一个表达式的当中，存在乘法和加法时，此时只需要一条 mad 指令；
总结：什么时候进行常量优化
  • 当运算符当中的值都是固定的值时，其会自动开启常量优化；
  • 但是当表达式中，有对象的值是需要在运行时才能确定时，其无法进行常量优化；
注意：
  • 常量缓冲 CB0[3] 说明该缓冲区至少需要容纳 3 个 float4 大小的数据；

条件分支与循环优化

概念：条件分支与循环优化

  • 概念：条件分支优化
    • 在 Shader 中，条件分支需要谨慎处理，因为 GPU 是并行处理器，处理分支的方式与 CPU 不同；
    • 具体而言，条件分支有几种类型：
    • 当一个 Warp (一组并行线程) 中的线程需要执行不同的分支路径时，GPU 通常会先执行所有线程的 if 路径，再执行所有线程的 else 路径，然后根据每个线程的条件掩码丢弃不需要的结果。这会导致实际计算量增加，因此分支发散会导致很大的性能消耗；
  • 概念：优化方法
    • （1）优先使用静态分支​：
      • 对于在 Shader 编写时就能确定的特性（如是否使用法线贴图），使用 #ifdef 或 #if 预处理指令。编译器会在编译时直接剔除未使用的代码，不会产生任何运行时开销；
    • （2）​使用 Step/Lerp 替代动态分支​：
      • 对于简单的条件判断（尤其是像素着色器中），使用 step 或 lerp 等数学函数来避免 if 语句，通常能获得更好的性能；
      • 编译器也常将简单 if 优化为 movc (条件移动) 指令，这与 lerp 原理类似；
  • 概念：循环优化
    • Shader 中的循环同样需要小心处理，因为它们在 GPU 上也是并行执行的；
  • 概念：循环优化方法
    • （1）​展开短循环​：
      • 如果循环次数很少且固定（例如 2-4 次），手动展开循环可以避免循环控制带来的开销，有时能带来显著的性能提升，尤其是在一些低端移动设备上；
    • （2）确保循环次数是常量​：
      • 尽量使用编译时常量作为循环次数上限；
      • 如果循环次数由 Uniform 变量决定，某些 GPU 可能无法进行优化，甚至可能导致性能下降；
总结：Unity 内置分支或循环优化指令

常用内置方法及其优化

概念：常用内置方法及其优化

  • 概念：归一化指令 - normalize(v)
    • normalize 函数用于将向量转换为单位向量​（长度为 1），保持方向不变；
    • 在 D3D 的 HLSL 中，normalize(v) 通常被编译为一系列底层指令：
        1. 计算向量长度​：使用 dp3（点积）指令计算 v.x*v.x + v.y*v.y + v.z*v.z，或者使用 length 函数；
        1. 计算平方根倒数​：使用 rsq 指令计算 1/sqrt(length)，这是关键步骤，因为平方根倒数计算在 GPU 上通常有专用指令；
        1. 缩放向量​：将原向量的每个分量乘以该平方根倒数，得到单位向量；
  • 概念：归一化优化建议
      1. 如果同一个向量在着色器中需要多次归一化，​计算一次并存储结果，而不是多次调用 normalize；
      1. 尽可能在 CPU 端或顶点着色器中完成归一化，并将结果传递给片元着色器。片元着色器的执行频率远高于顶点着色器，这样做能显著减少计算量；
  • 概念：绝对值指令 - abs(x)
    • abs 在 D3D 汇编中通常对应一条简单的 abs 指令，这是一条非常高效的基本指令，通常在一个时钟周期内完成，因为它直接操作寄存器的符号位；
  • 概念：点积指令 - dot
    • 在 D3D 汇编中，点积通常由 ​dp 系列指令​（如 dp2, dp3, dp4）实现，具体取决于向量的维度；
    • 比如：dot(float2, float2) → dp2 r0, v0, v1
  • 概念：优先将同一维度的向量进行相乘
    • 优化前：float3 e0 = _Value.xyz * a * b * _Value.yzw * c * d;
    • 优化后：float3 e0 = (_Value.xyz * _Value.yzw) * (a * b * c * d);
    • 编译后从 11 行代码减少到 5 行；
  • 概念：asin 函数
    • 这条指令十分消耗性能，会生成 11 条指令，开销很大，谨慎使用；
注意：常用内置方法的注意事项
    1. 关于归一化指令：如果输入是零向量，HLSL 的 normalize 函数会返回零向量以确保稳定性，但这在数学上是未定义的，因此最好在代码中避免；
    1. 进行点积操作时，负号可以适当的移动到变量当中；
总结：指令优化总结
