Lecture 5：异步编程

什么是异步

基础概念

概念：同步操作与异步操作

  • 概念：什么是同步操作与异步操作
    • 同步操作 (synchronous operation) 先完成其工作再返回调用者；
    • 异步操作 (asynchronous operation) 的大部分工作则是在返回给调用者之后才完成的；
  • 概念：异步编程的本质
    • 异步编程的核心在于非阻塞和并发处理；
    • 与同步操作必须等待当前任务完成才能继续执行下一个任务不同，异步操作允许程序在发起一个耗时任务后，​不必原地等待，而是立即返回并继续执行后续代码。当那个耗时任务在后台完成时，程序再通过某种机制（如回调、事件）来处理结果；
    • 在同步模式下，一个需要从网络获取数据的任务会“冻住”整个程序，CPU 只能空闲等待。而异步模式下，CPU 在数据到达前可以去处理其他计算任务，就像一位高效的管家，总能找到事情做，避免资源闲置；
  • 注意：异步 ≠ 多线程
    • 多线程是实现异步的一种方式，但并非全部；
    • 许多异步操作（特别是 I/O 密集型任务）依赖于操作系统的底层非阻塞机制，在等待期间并不占用线程，从而可以实现用少量线程处理大量并发任务；

常见应用场景

概念：常见应用场景

  • 概念：I/O 密集型应用
    • 包括网络请求​（如调用 API、微服务通信）、文件读写​（访问本地或网络存储）、以及数据库操作；
    • 这些操作的共同点是大量时间花在等待数据传输上，而非 CPU 计算。使用异步可以避免线程大量阻塞，用极少资源支撑高并发；
  • 概念：需要高实时性的客户端响应
    • 在图形界面（GUI）程序中，主线程负责响应用户交互。如果在这个线程上执行同步耗时操作（如复杂计算或网络请求），界面就会“卡住”；
    • 通过异步编程，可以将这些任务移出主线程，确保用户操作始终能得到即时响应；

问题：Callback Hell

概念：Callback Hell

  • 概念：
    • Callback Hell（回调地狱）是异步编程中一个经典问题，简单来说，它就像是你为了完成一件事（比如计划一次旅行），需要不停地等待前一个步骤的结果才能进行下一个步骤：
      • 先查天气，等天气结果出来了再订机票，等机票订好了再订酒店……
    • 如果这些“等待”和“后续动作”都以嵌套的方式写进代码，代码就会向右缩进得越来越深，形成一个难以阅读和维护的“金字塔”或“阶梯”状结构；
  • 示例：
const fs = require('fs');
fs.readFile('file1.txt', (err, data1) => {
  if (err) throw err;
  fs.readFile('file2.txt', (err, data2) => {
    if (err) throw err;
    fs.readFile('file3.txt', (err, data3) => {
      if (err) throw err;
      // ... 可能还有更多嵌套
      console.log(data1, data2, data3);
    });
  });
});
  • 解决：Async/Await
    • 其类似于同步的写法，但是却实现一个异步操作；
    • async ​关键字用于声明一个异步函数，​ await 关键字则用于等待一个任务完成，并返回其结果；

异步编程基础

await

概念：await 关键字

  • 概念：await
    • 用于在异步方法内部，​等待一个异步操作完成；
    • 当执行到 await 时，其暂停当前方法的执行，将控制权返回给调用方，待等待的任务完成后，再从暂停处恢复执行；
    • 其后通常跟一个返回 Task、Task<T> 等可等待对象的表达式；
  • 语法：
    • var result = await expression;
    • statement(s);
    • 等待一个异步操作（expression）完成，获取它的结果（result），然后继续执行后面的语句；
  • 分析：await 的实现
var result = await expression;
statement(s);
// 编译器会将上述代码转化为以下代码

// 1. 获取等待器 (Awaiter)
// expression 通常是一个返回 Task 或 Task<T> 的方法调用
// GetAwaiter() 方法返回一个“等待器”对象，这个对象负责协调异步操作
// 这个对象（通常是 TaskAwaiter<T>）知道异步操作的状态（是否完成）
var awaiter = expression.GetAwaiter();

// 2. 注册延续（Continuation）—— 异步的“回调函数”
// 这是最核心的一步，调用等待器的 OnCompleted 方法
// 它的作用是：“当这个异步操作（expression）在后台完成时，请执行我提供的这个回调函数”
// 这个回调函数就是一个“延续”（Continuation），它包含了 await 之后的所有代码
awaiter.OnCompleted ( () => 
{
    // 3. 获取结果并继续执行
    // 当异步操作完成后，运行时（如线程池）会调用这个回调
    // GetResult() 会获取异步操作的结果。如果操作失败，这里会抛出异常
    // 对于 Task<T>，GetResult() 返回类型 T；对于普通 Task，它返回 void
    var result = awaiter.GetResult();

    // 4. 执行原代码中 await 之后的所有语句 (statement(s))
    // 此时，result 已经可用，程序可以从 await 的地方“无缝”恢复执行。
    // 注意：此回调可能在原线程上执行，也可能在线程池的另一个线程上执行，这取决于同步上下文
});
  • 分析：
    • 编译器会将整个 async 方法编译成一个复杂的类（状态机），​这个状态机内部有字段来保存所有的局部变量和方法当前的执行位置；
    • 因此，即使方法“返回”了，代表这个方法的那个状态机对象依然被回调函数引用着，它和它的所有状态（局部变量）都保留在内存中。当回调执行时，状态机就知道应该从哪儿继续，并且能访问到之前所有的变量值；
  • 总结：
    • 将 await 之后的代码打包成一个回调函数，注册给异步操作。然后立即返回，释放当前线程；
    • 当异步操作完成后，系统在某个合适的时机调用这个回调，从而让程序从“中断”处恢复执行；

async

概念：async 关键字

  • 概念：async
    • async 是一个关键字，用于声明一个函数是异步函数，其向编译器表明该方法内部包含异步操作，允许使用 await 关键字；
    • 标记了 async 的函数，返回值通常返回 Task、Task<T> 或 ValueTask / ValueTask<T>；
  • 概念：
    • async 修饰符会指示编译器将 await 作为一个关键字而非标识符来避免二义性；
    • async 修饰符和 unsafe 修饰符类似，都不会对方法签名或者公共元数据产生影响，它只影响方法内部的执行细节，因此在接口上添加 async 是没有意义的；
    • 但是这种方式也是合法的。例如，可以在一个重写的非异步的虚方法上添加 async (但前提是方法签名必须保持不变)；
  • 分析：async 与异步函数的执行过程
    • 在一个方法签名前加上 async 关键字之后，它就成为了一个异步方法；
    • 这个修饰符本身并不使方法自动并行运行，它的核心作用是告知编译器该方法内部将包含 await 表达式，编译器会据此对方法进行特殊处理，将其转换成一个状态机，这是实现非阻塞等待的基础；
    • 然后当进入方法执行，当遇到 await 表达式时，当前异步方法的执行会被挂起（暂停），控制权立即返回给该方法的调用者。这就好比在迭代器中遇到 yield return，方法在此处“让步”；
    • 最重要的是，​挂起并不会阻塞当前的线程​（例如 Unity 的主线程）。该线程会被释放，从而可以去处理其他工作，比如响应用户操作或处理其他请求，这保证了应用程序的响应能力；
    • 同时会在等待的任务上附加一个延续（Continuation），保证任务结束时能跳回方法。在内部，await 会向它等待的那个任务（一个 Task 对象）注册一个回调函数，这个回调函数里封装了 await 之后的所有代码，当后台的耗时任务（如网络请求、文件读写）完成时，这个回调函数会被触发。运行时（通常是线程池）会安排执行这个回调，从而使异步方法从 await 语句之后的位置恢复执行​；

示例：异步方法基础使用

示例一：获取异步方法的返回值

using System;
using System.Threading.Tasks;

class Program
{
    static async Task Main(string[] args)
    {
        Console.WriteLine("开始获取数据...");
        
        // 因为使用了关键字，因此Main方法在await处暂停，等待FetchDataAsync完成
        string data = await FetchDataAsync();
        
        // 这行代码会在FetchDataAsync完全完成后执行
        Console.WriteLine("【主线程】Main方法在调用AsyncMethod后继续执行");
        Console.WriteLine($"获取到的数据是: {data}");
        Console.WriteLine("【主线程】Main方法在调用AsyncMethod后继续执行");
    }

    static async Task<string> FetchDataAsync()
    {
        // 模拟一个耗时的网络请求或文件读取操作
        await Task.Delay(2000); // 模拟耗时2秒的操作
        
        // 返回结果
        return "这是从异步方法返回的数据";
    }
}
  • 输出结果：
开始获取数据...
(这里等待两秒)
【主线程】Main方法在调用AsyncMethod后继续执行
获取到的数据是: 这是从异步方法返回的数据
【主线程】Main方法在调用AsyncMethod后继续执行

分析：为什么 async 函数使用 Task<string> 作为返回值类型

  • 分析：返回值包裹 Task 的作用
    • async 函数返回 Task<T>（而非直接返回 T，如 string）主要基于以下几个核心原因：
      1. 表示一个“未来”的值：
      • Task<string> 代表一个正在进行中的异步操作，它是对操作结果的“承诺”，这个操作可能很耗时（如读取网络文件），无法立即返回最终的 string。因此，Task<T> 封装了这个操作的生命周期，如未完成、完成、出错等状态；
      1. 实现非阻塞与协作：
      • 直接返回 string 意味着调用线程必须被阻塞，直到操作完成；
      • 而返回 Task<string> 使得方法可以立即返回，让出当前线程去处理其他工作，从而提高程序的响应性；
      1. 支持组合与状态监控：Task<T> 类型提供了如 Task.WhenAll, Task.WhenAny 方法来组合多个异步操作；
  • 分析：await 如何拆箱得到返回值
    • await 关键字的核心作用之一就是拆箱这个 Task<T> 对象，并提取出最终的 T 值。其内部过程可以概括为以下几步
    • （1）挂起与让出：当执行流遇到 await 表达式时，它会立即返回一个 Task<string> 给调用者。这使得调用线程不会被阻塞。异步方法内部的逻辑会在此处挂起；
    • （2）注册回调与等待：运行时系统会注册一个回调方法，然后开始异步等待 Task<string> 的完成。这个等待过程是高效的，不占用额外的线程；
    • （3）恢复与提取：一旦底层的异步操作（如文件加载完成），Task<string> 的状态会变为 RanToCompletion。这时，运行时会自动在适当的上下文中恢复异步方法的执行。await 运算符会自动从已完成任务的 Result 属性中取出 string 结果，并赋值给等号左边的变量；

示例二：基础异步使用

using System;
using System.Threading.Tasks;

class Program
{
    static async Task Main(string[] args)
    {
        Console.WriteLine("【主线程】开始执行Main方法");
        
        // 调用异步方法，但不等待它完成
        AsyncMethod();
        
        // 这行代码不会等待AsyncMethod完成就会执行
        Console.WriteLine("【主线程】Main方法在调用AsyncMethod后继续执行");
        Console.ReadKey(); // 等待用户按键，防止程序退出
    }

    static async Task AsyncMethod()
    {
        Console.WriteLine("【异步方法】开始执行");
        
        // 使用await等待一个异步操作（这里模拟一个耗时操作）
        await Task.Delay(2000); // 等待2秒
        
        // 这行代码会在2秒后执行
        Console.WriteLine("【异步方法】在await之后继续执行");
    }
}

分析

分析：是否在调用处使用 await 关键字

  • 概念：两种方法的不同
  • 分析：
    • await 是“等待”的开关​，它的存在与否，决定了调用方是否要等待异步方法完成后再继续；
    • 这提供了极大的灵活性，你可以选择等待一个耗时操作，也可以选择“即发即弃”（fire-and-forget）；
    • async / await 并不会创建新线程（除非你使用 Task.Run），其核心价值在于高效地等待，特别是在 I/O 密集型操作（如网络请求、文件读写）期间，可以释放当前线程（如 UI 线程）去处理其他工作，从而提升应用程序的响应能力和吞吐量；

示例三：结合异常处理

using System;
using System.Threading;
using System.Threading.Tasks;

class Program
{
    static async Task Main(string[] args)
    {
        // 创建一个取消令牌源，用于发送取消请求
        var cancellationTokenSource = new CancellationTokenSource();
        
        try
        {
            // 模拟在5秒后自动取消任务，你也可以手动调用cancellationTokenSource.Cancel()
            cancellationTokenSource.CancelAfter(5000);
            
            Console.WriteLine("开始执行一个可能被取消或出错的任务...");
            await ProcessDataAsync(cancellationTokenSource.Token);
            Console.WriteLine("任务成功完成！");
        }
        catch (OperationCanceledException)
        {
            Console.WriteLine("任务被用户取消了。");
        }
        catch (Exception ex)
        {
            Console.WriteLine($"任务执行过程中出错: {ex.Message}");
        }
    }

    static async Task ProcessDataAsync(CancellationToken cancellationToken)
    {
        for (int i = 0; i < 10; i++)
        {
            // 在每次循环开始时检查是否被取消，如果被取消会抛出OperationCanceledException
            cancellationToken.ThrowIfCancellationRequested();
            
            // 模拟一部分工作
            Console.WriteLine($"正在处理第 {i+1} 部分...");
            
            // 将取消令牌传递给Task.Delay，这样如果任务在延迟期间被取消，Delay也会被中断
            await Task.Delay(1000, cancellationToken);
        }
    }
}
