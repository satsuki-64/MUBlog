Lecture 4：线程安全

lock 关键字

基础概念

概念：lock 关键字

  • 概念：
    • lock 是 C# 中最常用、最简单的同步机制，本质上是 Monitor.Enter 和 Monitor.Exit 的语法糖；
    • 通过获取给定对象的互斥锁，确保同一时刻只有一个线程可以执行被锁保护的代码块（临界区）。它是可重入的，即同一线程可以多次获取同一个锁；
    • lock 语句在底层使用 try-finally 块实现，这意味着即使临界区代码抛出异常，锁也保证会被释放，不会造成永久死锁；
  • 使用：
    • 语法：
    • 解释：
      • 线程到达 lock 语句时，会尝试获取指定对象（_lockObj）的互斥锁；
      • 如果成功获取锁，该线程进入临界区执行代码。此时，其他任何线程若试图获取同一个锁对象，将被阻塞​；
      • 当线程执行完临界区代码，离开 lock 块时（无论是正常离开还是因异常离开），锁会被自动释放，此时等待锁的线程中的一个将获得锁并进入临界区；
注意：锁对象的选择
  • 锁对象必须使用引用类型​（class），不能是值类型（struct），因为值类型会被装箱，导致每次 lock 实际上作用于不同的对象，从而失去同步效果；
  • 建议将锁对象声明为一个专用于锁定的、私有、只读的字段；
  • 并且从 .NET 9 和 C# 13 开始，引入了新的 System.Threading.Lock 类型，专门用于 lock 语句：private readonly System.Threading.Lock _lockObj = new();

lock 的底层实现

概念：lock 的底层实现

  • 概念：Monitor 类
    • Monitor 类是 .NET Framework 提供的用于线程同步的底层原语，它通过向单个线程授予对象锁来控制对对象的访问；
    • 每个堆中的 .NET 对象都可以关联一个同步块索引​ (SyncBlock Index)，该索引指向一个真正的同步数据结构，这个数据结构在内部常称为 "Monitor" 或 "同步块"；
    • 当你对一个对象调用 Monitor.Enter 时，会进行以下步骤：
      1. CLR 会检查该对象的同步块索引；
      1. 如果当前没有线程持有该锁，CLR 会为当前线程获取锁（可能更新同步块信息）；
      1. 如果锁已被其他线程持有，当前线程会被阻塞，并放入与该锁关联的就绪队列中，等待锁释放；
      1. 当持有锁的线程调用 Monitor.Exit 时，CLR 会释放锁，并唤醒就绪队列中的线程（通常按一定顺序）来竞争锁；
  • 实现：lock 的实现
// 当你使用lock语句时，C# 编译器会在编译阶段将其转换为对Monitor类中Enter和Exit方法的调用

// 你编写的代码：
private readonly object _lockObj = new object();
lock (_lockObj)
{
    // 临界区代码
}

// 编译器大致会将其转换为：
object obj = _lockObj;

// 确保只有在Enter成功获取锁后（lockTaken为 true），才会在finally中执行Exit
// 避免在 Enter抛出异常后错误地调用 Exit
bool lockTaken = false;

try
{
	// Monitor.Enter 方法尝试获取指定对象上的排他锁
	// 如果其他线程已持有该锁，当前线程将阻塞，直到锁被释放
    Monitor.Enter(obj, ref lockTaken); 
    
    // 临界区代码
    // ...
}
finally
{
    if (lockTaken)
    {
	    // 无论如何都会释放锁
        Monitor.Exit(obj); 
    }
}

Interlocked 类

基础概念

概念：Interlocked 类

  • 概念：原子操作
    • 原子操作是指一个或多个不可被中断的操作序列；
    • 这些操作要么全部执行成功，要么完全不执行，不会出现部分执行的中间状态，也不会被其他线程打断；
    • 原子操作的实现通常依赖于 CPU 硬件指令（如 x86 平台的 CMPXCHG 指令）的支持；
  • 概念：什么是 Interlocked 类
    • System.Threading.Interlocked 类提供了一系列静态方法，用于执行基本的原子操作。这些方法在硬件级别实现，无需显式加锁，因此性能通常比传统的锁机制更高；
  • 特点：
      1. 相比锁机制（如 lock 或 Monitor），Interlocked 的开销更小，因为它避免了线程阻塞、上下文切换和内核态切换；
      1. 对于简单的读写、数学运算或条件替换，Interlocked 的方法非常直观；
      1. 功能有限，Interlocked 主要用于对单个变量进行简单操作。对于需要保护多个变量或复杂逻辑的临界区，仍然需要使用锁机制（如 lock, Monitor, Mutex）；
  • 补充：原子操作与 ABA 问题
    • 一个变量的值先从 A 变为 B，然后又变回 A。那么使用 CompareExchange 的线程可能会误以为该值从未改变，从而执行替换操作；
    • 解决方法通常是为数据附加版本号或使用带版本的引用；
