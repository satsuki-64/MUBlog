托管堆

面向对象中的资源

概念：面向对象中的资源

  • 概念：
    • 在面向对象的环境中，任何一种类型都代表了计算机中可供程序使用的一种资源。如果要使用这种资源，就需要为这种资源分配一定的内存；
  • 概念：资源分配流程
    • 第一步：new 操作符 -> 对应为 IL 代码 newobj
      • 调用 IL 代码 newobj，为代表资源的类型分配内存；
      • 由托管堆进行资源申请与分配
      • 此步骤通常由 new 操作符来触发并完成；
    • 第二步：类型的实例构造函数来设置初始状态
      • 根据类型的状态设置，初始化内存，设置资源的初始状态并使得此资源可用；
      • 由类型的实例构造函数来设置初始状态；
    • 第三步：使用资源
      • 访问类型的成员来使用此资源；
    • 第四步：摧毁资源
      • 摧毁资源的状态，以进行垃圾的清理
    • 第五步：GC 机制触发
      • 由 GC 回收期释放内存
    • 面向对象中通过 new 分配的资源，会由 GC 机制进行回收；

概念：为什么需要内存管理

  • 概念：以 C++ 为例
    • 在需要程序员手动管理内存的语言中（比如当 C++ 不使用 shareptr 或其他基于对象的指针时），程序员经常会产生两种错误：
        1. 忘记释放不再需要的内存，进而造成内存泄漏；
        1. 尝试使用已经释放的内存，造成安全漏洞；
    • 由于这两种错误经常是在程序运行时才能发现，因此程序发生异常时调试困难；
  • 概念：C# 中的内存管理
    • 自动内存管理是公共语言运行时在托管执行过程中提供的服务之一；
    • 公共语言运行时的垃圾回收器为应用程序管理内存的分配和释放，对开发人员而言，这就意味着在开发托管应用程序时不必编写执行内存管理任务的代码。自动内存管理可解决常见问题，例如，忘记释放对象并导致内存泄漏，或尝试访问已释放对象的内存；
  • 概念：C# 中的解决方法 - 可验证的、类型安全的代码
    • 在资源分配阶段中的【第四步】，此步骤若不是由程序员来完成，而是交给程序的托管堆和垃圾回收器来自动进行时，此时程序就能进一步的保证安全；
    • 托管堆会自动完成垃圾的回收，并且当需要手动进行回收时，托管堆还提供了一个简化的模型；
  • 补充：需要特殊清理的类型
    • 部分包装了本机资源的类型，比如对文件、Socket、数据库连接资源的类型，可能无法使用自动进行的垃圾回收，而是需要程序员使用 GC 的手动触发；

什么是托管堆

概念：托管堆

  • 概念：
    • 在 C# 中，当初始化一个新进程时，CLR 会为进程保留一个连续的地址空间区域。这个保留的地址空间被称为托管堆；
    • 托管堆是一种用于存储和管理托管对象（即运行在 CLR 中的对象）的内存区域。它是 CLR 的一部分，为托管对象提供动态内存分配和释放的功能；
    • 托管堆是一种自动化内存管理机制，它通过垃圾回收器自动回收不再使用的对象，从而释放内存并确保内存的正确使用；
  • 概念：NextObjPtr
    • CLR 会在程序的托管堆中管理一个指针 NextObjPtr，托管堆维护着一个指针 NextObjPtr，用它指向将在堆中分配的下一个对象的地址；
    • 最初，该指针设置为指向托管堆的基址，即此指针指向地址空间的基地址。在一个区域的内存备非垃圾对象填满之后，CLR 会自动分配更多的区域。此过程会一直进行，直到进程的地址空间被填满；

CLR 中的 new 行为

概念：CLR 中的 new 行为

  • C# 的 new 操作符将导致 CLR 执行以下步骤：
  • 第一步
    • 计算类型的字段 (包括从基类型继承的字段) 所需的字节数；
  • 第二步
    • 加上对象额外所需的两部分开销所需的字节数；
    • 每个对象都有两个开销字段：
      • （1）类型对象指针；
      • （2）同步块索引
    • 对于 32 位应用程序，这两个字段各自需要 32 位，所以每个对象要增加 8 字节；
    • 对于 64 位应用程序，这两个字段各自需要 64 位，所以每个对象要增加 16 字节；
  • 第三步
    • 检查阶段：
      • CLR 检查区域中是否有分配对象所需的字节数；
      • 如果托管堆有足够的可用空间，就在 NextObiPtr 指针指向的地址处放入对象，为对象分配的字节会被清零；
    • 调用阶段
      • 调用类型的构造器，为类型构造器中的 this 参数传递当前 NextObiPtr 指向的对象，new 操作符返回对象引用；
      • 在返回这个引用之前，NextObjiPtr 指针的值会加上对象占用的字节数来得到一个新值，即下个对象放入托管堆时的地址；

概念：new 基于托管堆分配的空间局部性原理

  • 概念：
    • 由于托管堆的分配是在指针之上加一个值，因此其分配的速度非常的快；
    • 差不多同时分配的对象彼此间有较强的联系，而且经常差不多在同一时间访问，由于托管堆在内存中连续分配这些对象，所以会因为引用的“局部化”(locality)而获得性能上的提升；
  • 特点：
    • （1）进程的工作集会非常小，应用程序只需使用很少的内存，从而提高了速度；
    • （2）代码使用的对象可以全部驻留在 CPU 的缓存中。应用程序能以非常快的速度访问这些对象，因为 CPU 在执行大多数操作不会因 CacheMiss 而被迫访问较慢的 RAM；

垃圾回收算法

垃圾回收基础概念

概念：基于引用跟踪的算法

  • 概念：引用跟踪的算法
    • 在应用程序调用 new 操作符创建对象时，可能会没有足够地址空间来分配该对象，此时 CLR 就需要执行垃圾回收。C# 当中使用的是称之为：引用跟踪算法；
    • 引用跟踪算法只关心引用类型的变量，不关心值类型变量。因为只有引用类型的变量才能引用堆上的对象，而值类型变量直接包含值类型实例本身；
  • 概念：根
    • 在 CLR 中，将所有引用类型的变量称为根；
    • 引用类型变量可以在许多场合使用：
      • 类的静态字段
      • 类的实例字段
      • 方法参数
      • 局部变量
    • 这些内容都可以成为根；

概念：托管堆与代

  • 概念：3 代托管堆
    • 托管堆只支持三代：第 0 代、第 1 代和第 2 代，没有第 3 代；
  • 概念：代与预算
    • CLR 初始化时，会为每一代选择预算；
    • CLR 的垃圾回收器是自调节的，垃圾回收器会在执行垃圾回收的过程中了解应用程序的行为；
    • 假定应用程序构造了许多对象，但每个对象用的时间都很短，在这种情况下，对第 0 代的垃圾回收会回收大量内存。并且事实上，第 0 代的所有对象都可能被回收；
    • 当 GC 发现回收 0 代后存活下来的对象很少，就可能减少第 0 代的预算，反之增加预算。这种启发式的算法对第 1 代以及第 2 代通用适用；
    • 每当垃圾回收进行后，会检查存活下来的对象多少，进而增大或减小这些代的预算；

概念：两个假设

  • 假设一：对象越新，生存期越短
    • 因为新的对象在第 0 代中，并且第 0 代包含更多垃圾的可能性很大，在 GC 时能回收更多的内存；
    • 即对象越新，生存期越短；
    • 通过在某些情况下忽略第 1 代中的对象，可以加快垃圾回收速度；
  • 假设二：回收堆的一部分，速度快于回收整个堆
    • 忽略第 1 代中的对象能提升垃圾回收器的性能。对性能有更大提振作用的是：现在不必遍历托管堆中的每个对象；
    • 如果根或对象引用了老一代的某个对象，垃圾回收器就可以忽略老对象内部的所有引用，能在更短的时间内构造好可达对象图；

垃圾回收过程简述

概念：算法实现阶段

  • 第一阶段：暂停阶段
    • 概念：
      • 暂停进程中的所有线程。这防止线程在 CLR 检查期间访问对象并更改其状态；
    • 图示：回收之前的托管堆
  • 第二阶段：标记阶段
    • CLR 遍历堆中的所有对象，将同步块索引字段中的一位设为 0；
        1. 当此位设为 0 时，表示此对象应该被删除；
        1. 为 1 时表示不用删除；
    • 设置为 0 之后，表明所有对象都应删除。然后，CLR 检查所有活动根，查看它们引用了哪些对象。如果一个根包含 null，CLR 忽略这个根并继续检查下个根；
    • 任何根如果引用了堆上的对象，CLR 都会标记那个对象，也就是将该对象的同步块索引中的位设为 1；
    • 一个对象被标记后，CLR 会检查那个对象中的根，标记它们引用的对象。如果发现对象已经标记，就不重新检查对象的字段。这就避免了因为循环引用而产生死循环；
    • 全部检查完毕后，堆中的对象要么已标记，要么未标记。
    • 已标记的对象不能被垃圾回收，因为至少有一个根在引用它。这种对象是 可达 reachable 的对象。因为应用程序代码可通过仍在引用它的变量抵达它；
    • 未标记的对象是 不可达 unreachable 的，因为应用程序中不存在使对象能被再次访问的根；
  • 第三阶段：重定位阶段
    • 概念：
      • CLR 对堆中已标记的对象进行内存位置的转移，压缩所有幸存下来的对象，使它们占用连续的内存空间，如图所示
    • 分析：
      • （1）压缩的好处：
        • 所有幸存对象在内存中紧挨在一起，确保了引用的“局部化”，减小了应用程序的工作集，从而提升了将来访问这些对象时的性能；
        • 解决了原生堆的空间碎片化问题；
      • （2）内存偏移
        • 在内存中移动了对象之后有一个问题待解决：引用幸存对象的根现在引用的还是对象最初在内存中的位置，而非移动之后的位置；
        • CLR 要从每个根减去所引用的对象在内存中偏移的字节数，保证每个根还是引用和之前一样的对象--只是对象在内存中变换了位置；

堆的分配与回收过程

分析：堆的分配与回收过程

  • 过程：第 0 代对象
    • 预算容量：
      • CLR 初始化时，为第 0 代对象选择一个预算容量；
    • 启动垃圾回收：
      • 如果分配一个新对象造成第 0 代超过预算，就必须立马就启动一次垃圾回收；
      • 假设此时初始化时分配的对象刚好用完第 0 代的空间，那么分配下一个对象就必须启动垃圾回收；
    • 压缩对象
      • 垃圾回收器判断对象 C 和 E 是垃圾，所以会压缩对象 D，使之与对象 B 相邻；
    • 图示：
  • 过程：进入第 1 代
    • 在垃圾回收中存活的对象 A，B 和 D 现在成为第 1 代对象。第 1 代对象已经经历了垃圾回收器的一次检查；
    • 此时第 0 代不包含任何对象，当下一次再分配内存时，新对象会被继续分配到第 0 代之中；
  • 过程：继续分配第 0 代对象
    • 继续分配
      • 新分配的对象继续进入第 0 代；
      • 部分对象（包含第 1 代与第 0 代）在程序运行时，逐渐变得不可达；
    • 新对象造成第 0 代超出预算，启动垃圾回收
    • 决定 GC 检查那一代
      • 垃圾回收器必须决定检查哪些代。开始一次垃圾回收时，垃圾回收器除了检查第 0 代时，还会检查第 1 代占用了多少内存；
    • 在下图中，由于第 1 代占用的内存远少于预算，所以垃圾回收器只检查第 0 代中的对象；
  • 过程：从第 1 代对象到第 2 代对象
    • 全部预算已满
      • 假定第 1 代的增长导致它的所有对象占用了全部预算。此时应用程序继续运行并分配对象，使第 0 代对象达到它的预算容量；
      • 此时由于第 0 代已满，必须进行垃圾回收；
      • 此次垃圾回收时发现第 1 代占用太多空间，导致全部预算用完（并且第 1 代当中已经有很多对象变得不可达）；
      • 此次垃圾回收需要堆第 0 代以及第 1 代对象进行回收；
    • 产生第 2 代
      • 经过垃圾回收，产生第 2 代对象；第 0 代继续空余出来；
      • 只有在第 1 代超出预算时才会检查第 1 代中的对象；

垃圾回收触发条件

概念：几种垃圾回收触发条件

    1. CLR 检测垃圾回收超出代的预算；
    1. 代码显式调用 System.GC 的静态方法 Collect()；
    1. Windows 报告低内存情况；
    1. CLR 卸载 AppDomain；
    1. CLR 关闭；

方法：强制垃圾回收

  • 概念：
    • System.GC 类型允许应用程序对垃圾回收器进行一些直接控制。可以用 Collect 进行强制垃圾回收，但大部分情况下不推荐；
  • 定义：
    • void Collect(Int32 generation,GCCollectionMode mode,Boolean blocking);
  • 描述：
    • generation
      • 表示回收第几代的垃圾
    • GCCollectionMode
      • Default：不传递任何符号名称；
      • Forced：强制回收指定的代 (以及低于它的所有代)；
      • Optimized：只有在能释放大量内存或者能减少碎片化的前提下，才执行回收（强制执行 Collect 时，尽可能使用这种模式）；
